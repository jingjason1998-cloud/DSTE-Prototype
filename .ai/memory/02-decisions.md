## 2026-06-02 生产回滚决策

- **决策**：v0.4.2 代码拆分导致 cockpit.html 白屏，选择完整回滚到 6月1日备份
- **原因**：生产环境优先止损，回滚是最快最安全的恢复方式
- **替代方案**：正向修复（补全缺失的 js 文件）风险更高，需要本地构建验证
- **影响**：用户恢复访问，但 v0.4.2 的新功能（经营分析会独立页面、诊断改进模块）暂时不可用

## 2026-06-02 备份文件重命名决策

- **决策**：将 `meeting-reviewer-backup-*.tar.gz` 重命名为 `dste-*-backup-*.tar.gz`

## 2026-06-01 文件化记忆系统决策

- **决策**：建立 `.ai/memory/` 文件化记忆系统解决 AI 会话无状态问题

## 2026-06-16 DSTE 功能框架补齐决策

- **决策**：在现有 cockpit.html SPA + 独立页面混合架构下，把所有缺失模块至少以「页面骨架 + 路由 + 导航」形式补齐，缺的功能先用占位页/基础骨架补全，不立即做深功能。
- **原因**：
  - 平台在导航和页面层面不完整，后续迭代容易遗漏全局视角
  - 占位页能让用户/团队看清 DSTE 全图，便于按 P0/P1/P2 排期
  - 避免在框架未清晰时直接深入某个模块，导致重复返工
- **关键设计**：
  - 占位页标准化：统一为「图标 + 标题 + 描述 + 路线图步骤」结构，替代简单 `renderPlaceholder()`
  - 导航补全：新增 SP/BEM、Execute/重点工作、Review 四模块、用户中心、通知中心
  - 实施顺序：先占位页+导航，再主流程骨架，最后数据打通标记
- **影响**：
  - 新增 `.ai/tasks/active/T040-functional-framework.md` 任务配方
  - 后续开发必须以框架全景图为参考，避免重复造轮子
  - 已占位模块未来迭代时，优先替换占位页而非新增独立入口

## 2026-06-17 统一子页面切换效果与导航一致性方案决策

- **决策**：采用「渐进式统一」三阶段方案，先统一外部页面进入动画和链接格式（Phase 1），再统一 Shell 渲染（Phase 2），最后逐步 SPA 化（Phase 3）
- **原因**：
  - 项目采用 SPA Shell + 独立 HTML 页面混合架构，一次性整体重构风险高
  - cockpit.html 内部路由有 fadeIn 动画，独立页面完整刷新无过渡，视觉上割裂
  - 各独立页面的 top nav/sidebar 存在硬编码复制，与 cockpit 导航结构不一致
  - `src/lib/config.js` 和 `src/lib/shell.js` 共享框架已存在但未被使用
- **关键设计**：
  - Phase 1 只加 CSS 动画和修正链接，不改动业务逻辑
  - Phase 2 创建 `src/lib/shell-injector.js`，让独立页面共享同一套 Shell 配置
  - Phase 3 将独立页面逐步提取为 cockpit 内部路由
  - 每次迁移一个独立页面后立即跑对应 E2E
- **影响**：
  - 新增 `.ai/tasks/active/T060-navigation-unification.md` 任务配方
  - 后续新增独立页面时，优先走统一 Shell 注入，避免新增硬编码导航
  - `?embed=1` 嵌入模式必须保留，注入器需跳过嵌入模式

## 2026-06-17 会议评分算法 v2.0 模型决策

- **决策**：将 `src/meetings/utils/scoring.js` 从四维度模型（preparation/discussion/decision/execution）重构为三阶段模型（before/during/after），并引入材料审核评分 `reviewScores`
- **原因**：
  - 四维度模型与 DSTE 会前/会中/会后闭环更贴合业务场景
  - 会前材料审核评分需要回流到会议评估中
  - 原模型依赖 `metrics` 对象中的硬编码百分比，数据不可控
- **关键设计**：
  - 会前 35 分：材料完整性 7 + 议程类型覆盖 10.5 + 材料评分 17.5
  - 会中 30 分：有效讨论 12 + 参与人员 12 + 时间控制 6
  - 会后 35 分：决议&待办 30 + 评分及时性 5
  - 议程顺延扣 5 分
  - 单元测试覆盖边界 case
- **待决事项**：
  - `src/meetings.html` 仍内联旧版四维度算法，需决策是否将评估 UI 也迁移到三阶段模型
  -  mock 数据中的 `effectiveness.dimensions` 字段需同步迁移
- **影响**：
  - 算法模块与 UI 存在暂时不一致，后续打通时需同步更新 meetings.html 评估表单、卡片、详情页、mock 数据
