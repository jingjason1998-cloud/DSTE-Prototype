# RFC：会议议程材料一键送审与批量送审

> 状态：设计评审中  
> 关联系统：经营分析会（meetings）× 会议材料审核（reviewer）

---

## 一、需求澄清

### 用户原话
> 在编写会议议程的时候，能否一键送审和批量送审（关联会议材料的评审界面），有了结果后，会回传评分回来。

### 拆解后的需求

| 需求 | 说明 | 复杂度 |
|------|------|--------|
| **一键送审** | 单个议程项的材料链接 → 调用 AI 评审 → 获取评分 | 低 |
| **批量送审** | 多个议程项的材料同时送审 → 获取批量评分 | 中 |
| **回传评分** | 评审结果自动关联回对应议程项，在编辑页可见 | 低 |

### 关键约束
- **评审是异步的**：AI 评审耗时 5~30 秒，UI 不能阻塞等待
- **评审服务是独立部署**：本地端口 8766（Flask）或线上反向代理
- **已有评分回传机制**：`localStorage['dste_review_scores']` 天然支持

## 二、现有系统调研结论

### 2.1 评审系统（reviewer）API

```javascript
// 单条评审
POST {PROXY_URL}/api/review
Body: { url: "https://kms...", scene: "general-topic-review" }
Response: { success, total_score, dimension_scores, issues, report, ... }

// 批量评审
POST {PROXY_URL}/api/batch
Body: { urls: ["url1", "url2"], scene_id: "general-topic-review" }
Response: { success, task_id }

// 查询批量进度
GET {PROXY_URL}/api/batch/{task_id}
Response: { status, total, completed, failed }

// 获取批量结果
GET {PROXY_URL}/api/batch/{task_id}/results
Response: [{ url, total_score, status, ... }]
```

### 2.2 评分存储机制（已有）

评审完成后，reviewer 前端会自动写入：

```javascript
localStorage.setItem('dste_review_scores', JSON.stringify({
  "https://kms.example.com/page1": { maxScore: 82, lastReviewAt: 1718000000000 },
  "https://kms.example.com/page2": { maxScore: 65, lastReviewAt: 1718000100000 }
}));
```

经营分析会已有 `getMaterialScore(url)` 函数读取该存储，并在详情页展示 `⭐ 82分`。

**结论：回传评分无需额外开发，复用现有机制即可。**

### 2.3 议程编辑界面现状

当前议程编辑表单每行包含：

```
[序号] [类型下拉] [标题输入] [时长] [材料链接] [负责人] [上移] [下移] [删除]
```

`material_link` 字段已存在，只是纯文本输入框。

## 三、方案评估

### 方案 A：iframe 嵌入 reviewer（不推荐）

在议程编辑页弹出一个 iframe，嵌入 `reviewer.html?embed=1&url=xxx`。

| 维度 | 评估 |
|------|------|
| 开发量 | 大（iframe 尺寸适配、跨页通信、状态同步） |
| 用户体验 | 差（iframe 内滚动嵌套、两次跳转感） |
| 技术风险 | 中（跨域、iframe 生命周期管理） |
| 回传评分 | 复杂（需要 iframe → 父页通信） |

**结论：过度设计，体验割裂。**

### 方案 B：直接调用 reviewer API（推荐 ✅）

在议程编辑页直接 `fetch` 调用 reviewer 后端 API，将评审逻辑内聚在编辑流程中。

| 维度 | 评估 |
|------|------|
| 开发量 | 中（新增按钮 + API 封装 + loading/轮询状态） |
| 用户体验 | 好（不离开编辑页，评审进度 inline 展示） |
| 技术风险 | 低（复用 reviewer 后端，无需改评审系统） |
| 回传评分 | 零成本（复用 `dste_review_scores` localStorage） |

**结论：最小侵入、最大复用、体验最连贯。**

## 四、推荐方案：方案 B 详细设计

### 4.1 交互流程（单条送审）

```
用户填写议程 → 输入 material_link
                    ↓
            材料链接框右侧出现 [🔍 送审] 按钮
                    ↓
            点击 → 按钮变为 [⏳ 评审中...]
                    ↓
            POST /api/review (5~15秒)
                    ↓
            评审完成 → 按钮变为 [✅ 82分]
                    ↓
            评分写入 dste_review_scores
                    ↓
            该行材料链接框右侧显示彩色评分标签
```

### 4.2 交互流程（批量送审）

```
用户填写多个议程 → 每个有 material_link 的项自动出现复选框
                    ↓
            顶部出现 [批量送审 (已选 3/5)] 按钮
                    ↓
            勾选 → 点击批量送审
                    ↓
            POST /api/batch → 返回 task_id
                    ↓
            显示批量进度浮层（轮询 /api/batch/{task_id}）
                    ↓
            全部完成 → 进度浮层关闭，所有选中项刷新评分
```

### 4.3 议程类型 → 评审场景映射

reviewer 有 4 种评审场景，建议根据议程类型自动映射：

| 议程类型 | 映射评审场景 | 场景 ID |
|----------|-------------|---------|
| `goal_management` | 通用议题材料审核 | `general-topic-review` |
| `key_task_management` | 通用议题材料审核 | `general-topic-review` |
| `budget_finance` | 通用议题材料审核 | `general-topic-review` |
| `human_resources` | 通用议题材料审核 | `general-topic-review` |
| `business_special` | 通用议题材料审核 | `general-topic-review` |
| `other` | 通用议题材料审核 | `general-topic-review` |

> **说明**：当前 reviewer 的 4 种场景（通用议题、垂直客群落后述职、落后战区业绩承诺会、负责人年度述职）都是面向特定会议类型的，而非议程项类型。议程项的评审建议统一使用 `general-topic-review`，后续可按需扩展。

### 4.4 UI 设计

#### 单条议程行改造（编辑页）

```
Before:
[1] [目标管理 ▼] [Q1 财务复盘        ] [45min] [材料链接          ] [负责人] [↑] [↓] [🗑]

After:
[1] [目标管理 ▼] [Q1 财务复盘        ] [45min] [材料链接          ] [🔍送审] [负责人] [↑] [↓] [🗑]
                              评审后 →  [✅82分]  (绿色/橙色/红色标签)
```

- `🔍 送审` 按钮：材料链接非空时显示
- `✅ 82分` 标签：评审完成后显示，颜色按分数区间：≥80 绿、≥60 橙、<60 红
- 点击评分标签可展开：显示总分 + 各维度得分 + 红线问题（tooltip 或 mini 弹层）

#### 批量操作栏（编辑页顶部）

```
┌─────────────────────────────────────────────────────────────────┐
│  📋 议程列表                                      [批量送审 3/5] │
├─────────────────────────────────────────────────────────────────┤
│ ☑ [1] [目标管理] [Q1 财务复盘] ... [✅82分]                     │
│ ☑ [2] [人力资源] [组织架构调整] ... [⏳评审中...]                │
│ ☐ [3] [其他]     [未命名]       ... (无材料链接)                │
└─────────────────────────────────────────────────────────────────┘
```

- 全选/取消全选按钮
- 只勾选有 `material_link` 的项
- 批量送审按钮显示已选数量

#### 批量进度浮层

```
┌──────────────────────┐
│ 📦 批量材料评审中     │
│ ████████░░░░ 3/5     │
│                      │
│ ✅ Q1 财务复盘  82分  │
│ ⏳ 组织架构调整 ...   │
│ ❌ 市场策略     (失败)│
│                      │
│      [关闭]          │
└──────────────────────┘
```

### 4.5 数据流

```
┌─────────────────┐     POST /api/review      ┌─────────────────┐
│ 议程编辑页       │ ────────────────────────→ │ reviewer 后端   │
│ (meetings.html)  │                           │ (port 8766)     │
│                 │ ←──────────────────────── │                 │
└─────────────────┘    { total_score, ... }    └─────────────────┘
       │
       │ 自动写入
       ▼
┌─────────────────────────────┐
│ localStorage                 │
│ dste_review_scores[url] =    │
│   { maxScore, lastReviewAt } │
└─────────────────────────────┘
       │
       │ 读取
       ▼
┌─────────────────┐
│ getMaterialScore │ → 返回 maxScore
│ (已有函数)       │
└─────────────────┘
```

### 4.6 错误处理

| 场景 | 处理 |
|------|------|
| 材料链接为空 | 送审按钮 disabled |
| reviewer 后端不可用 | toast: "评审服务未启动，请检查 localhost:8766" |
| 评审超时（>30s） | 按钮恢复，toast: "评审超时，请重试" |
| 批量任务失败 | 进度浮层标记失败项，支持单独重试 |
| KMS 页面无法访问 | 后端返回错误，前端展示具体原因 |

## 五、实施路径（分阶段）

### Phase 1：单条送审（MVP，约 2~3 小时）

1. 在 `renderAgendaList()` 中，每个议程项的材料链接右侧添加「送审」按钮
2. 封装 `reviewMaterial(url, sceneId)` 函数，调用 `POST /api/review`
3. 按钮状态机：`送审` → `⏳` → `✅ 82分`
4. 评审完成后，刷新该行的评分标签（调用 `getMaterialScore()`）

**价值**：用户无需离开编辑页即可逐个送审，立即可用。

### Phase 2：批量送审（约 3~4 小时）

1. 每行 agenda item 前加 checkbox
2. 顶部加「批量送审」按钮 + 计数
3. 封装 `batchReviewMaterials(urls, sceneId)` 函数
4. 实现批量进度浮层（轮询 `/api/batch/{task_id}`）
5. 完成后批量刷新评分标签

**价值**：大幅提效，5 个议程项的材料一次送审。

### Phase 3：评分详情展开（可选，约 1~2 小时）

1. 点击评分标签，展开 mini 评审摘要
2. 显示：总分、各维度得分、红线问题、通过/不通过结论
3. 点击「查看完整报告」跳转 reviewer.html

**价值**：在编辑页就能快速判断材料是否达标，无需跳转。

## 六、交互复杂度定级

| 维度 | 复杂度 | 说明 |
|------|--------|------|
| 新增 UI 控件 | 🟢 低 | 按钮 + 标签 + checkbox，无复杂布局 |
| 状态管理 | 🟡 中 | 按钮三态 + 批量轮询，需统一状态机 |
| 后端依赖 | 🟢 低 | 零后端改动，复用 reviewer API |
| 跨系统通信 | 🟢 低 | 通过 localStorage 弱关联，无需改 reviewer |
| 错误边界 | 🟡 中 | 网络超时、后端不可用、KMS 页面异常 |
| **综合评估** | **🟡 中** | 技术风险可控，主要工作量在状态管理和轮询 |

## 七、用户需要确认的决策点

1. **评审场景**：当前统一用 `general-topic-review`，是否需要按议程类型映射不同场景？
2. **评分展示深度**：Phase 1 只展示总分标签即可，还是需要同时展示维度得分？
3. **批量选择策略**：默认勾选所有有材料链接的项，还是默认全不勾选让用户手动选？
4. ** reviewer 代理地址**： meetings.html 中如何配置 reviewer 的 PROXY_URL？（当前 reviewer 自己判断，meetings 需要独立获取）
