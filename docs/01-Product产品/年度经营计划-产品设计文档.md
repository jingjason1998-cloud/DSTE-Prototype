## 一、场景还原：2026 年营销线组织绩效与年度经营预算

### 1.1 业务背景

营销线作为公司核心增长引擎，2026 年组织绩效分解涉及 **11 个大区**、**3 个职能小组**（销售/售前/项目）、**多个本部团队**，覆盖财务/客户/内部流程三个 BSC 维度。与此同时，年度经营预算需要回答「目标需要多少资源、资源如何分配、花得怎么样」三个问题，与组织绩效目标形成闭环。

### 1.2 当前业务表单结构（KMS 页面还原）

```
2026年组织绩效_营销线 (KMS页面)
├── 一、营销线组织绩效（总）目标
│   └── 表格：牵引点 × KPI指标 × 全年目标 × H1占比 × 权重 × 三档目标(底线/达标/挑战) × 备注
├── 二、战区组织绩效主要指标及目标值
│   └── 表格：考核指标 × 合计 × 11个大区列
├── 三、本部团队组织绩效目标指标与目标值以及重点工作清单
│   └── 表格：职能线 × KPI × 权重 × 备注
├── 四、各子部门组织绩效 & 负责人PBC/OKR链接
│   └── 表格：部门 × 战区组织绩效链接 × PBC链接
├── 五、年度经营预算总表
│   ├── 收入预算：销售额-D 17.86亿、回款额-K 13.01亿、新产品销售额 2.78亿
│   ├── 费用预算：市场费用、研发费用、人力成本、运营成本
│   ├── 投资预算：重点项目/能力建设投入
│   └── 预算分解：按战区/部门/产品/职能线
└── 六、专项激励方案（拟）
```

### 1.3 从业务表单中提取的产品能力需求

| 业务现象 | 产品能力需求 |
|---------|------------|
| 一张大表同时包含"总目标 + 战区分解 + 职能线权重" | 需要**多视角切换**：总览视图 / 战区分解视图 / 职能线视图 |
| 同一 KPI（如销售额-D）在战区分解和职能线评价中都有出现，但权重不同 | 需要**同一指标的多维度挂载**：战区维度 × 职能维度 |
| 三档目标（底线70%/达标100%/挑战120%） | 需要**目标档位管理**，支持自定义档位和计算公式 |
| 复杂的计分规则（如"同比增长率±5个百分点为上下限"） | 需要**计分规则引擎**，支持公式配置 |
| 重点工作清单与绩效目标并列展示 | 需要**KPI ↔ 重点工作双向关联** |
| 各战区/团队有自己的 KMS 子页面 | 需要**子计划引用/链接**，支持穿透到下级分解 |
| PBC/OKR 链接独立维护 | 需要**个人绩效计划关联入口** |
| H1/H2 半年度拆分（45%/55% 默认） | 需要**周期拆分规则**，支持自动按比例拆分 |
| 年度预算总表与绩效目标分属不同表单但口径必须一致 | 需要**预算与目标同源**：KPI 目标值即预算目标，预算分解与绩效分解对齐 |
| 费用/投资/收入预算需要按组织/项目/产品拆分 | 需要**预算池 + 预算项目 + 预算分解矩阵** |
| 预算执行需要与实际花费对比 | 需要**预算执行监控**：计划/实际/预测/偏差 |
| 预算调整需要审批和版本管理 | 需要**预算版本 + 预算调整流程** |

## 二、模块定位与边界

### 2.1 在 DSTE 中的位置

```
战略制定 (SP)
    └── 战略地图（编辑态）
            ↓ 只读引用
战略解码 (BP)
    ├── 战略指标库（字典）←──── 指标定义、口径、类型
    ├── BEM 战略解码 ←──────── 战略级 KPI 锚定（5~8 个公司级）
    └── 【年度经营计划】←────── 本模块：组织级绩效分解 + 全面预算管理
            │
            ├── 组织绩效子系统 ──→ KPI 分解、职能线评价、重点工作
            └── 全面预算管理子系统 ──→ 预算编制、预算分解、预算监控
            │
            ↓ 发布
战略执行 (Execute)
    ├── 组织绩效管理 ←──────── 执行追踪、actual 录入、达成看板
    ├── 重点工作管理 ←──────── 重点工作执行与成本跟踪
    └── 经营分析会 ←────────── 预算执行回顾、滚动预测、调整决议
```

### 2.2 双轮驱动：组织绩效 + 全面预算管理

```
                    ┌─────────────────────────────────────┐
                    │          年度经营计划               │
                    │    （目标与资源一体化制定与发布）    │
                    └─────────────────┬─────────────────┘
                                      │
            ┌─────────────────────────┼─────────────────────────┐
            │                         │                         │
            ▼                         ▼                         ▼
    ┌───────────────┐         ┌───────────────┐         ┌───────────────┐
    │  组织绩效轮    │         │   预算约束    │         │  全面预算轮    │
    │               │◄───────►│   目标牵引    │◄───────►│               │
    │ • KPI 目标     │         │               │         │ • 预算池       │
    │ • 组织分解     │         │ 绩效目标决定  │         │ • 预算编制     │
    │ • 职能线权重   │         │ 资源需求上限  │         │ • 预算分解     │
    │ • 重点工作     │         │               │         │ • 预算执行监控 │
    │ • KTI 挂载     │         │ 预算约束决定  │         │ • 预算调整     │
    │ • 计分规则     │         │ 目标可行性    │         │ • 滚动预测     │
    └───────┬───────┘         └───────────────┘         └───────┬───────┘
            │                                                   │
            └─────────────────────┬─────────────────────────────┘
                                  │
                                  ▼
                    ┌─────────────────────────────┐
                    │  发布到执行 (OMP + 重点工作)  │
                    │  目标基线 + 预算基线 同步下发  │
                    └─────────────────────────────┘
```

**设计原则**：
1. **预算即目标**：KPI 的年度目标值可以直接作为收入/回款等预算目标，避免「目标一套数、预算一套数」。
2. **目标牵引资源**：绩效目标分解完成后，系统自动估算资源需求，作为预算编制的输入。
3. **预算约束目标**：预算总额不足时，需要反向挑战目标合理性，形成闭环。
4. **同一数据源**：战区分解、职能线评价、预算分解使用同一套组织/职能/产品维度。

### 2.3 职责边界

| 模块 | 做什么 | 不做什么 |
|------|-------|---------|
| **年度经营计划** | 承接 BEM 战略级 KPI → 按组织/职能分解 → 挂载 KTI → 关联任务 → **编制年度预算 → 分解预算 → 锁定预算版本** | 不定义指标（指标库负责）、不做战略推导（BEM 负责）、不做任务执行管理（重点工作负责）、**不录入实际成本（OMP 负责）** |
| **战略指标库** | 指标字典：名称、口径、公式、BSC 分类、类型 | 不存年度目标值、不存组织分解关系、不存预算数据 |
| **BEM 战略解码** | 从战略地图推导 CSF → 战略举措 → 锚定公司级 KPI | 不做组织分解、不做预算编制 |
| **组织绩效管理 (OMP)** | 执行期追踪：actual 录入、达成率计算、趋势看板、**实际成本录入、预算执行偏差监控** | 不做树结构编辑（经营计划发布后冻结）、不做预算编制 |
| **重点工作管理** | 重点工作执行、进度更新、里程碑跟踪、成本发生记录 | 不做年度预算目标制定 |
| **经营分析会** | 预算执行回顾、滚动预测讨论、预算调整决议 | 不做预算版本锁定 |

#### 2.3.1 重点工作：年度计划与 OMP 的职责边界

| 数据/操作 | 年度经营计划 (BP) | 组织绩效管理 (OMP) |
|---|---|---|
| 制定年度重点工作清单 | ✅ | ❌ |
| 挂载重点工作到 KPI / BSC 维度 | ✅ | ❌ |
| 发布重点工作到执行阶段 | ✅ | ❌ |
| 补全执行字段（类型/优先级/部门/起止日期/预算） | ❌ | ✅ |
| 更新实际成本、进度、里程碑 | ❌ | ✅ |
| 维护重点工作来源关系 | ❌（源头） | ✅（引用 `annualPlanTaskId`） |

### 2.4 数据主数据边界

| 数据 | 主数据拥有者 | 说明 |
|------|------------|------|
| targetValue / baseline / challenge | **年度经营计划** | 规划期制定，发布后只读 |
| weight / 职能线权重 | **年度经营计划** | 规划期配置，发布后只读 |
| scoringRule | **年度经营计划** 定义 | OMP 只读引用并执行计算 |
| **budgetPlanned / 预算计划值** | **年度经营计划** | 规划期编制，发布后只读 |
| **budgetAllocated / 预算分配值** | **年度经营计划** | 按 KPI / 任务 / 组织维度分配 |
| **budgetVersion / 预算版本** | **年度经营计划** | 版本锁定、调整、对比 |
| actualValue | **OMP** | 执行期录入/采集 |
| achievementRate | **OMP** 计算 | 基于年度经营计划目标 + OMP actual |
| **actualCost / 实际成本** | **OMP** | 执行期录入 |
| **budgetVariance / 预算偏差** | **OMP** 计算 | 基于年度经营计划预算 + OMP actualCost |
| 重点工作计划里程碑 | **年度经营计划** 规划 | OMP 跟踪实际完成状态 |
| 风险与假设 | **年度经营计划** 识别 | OMP 执行期更新状态 |

## 三、子系统划分

### 3.1 组织绩效子系统

负责「定目标、做分解、配权重、挂任务」。

| 功能模块 | 说明 |
|---------|------|
| 年度周期管理 | 年度经营计划的容器，含阶段状态 |
| KPI 实例管理 | 承接指标库，形成年度 KPI 树 |
| 组织分解 | 按战区/行业/产品/部门分解 KPI 目标 |
| 职能线评价 | 同一 KPI 在不同职能线的权重矩阵 |
| 重点工作规划 | 年度重点工作 CRUD 及与 KPI 双向关联 |
| KTI 挂载 | 过程指标关联到结果 KPI |
| 计分规则 | 三档目标与计分方式配置 |
| 画布视图 | 指标树可视化 |

### 3.2 全面预算管理子系统

负责「配资源、控花费、调偏差」。

| 功能模块 | 说明 |
|---------|------|
| 预算池管理 | 年度总预算及分类预算池 |
| 预算编制 | 按 KPI / 重点工作 / 组织维度编制预算项目 |
| 预算分解 | 与绩效组织分解对齐的预算分解矩阵 |
| 预算版本 | 版本锁定、版本对比、调整生成新版本 |
| 预算执行监控 | 计划/实际/预测/偏差/达成率 |
| 预算调整 | 调整申请、审批、生效 |
| 滚动预测 | 基于执行数据更新后续周期预测 |

### 4.5 重点工作派生到执行

年度计划中的重点工作在「发布到执行」时，会同步生成 OMP 执行工作骨架，建立可溯源的派生关系。

**派生规则：**

| 字段 | 年度计划源头 | OMP 执行任务 |
|---|---|---|
| `id` | `task_1` | `omp_task_1`（由 `omp_` + 源头 ID 生成） |
| `source` | `annual_plan` | `omp` |
| `annualPlanTaskId` | 无 | `task_1`（回指源头） |
| `cycleId` | `cycle_2026_marketing` | 继承同一 `cycleId` |
| `name` | 源头名称 | 继承，允许 OMP 端独立修改 |
| `bscDimension` | 源头牵引方向 | 继承 |
| `owner` | 源头负责人 | 继承，允许 OMP 端调整 |
| `status` | 源头状态 | 固定初始化为 `planning` |
| `progress` | 源头进度 | 固定初始化为 `0` |
| `type / priority / dept / startDate / endDate / budget` | 可选 | 由责任人在 OMP 中补全 |
| `actualCost / milestones / progressRecords` | 无 | 执行期在 OMP 中维护 |

**同步规则：**

1. 触发条件：用户在年度经营计划页面点击「发布到执行」。
2. 范围：仅同步当前 `cycleId` 下 `status !== 'closed'` 的年度重点工作。
3. 幂等：已存在 `annualPlanTaskId === 源头ID` 的 OMP 任务不再重复创建；未重命名的任务同步 `name / owner / bscDimension`。
4. 删除/关闭：年度计划端删除或关闭重点工作，不会级联删除 OMP 已生成的执行任务，仅标记源头状态。
5. 反向写入：OMP 端维护的 `actualCost` 可回写到年度计划重点工作的 `actualCost` 字段，用于预算执行监控。

```yaml
  # 预算分类
  categories:

      name: "收入预算"
      type: "revenue"
```

```typescript
  variance: number;                // 偏差 = actualAmount - plannedAmount
  completionRate: number;          // 执行率 = actualAmount / plannedAmount * 100

    h2Amount: number;
```

### 6.1 年度经营计划创建流程

```

[新建弹窗]
    ├─ 年度: 2026
    ├─ 所属组织: 营销线（可选）
    ├─ 是否复制上一年结构? [是/否]
    │     └─ 是 → 选择源年度（如 2025）
    │             └─ 复制范围: [ ] KPI树结构  [ ] 战区列表  [ ] 职能线配置  [ ] 预算科目
    │             └─ 注意：目标值/实际值/achievementRate/预算实际值 将被清空
    ├─ 是否从 BEM 导入战略 KPI? [是/否]
    │     └─ 是 → 选择 BEM 解码方案 → 自动导入战略级 KPI 作为根节点
    └─ 是否初始化预算池? [是/否]
          └─ 是 → 设置年度总预算及分类预算
[创建完成 → 进入 planning 状态]
    ↓ 自动打开「绩效总览」Tab
```

```
    │   ├─ ✅ 所有结果KPI权重和 = 100%

    │   ├─ ✅ 预算已锁定
    │   ├─ ✅ 预算分配合计 = 总预算
    │   ├─ ⚠️ 以下KPI未关联任务：数据治理、MTL流程项目

    ↓
[周期状态切换: planning → executing]
    ├─ KPI 树结构冻结（不可增删节点）
    ├─ 目标值进入锁定状态（修改需审批标记）
    ├─ 预算版本变为 baseline，不可直接修改
    ├─ 自动同步 KPI 目标到「组织绩效管理」
    └─ 自动同步预算基线到「组织绩效管理」和「重点工作管理」
```

### 7.5 年度重点工作清单排序与自动编号

年度重点工作清单在「绩效总览」Tab 中以表格形式展示，支持通过行内上下箭头调整显示顺序，并由系统自动维护连续序号。

#### 排序规则

- 列表按 `seq` 升序渲染，`seq` 相同的按数据写入顺序稳定展示。
- 每行操作列提供「上移」「下移」按钮：
  - 首行隐藏「上移」，末行隐藏「下移」。
  - 点击后与该方向上相邻的任务交换位置。
  - 交换后系统按当前列表顺序重新生成连续 `seq`（从 1 开始）。
- 新增重点工作时，系统自动分配当前最大 `seq + 1`，无需用户填写序号。
- 删除重点工作时，剩余重点工作按当前顺序重新编号，保持连续。
- 排序及重新编号后，同步更新已派生 OMP 执行任务的 `seq`，确保两端顺序一致。

#### 自动编号规则

- 表格「序号」列直接展示 `seq` 值。
- `seq` 始终为从 1 开始的连续整数，无重复、无跳号。
- 自动编号为系统行为，添加/编辑弹窗中不再提供序号输入框。

#### 边界场景

| 场景 | 处理 |
|------|------|
| 历史数据存在重复或跳号 | 首次渲染/操作时按当前顺序重新编号 |
| 已发布到执行的年度重点工作 | 排序后同步更新 OMP 派生任务的 `seq`；不影响 OMP 端独立创建的子任务 |
| 删除最后一个重点工作 | 列表为空，后续新增从 `seq = 1` 开始 |

```
    ↓                                ↓

```

```
    │ 发布预算基线 + 重点工作里程碑

    │   • 预算执行回顾

    │   • 预算调整决议
```

### 8.6 与规则引擎中心 / 预警中心

| 规则类型 | 规则示例 | 触发场景 |
|---------|---------|---------|
| 预算超支预警 | 某分类预算实际执行 ≥ 80% | OMP 回写 actualCost 时 |
| 预算偏差预警 | 单项预算偏差 ≥ 20% | 滚动预测或 actual 录入时 |
| 分解校验规则 | 子维度预算合计 ≠ 父预算 | 预算分解保存时 |
| 版本锁定规则 | 发布后禁止直接修改预算 | 预算版本状态为 locked 时 |

| 预算池 | Budget Pool | 年度总预算及分类预算的容器 |
| 预算项目 | Budget Item | 预算编制的最小单元 |
| 预算版本 | Budget Version | 预算的某一次锁定快照 |
| 预算调整 | Budget Adjustment | 对已锁定预算的变更申请与审批 |
| KTI | Key Tracking Indicator | 过程指标/追踪指标 |
| 职能线评价 | Function Line Evaluation | 同一 KPI 在不同职能线的权重 |
| 滚动预测 | Rolling Forecast | 基于实际执行的动态预算预测 |

## 十二、年度周期管理与多年度切换

### 12.1 设计原则

- **单页多年度**：年度经营计划保持为单一页面（`bp/annual-plan`），通过全局周期选择器切换年份，不拆分为独立页面。
- **数据隔离**：每个年度的 KPI、重点工作、预算数据通过 `cycleId` 字段隔离，存储于同一 localStorage key 中。
- **向后兼容**：2026 年度作为默认周期，保持现有数据结构和 ID 命名，确保已有测试和演示场景不受影响。
- **安全回退**：周期相关默认值不再硬编码为 2026，避免切换年份后静默显示错误数据。

### 12.2 周期模型

```yaml
Cycle:
  id: "cycle_{year}_marketing"      # 周期唯一标识
  year: 2025 | 2026 | 2027          # 年度
  name: "{year}年度 — 营销线"         # 展示名称
  phase: planning | executing | reviewing | archived
  organization: "营销线"
  parentCycleId: null               # 可关联到上级战略周期
```

### 12.3 多年度数据初始化

系统首次加载时自动初始化三个年度周期：

| 年度 | 阶段 | 数据策略 |
|------|------|---------|
| 2025 | archived | 历史归档数据，目标值按 2026 的 90% 生成，ID 加年份前缀 |
| 2026 | planning | 默认当前周期，保持现有真实业务数据与原始 ID |
| 2027 | planning | 未来规划周期，目标值按 2026 的 110% 生成，ID 加年份前缀 |

通过 `omp_buildYearSeed(year)` 工厂函数按年度生成 cycle、KPI、重点工作、里程碑、进度记录，避免数据重复定义。

### 12.4 切换机制

```
用户操作: 顶部导航栏周期选择器切换年度
    ↓
更新 window._dsteState.currentCycleId
    ↓
触发 navigate(currentHash) 重新渲染当前页面
    ↓
各模块通过 cycleId 过滤数据，显示对应年度内容
```

年度经营计划页面内部已按 `cycleId` 过滤：

```javascript
const cycleId = window._dsteState.currentCycleId;
const kpis = omp_load('kpiInstances').filter(k => k.cycleId === cycleId);
```

### 12.5 数据迁移策略

- **版本号机制**：`DATA_VERSION` 从 `canvas-v10` 升级到 `canvas-v11`，自动清除旧 localStorage 数据并重新初始化三年数据。
- **ID 兼容性**：2026 年度 KPI ID 保持原样（如 `kpi_sales_d`），2025/2027 使用年份前缀（如 `kpi_2025_sales_d`）。
- **测试兼容**：现有 E2E 测试继续以 2026 为默认周期运行，无需修改。

### 12.6 边界与限制

- 指标库（`indicators`）为全局共享数据，不随年度切换而改变。
- 战略专题（`strategy_topics`）的 `year` 字段独立管理，不受周期切换影响。
- Dashboard、报告中心等其他模块的 cosmetic 2026 文案不在本次范围内。
- 预算管理子系统（后续实现）将按周期独立存储预算池和版本。
