## 一、产品定位

### 1.1 模块定位

**战略专题管理（Strategic Topic Management）** 是 DSTE **战略制定 (SP)** 阶段的核心入口之一，与**战略地图**、**战略洞察**共同构成 SP 阶段的核心视图。

自 v1.2 起，本模块从产品层面独立为单独页面，职责边界如下：

- **战略专题管理（Strategic Topic Management）**
  - 负责将洞察或手动输入转化为中长期战略专题，并跟踪其里程碑、进度、资源与闭环；
  - 核心动作：专题新建/查看/编辑/软删除、按年份分组筛选、甘特图视图、年度深化链；
  - 页面内部仅保留**专题列表**一个 Tab，已闭环专题的沉淀统一由独立「知识中心」承接。

- **战略洞察（Strategic Insights）**（参见《战略洞察-完整设计方案.md》）
  - 负责基于华为 DSTE "五看"模型录入、管理市场洞察；
  - 核心动作：洞察新增/编辑/删除、按五看类型过滤、将高影响洞察一键生成战略专题；
  - 不直接管理专题生命周期，只通过 `linkedTopicId` 与战略专题建立单向引用。

v1.1 及之前，两者以 Tab 形式共处在同一个页面（`sp/insights-topics`）。v1.2 产品方案要求：**战略洞察**单独成页，**战略专题管理**单独成页；当前代码实现仍共用 `renderInsightsTopics()`，需后续按本方案进行页面拆分重构。

### 1.2 与相关模块的关系

```
战略制定 (SP)
├── 战略地图 (sp/strategy-map)          ← 战略全景与目标因果链
├── 战略洞察 (sp/insights)              ← 五看洞察录入、洞察 → 生成专题
└── 战略专题管理 (sp/strategy-topics)    ← 战略专题全生命周期管理
           ↓
战略解码 (BP) —— BEM / 年度经营计划    ← 战略专题输出可作为解码输入
           ↓
战略执行 (Execution) —— 业务专题管理    ← 非战区战略专题可分解为业务专题执行；战区专题独立闭环，不进入业务专题
           ↓
知识中心 (knowledge-center)            ← 已闭环专题沉淀为可复用知识资产
```

| 模块         | 时间跨度   | 核心对象            | 与本模块关系         |
| ---------- | ------ | --------------- | -------------- |
| **战略地图**   | 3-5 年  | 战略目标、BSC 四维度    | 专题可关联战略目标（预留）  |
| **战略洞察**   | 持续     | 洞察（五看）          | 生成战略专题的输入源     |
| **战略专题管理** | 1-3 年  | 战略专题            | 本方案核心模块        |
| **业务专题管理** | 季度-1 年 | 业务议题、ST/AT 会议议题 | 非战区战略专题可分解为业务专题；战区专题不关联业务专题 |
| **年度经营计划** | 1 年    | KPI 实例、组织绩效     | 专题目标可输入到年度经营计划 |
| **知识中心**   | 持续     | 已闭环专题、方法论、报告   | 已闭环专题自动沉淀为知识资产 |

### 1.3 关键设计原则

- **洞察驱动专题**：每条高影响洞察可一键生成专题，避免洞察与执行脱节；
- **页面职责分离**：战略洞察聚焦"发现与输入"，战略专题管理聚焦"规划与闭环"，两个页面独立导航、独立 URL；
- **战区专题独立闭环**：战区专题作为战略专题的特殊类型，用于沉淀战区三年规划等知识资产，不向下分解为业务专题，避免跨年度规划与当年执行混淆；
- **年份分组管理**：专题按自然年分组，支持跨年度战略对比；
- **闭环沉淀**：专题闭环后，其总结摘要同步到独立知识中心展示，成为可复用资产；
- **共享数据模型**：洞察与专题继续共用 `dste_insights_v1` / `dste_strategy_topics_v2`，但分别由两个页面读写各自关心的字段；
- **纯前端实现**：与当前系统一致，使用 localStorage 持久化，单页 HTML + 原生 JS。

## 三、数据模型

### 3.1 战略专题（StrategyTopic）

```typescript
interface StrategyTopic {
  // === 主键 ===
  id: string;                    // 格式: st_{timestamp}_{random}，safeTopicId 校验

  // === 基本信息（XSS 防护必需）===
  name: string;                  // 专题名称
  summary: string;               // 结项总结/背景说明
  researchObjectives: string;    // 专题研究目标

  // === 外部链接 ===
  kmsUrl: string;                // KMS 页面链接，如 https://kms.fineres.com/pages/...

  // === 分类 ===
  type: 'breakthrough' | 'incremental' | 'maintenance';  // 突破型 / 增量型 / 维持型

  // === 编码与排序 ===
  code: string;                  // 自动生成的专题编码，如 ST-2026-001
  sortOrder: number;             // 清单内排序序号，支持拖拽/手动调整

  // === 交付件 ===
  deliverables: string[];         // 交付件清单，如 ['市场调研报告', '三年业务规划']

  // === 生命周期 ===
  status: 'insight' | 'planning' | 'execution' | 'closed';

  // === 时间与年份 ===
  year: number;                  // 专题归属年份，用于分组与筛选
  startDate: string;             // YYYY-MM-DD
  endDate: string;               // YYYY-MM-DD

  // === 负责人与成员 ===
  owner: string | PersonObject;  // 支持人员选择器
  members: (string | PersonObject)[]; // 专题成员，支持多人

  // === 年度深化链 ===
  previousTopicId: string | null; // 上一年/上一阶段关联专题 ID
  nextTopicId: string | null;     // 下一年深化专题 ID

  // === 里程碑 ===
  milestones: Milestone[];

  // === 软删除 ===
  deleted: boolean;              // true 表示已删除，列表中过滤

  // === 元数据 ===
  createdAt: string;             // YYYY-MM-DD
  updatedAt: string;             // YYYY-MM-DD
}

interface Milestone {
  title: string;                 // 里程碑标题（XSS 防护必需）
  date: string;                  // YYYY-MM-DD
  completed: boolean;            // 是否已完成
}
```

### 3.2 关键枚举映射

| 字段 | 存储值 | 展示值 | 样式 |
|------|--------|--------|------|
| `type` | `breakthrough` | 突破型 | 主色标签 |
| `type` | `incremental` | 增量型 | 信息色标签 |
| `type` | `maintenance` | 维持型 | 默认标签 |
| `status` | `insight` | 洞察阶段 | `status-default` |
| `status` | `planning` | 规划阶段 | `status-primary` |
| `status` | `execution` | 执行阶段 | `status-warning` |
| `status` | `closed` | 已闭环 | `status-success` |

### 3.3 数据存储键名

```javascript
const STORAGE_KEY_TOPICS = 'dste_strategy_topics_v2';         // StrategyTopic[]
const STORAGE_KEY_TOPICS_LEGACY = 'dste_strategy_topics_v1';  // 旧版数据，迁移后删除
```

### 3.4 数据迁移

从旧版 `dste_strategy_topics_v1` 迁移到 v2 时，为没有 `year` 字段的专题补充年份：

```javascript
function migrateTopicsFromV1() {
  const legacy = DSTE.Storage.getString(STORAGE_KEY_TOPICS_LEGACY);
  if (!legacy) return false;
  const topics = JSON.parse(legacy);
  const migrated = topics.map((t, index) => {
    const year = t.year || (t.startDate ? parseInt(t.startDate.split('-')[0], 10) : new Date().getFullYear());
    const item = {
      ...t,
      year,
      code: t.code || generateTopicCode(year, index + 1),
      sortOrder: typeof t.sortOrder === 'number' ? t.sortOrder : index,
      members: t.members || [],
      kmsUrl: t.kmsUrl || '',
      researchObjectives: t.researchObjectives || '',
      deliverables: t.deliverables || [],
      previousTopicId: t.previousTopicId || null,
      nextTopicId: t.nextTopicId || null
    };
    // v2 起取消 budget、actualCost、priority、dimension、progress、insights 字段
    delete item.budget;
    delete item.actualCost;
    delete item.priority;
    delete item.dimension;
    delete item.progress;
    delete item.insights;
    return item;
  });
  DSTE.Storage.set(STORAGE_KEY_TOPICS, migrated);
  DSTE.Storage.remove(STORAGE_KEY_TOPICS_LEGACY);
  return true;
}
```

## 四、功能模块设计

### 4.1 页面职责

- 管理战略专题的全生命周期（洞察阶段 → 规划阶段 → 执行阶段 → 已闭环）；
- 按年份分组展示专题，支持年份筛选；
- 维护里程碑、进度、负责人、成员等资源信息；
- 支持列表视图与甘特图视图切换；
- 支持跨年专题的年度深化链；
- 专题闭环后，其 `summary` 等数据将同步到**独立知识中心**进行沉淀与复用（参见《知识中心-完整设计方案.md》）。

页面内部仅保留**专题列表 Tab**，不再内置「知识沉淀 Tab」。

### 4.2 专题列表 Tab

#### 年份选择器

- 下拉选项包含"全部年份"、所有专题已有年份、当前年、下一年；
- 默认选中"全部年份"，确保历史专题（如 2025 年已闭环专题）不会被默认隐藏；
- 切换年份后，统计卡片、专题数量、专题表格均按选中年份过滤。

#### 统计卡片

基于当前筛选范围计算：

| 指标 | 计算方式 | 说明 |
|------|---------|------|
| 进行中 | `status !== 'closed' && status !== 'insight'` 的专题数 | 已进入规划/执行阶段 |
| 已完成 | `status === 'closed'` 的专题数 | 已闭环 |
| 已洞察 | `status === 'insight'` 的专题数 | 尚未进入规划阶段 |

#### 专题表格

按年份分组展示，选择"全部年份"时展示所有年份分组；选择单一年份时仅展示该年份分组。

每组一个卡片，包含：
- 年份标题 + 专题数量标签
- 进行中/已完成计数
- 数据表格：编码、名称、类型、负责人、成员、阶段、更新日期、操作

操作按钮：查看、编辑、删除。列表支持拖拽排序，排序结果持久化到 `sortOrder` 字段。自动编码按当前年份内专题创建顺序生成，编码规则 `ST-{year}-{序号}`。

#### 专题详情弹窗

展示字段：
- 标题区：专题编码、专题名称、年份、类型
- 信息网格：负责人、成员、阶段、开始日期、结束日期
- 研究目标：独立区块展示
- 交付件：列表展示，支持批量查看
- KMS 链接：可点击跳转，无链接时显示 "—"
- 里程碑列表：标题、日期、完成状态
- 年度深化链：横向时间轴展示上一阶段 / 当前 / 下一阶段，可跳转
- 操作按钮：关闭、编辑、[下一年继续深化]/[查看下一年深化专题]

#### 专题编辑弹窗

可编辑字段：

| 字段     | 控件                 | 说明                                   |
| ------ | ------------------ | ------------------------------------ |
| 专题编码   | text(disabled)     | 自动生成，编辑时不可修改                      |
| 专题名称   | text               | 必填                                   |
| 负责人    | person-input       | 支持人员选择器                              |
| 成员     | person-input-multi | 支持多人选择                               |
| 研究目标   | textarea           | 必填，长度 1-2000                         |
| KMS 链接 | url                | 可选，需符合 URL 格式                        |
| 类型     | select             | breakthrough/incremental/maintenance |
| 阶段     | select             | insight/planning/execution/closed    |
| 交付件    | deliverable-input  | 动态列表，支持批量粘贴、批量添加、批量删除                |
| 开始日期   | date               | —                                    |
| 结束日期   | date               | —                                    |
| 年份     | text(disabled)     | 编辑时不可修改                              |

里程碑管理：
- 动态添加/删除里程碑行；
- 每行包含：完成 checkbox、标题输入、日期输入、删除按钮；
- 保存时过滤标题为空的行。

交付件管理（支持批量修改）：
- 提供「逐条添加」和「批量粘贴」两种模式；
- 批量粘贴：在 textarea 中每行输入一个交付件，保存时按行拆分为 `deliverables` 数组；
- 列表展示当前所有交付件，每行可单独删除；
- 保存时过滤空白项。

#### 自动编码与清单排序

**自动编码**：
- 新建专题时自动生成唯一编码 `ST-{year}-{序号}`，序号按该年份内已存在专题数 + 1 计算；
- 编码在详情页、表格首列、甘特图标签等位置展示；
- 编码一旦生成不可修改，删除的专题所占序号不回收，避免历史引用混乱；
- 保存前校验同一年份内编码唯一性，若冲突则自动顺延序号。

**清单排序**：
- 每个专题拥有 `sortOrder` 字段，用于控制同一年份内的展示顺序；
- 列表页支持拖拽行调整顺序，拖拽后自动重算范围内 `sortOrder`；
- 编辑弹窗提供「排序号」数字输入，用于精确调整位置；
- 同一年份内专题默认按 `sortOrder` 升序排列，「全部年份」视图按年份分组后各组内再按 `sortOrder` 排序；
- 跨年专题排序号相互隔离，不同年份可存在相同 `sortOrder`。

#### 新建专题

复用编辑弹窗结构，差异点：
- 标题为"新建专题"；
- 所有字段为空/默认值；
- **年份可编辑**（下拉选择当前年/下一年/历史年份）；
- 保存时生成 `id = generateTopicId()`，`code = generateTopicCode(year, 当前年份专题数 + 1)`，`sortOrder = 当前年份最大排序号 + 1`，`createdAt = today`，`deleted = false`；
- 编码一旦生成，后续不可修改。

#### 删除专题

采用**软删除**：
1. 确认弹窗展示专题编码与名称；
2. 执行删除时设置 `deleted = true`、`updatedAt = today`；
3. 保存后刷新页面，已删除专题不再显示；
4. 删除专题时，同步清空关联专题的 `previousTopicId` 或 `nextTopicId`，防止断链指向已删除专题。

### 4.3 甘特图视图

战略专题管理页面提供**甘特图视图**，用于按时间轴直观展示各专题的周期、重叠关系与里程碑节点。

#### 视图切换

页面顶部提供「列表视图 / 甘特图视图」切换按钮，切换时保留当前年份筛选条件。

#### 甘特图数据

- **主条目**：每个战略专题一条横向条形，起点为 `startDate`，终点为 `endDate`；
- **条形颜色**：按 `status` 着色：
  - `insight` / `planning`：`--primary`
  - `execution`：`--warning`
  - `closed`：`--success`
- **里程碑标记**：在条形上叠加里程碑节点（菱形/圆点），hover 显示里程碑标题与日期；
- **文本标签**：条形左侧显示专题名称，右侧显示负责人（空间不足时截断并 tooltip 展示）。

#### 筛选与缩放

- **年份筛选**：与列表视图共用，选择单一年份时仅展示该年份内的专题；
- **时间刻度**：默认按「月」展示，支持切换到「季度」；
- **状态筛选**：可勾选只展示特定阶段的专题。

#### 交互

- 点击条形或里程碑 → 打开专题详情弹窗；
- hover 条形 → 展示专题卡片：编码、名称、阶段、起止日期；
- 当前日期线：在甘特图上叠加垂直「今天」标记线。

#### 空状态

无专题或当前筛选无结果时，显示占位提示并引导新建专题。

#### 实现建议

- 使用 SVG 或纯 CSS + div 实现，不引入外部图表库以保持单文件 HTML 架构；
- 计算每个专题的偏移量与宽度时，以所选时间范围的第一天为基准；
- 对跨年专题，按年份筛选时仍完整展示其条形（可能超出当年范围），避免信息截断；
- 对属于同一年度深化链的专题，使用同色系或虚线连接，hover 显示链上其他年份专题。

### 4.4 年度深化链

针对跨年持续深化的战略专题（如 2025 年专题在 2026 年继续深化），提供**年度深化链**功能，通过 `previousTopicId` / `nextTopicId` 建立链式关联，既保留按年分组的独立性，又支持追溯历年深化脉络。

#### 触发条件

专题详情页显示「年度深化」入口：
- 当专题 `status === 'execution' || status === 'closed'` 时可见；
- 若该专题已存在 `nextTopicId`，则入口改为「查看下一年深化专题」。

#### 创建下一年专题

点击「2026 年继续深化」（按钮文案动态取当前专题年份 + 1）后，打开预填充的专题创建弹窗：

| 字段 | 预填充来源 | 默认值/说明 |
|------|-----------|------------|
| `name` | 原专题 `name` | 原名称 + "（2026）"，用户可改 |
| `researchObjectives` | 原专题 `researchObjectives` | 继承原目标 |
| `members` | 原专题 `members` | 继承成员 |
| `kmsUrl` | 原专题 `kmsUrl` | 继承 KMS 链接 |
| `type` | 原专题 `type` | 继承 |
| `year` | — | 原专题年份 + 1 |
| `status` | — | `'planning'` |
| `previousTopicId` | 原专题 `id` | 建立反向关联 |

保存流程：
1. 用户确认/修改预填充内容；
2. 保存新专题到 `dste_strategy_topics_v2`；
3. 回写原专题 `nextTopicId = newTopic.id`；
4. 若原专题未闭环，提示用户是否同步将原专题标记为 `closed` 并填写阶段总结。

#### 列表页链式标识

- 存在 `nextTopicId` 的专题：行尾显示「↳ 2026 年已深化」标签，点击跳转；
- 存在 `previousTopicId` 的专题：行首显示「↰ 承继 2025」标签，点击跳转；
- 存在双向关联的专题：同时显示两个标签，并用同色系底色标识为链上专题。

#### 详情页关联展示

专题详情弹窗底部增加「年度深化链」区域：
- 上一阶段：显示 `previousTopicId` 对应专题名称，可点击跳转；
- 当前阶段：高亮当前专题；
- 下一阶段：显示 `nextTopicId` 对应专题名称，可点击跳转；
- 若链上有多个阶段，以横向时间轴形式展示。

#### 甘特图跨年链

- 同一年度深化链的专题在甘特图中使用同色系；
- 用虚线或箭头连接相邻年份专题的条形；
- hover 整条链时展示各年份专题名称与阶段。

#### 知识中心关联

- 链上每年的专题各自作为独立知识资产沉淀；
- 在知识资产详情中展示「关联年度专题」时间轴，可跳转到链上其他年份专题详情；
- 支持按链聚合搜索（未来 v1.1 可扩展）。

#### 数据安全

- 创建下一年专题前校验 `safeTopicId(previousTopicId)`；
- 删除专题时，同步清空关联专题的 `previousTopicId` 或 `nextTopicId`，防止断链指向已删除专题；
- 加载专题列表时，对孤儿 `previousTopicId` / `nextTopicId` 自动置空。

## 七、安全设计

### 7.1 防御矩阵

| 防线 | 措施 | 说明 | 优先级 |
|------|------|------|--------|
| **XSS 防护** | 所有用户输入字段使用 `escapeHtml()` 后再插入 DOM | 包括专题名称/摘要/里程碑标题/交付件 | P0 |
| **ID 校验** | `safeTopicId()` 白名单校验 | 防止非法 ID 注入 | P0 |
| **编码唯一性** | `generateTopicCode()` 按年份顺序生成，保存前校验同一年份内编码不重复 | 防止编码冲突 | P0 |
| **软删除** | 专题标记 `deleted: true`，列表过滤 | 防止误删导致数据丢失 | P0 |
| **人员输入安全** | 使用 `ensurePersonInput()` + `getPersonValue()` | 统一人员选择器取值，避免内联脚本 | P1 |
| **数据拷贝** | `loadTopics()` 返回浅拷贝 | 防止外部直接修改缓存 | P1 |
| **存储保护** | localStorage 读写加 try-catch | 损坏时返回默认值 | P1 |
| **链安全** | 创建下一年专题前校验 `safeTopicId(previousTopicId)`；删除时清理链指针 | 防止断链指向已删除专题 | P1 |

### 7.2 安全函数实现

```javascript
/**
 * HTML 转义
 */
function escapeHtml(text) {
  if (text == null) return '';
  const div = document.createElement('div');
  div.textContent = String(text);
  return div.innerHTML;
}

/**
 * 专题 ID 白名单校验
 */
function safeTopicId(id) {
  return typeof id === 'string' && /^st_\d+_\d+$/.test(id);
}

/**
 * 安全加载专题数据
 */
function safeLoadTopics() {
  try {
    const raw = DSTE.Storage.getString(STORAGE_KEY_TOPICS);
    if (!raw) return [];
    const topics = JSON.parse(raw);
    if (!Array.isArray(topics)) return [];
    return topics.filter(t => safeTopicId(t.id));
  } catch (e) {
    console.error('[SafeLoad] 专题数据损坏:', e);
    return [];
  }
}
```

### 9.1 localStorage 满降级

```javascript
function handleStorageFull() {
  const quota = DSTE.Storage.checkQuota();
  if (quota.isCritical) {
    // 策略1: 清理知识沉淀占位数据
    DSTE.Storage.remove(STORAGE_KEY_KNOWLEDGE);
    // 策略2: 若仍满，提示用户导出
    if (DSTE.Storage.checkQuota().isCritical) {
      showToast('存储空间不足，请导出数据后清理历史记录', 'error');
      return false;
    }
  }
  return true;
}
```

### 9.3 孤儿链清理

```javascript
function cleanupOrphanTopicChainLinks(topics) {
  const activeTopicIds = new Set(topics.filter(t => !t.deleted).map(t => t.id));
  let changed = false;
  const cleaned = topics.map(t => {
    const prev = activeTopicIds.has(t.previousTopicId) ? t.previousTopicId : null;
    const next = activeTopicIds.has(t.nextTopicId) ? t.nextTopicId : null;
    if (prev !== t.previousTopicId || next !== t.nextTopicId) {
      changed = true;
      return { ...t, previousTopicId: prev, nextTopicId: next };
    }
    return t;
  });
  return { topics: cleaned, changed };
}
```
