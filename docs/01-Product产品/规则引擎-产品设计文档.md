## 一、产品概述

### 1.1 定位与价值主张

**规则引擎中心**是DSTE战略管理平台的**核心计算中枢**，承载企业管理会计规则与薪酬激励规则的**配置、计算、触发、审计**全生命周期管理。

**核心价值：**
- **规则集中化**: 将分散在Excel/制度文件中的计算规则数字化、结构化
- **计算自动化**: 预算编制、偏差分析、奖金计算、预警触发等场景自动执行规则运算
- **口径一致性**: 确保DSTE全流程使用统一的规则口径，避免"数出多门"
- **变更可追溯**: 规则版本管理，任何变更留痕，满足审计要求
- **预警规则统一**: 预警触发条件作为规则的一种，纳入统一管理，避免各模块各自定义阈值

## 二、用户角色与故事

### 2.1 角色定义

| 角色 | 职责 | 核心诉求 |
|------|------|---------|
| **规则制定者** | 制定/修订规则 | 规则表达准确、无歧义、易于维护 |
| **规则审批者** | 审批规则变更 | 变更影响可控、风险可评估 |
| **规则使用者** | 在日常工作中触发规则 | 计算结果准确、响应及时、可解释 |
| **规则审计者** | 审计规则执行情况 | 变更可追溯、执行可验证 |

### 2.2 用户故事

**规则制定者**
```
作为 财务经理
我希望 通过可视化界面配置预算编制规则
以便 不需要开发人员参与即可调整预算计算逻辑

作为 HR经理
我希望 配置年终奖计算公式，支持绩效系数、部门系数、个人系数多层嵌套
以便 灵活适应不同业务线的激励策略

作为 财务经理
我希望 定义基线管理规则，包括基线来源、调整因子、审批流程
以便 确保年度目标设定有统一标准
```

**规则审批者**
```
作为 财务总监
我希望 查看规则变更的影响分析（哪些模块会受影响）
以便 审批前评估变更风险

作为 财务总监
我希望 规则变更需要多级审批才能生效
以便 重要规则不被随意修改
```

**规则使用者**
```
作为 战略运营
我希望 在编制预算时自动应用预算规则生成预算建议
以便 提高编制效率，减少人工计算错误

作为 部门经理
我希望 查看本部门的奖金计算明细（规则公式+输入值+结果）
以便 理解奖金构成，有异议时可追溯

作为 经营分析会组织者
我希望 系统自动按核算规则归集各专题的成本数据
以便 快速生成经营分析报告
```

**规则审计者**
```
作为 内审人员
我希望 查看规则的历史版本和变更记录
以便 审计时验证计算口径的一致性

作为 系统管理员
我希望 查看规则的执行日志（何时、何地、何规则被触发）
以便 排查计算异常问题
```

### 模块1: 规则配置管理

> **定位升级**: 从"简单的参数配置"升级为"企业级规则全生命周期管理平台"，支持复杂公式编辑、版本控制、多级审批，确保规则变更的可控性和可追溯性。

#### 1.1 模块职责
- **做什么：** 提供规则的分类管理、可视化编辑、版本控制、发布审批全流程
- **不做什么：** 不替代Excel做复杂数据分析、不处理非结构化规则文档

#### 1.2 涉及的数据实体

| 实体 | 说明 | 关键字段 |
|------|------|---------|
| 规则分类 | 规则的层级目录结构 | 分类ID、父分类ID、分类名称、分类编码、排序号、状态 |
| 规则定义 | 单条规则的元数据 | 规则ID、规则名称、规则编码、分类ID、规则类型、版本号、状态、生效时间、失效时间、Owner |
| 规则版本 | 规则的历史版本记录 | 版本ID、规则ID、版本号、规则内容JSON、变更说明、创建人、创建时间 |
| 规则审批流 | 规则变更的审批记录 | 审批ID、规则ID、版本号、提交人、提交时间、审批人、审批状态、审批意见 |
| 规则参数 | 规则中引用的外部参数 | 参数ID、参数名称、参数编码、参数类型、默认值、取值范围、说明 |

#### 1.3 功能列表

| 功能 | 用户故事 | 输入 | 输出 |
|------|---------|------|------|
| 规则分类管理 | 管理员希望建立规则目录结构 | 分类名称、父分类、排序 | 层级分类树 |
| 规则创建 | 财务经理希望新增一条预算规则 | 规则名称、分类、规则类型、公式 | 规则草稿 |
| 规则编辑 | 财务经理希望修改现有规则 | 规则ID、修改内容 | 新版本草稿 |
| 规则版本对比 | 审批者希望查看变更内容 | 两个版本号 | 差异对比（高亮显示） |
| 规则提交审批 | 规则制定者希望提交规则变更 | 规则ID、变更说明、影响分析 | 审批流程启动 |
| 规则审批 | 审批者希望审批或驳回变更 | 审批ID、审批意见、通过/驳回 | 审批状态更新 |
| 规则发布 | 管理员希望将审批通过的规则发布生效 | 规则ID、版本号 | 规则状态变为"已发布"，通知相关模块 |
| 规则停用 | 管理员希望停用失效规则 | 规则ID、停用原因 | 规则状态变为"已停用"，关联模块收到通知 |
| 规则复制 | 用户希望基于现有规则快速创建新规则 | 源规则ID、新规则名称 | 新规则草稿（复制源规则内容） |
| 规则查询 | 用户希望按条件查找规则 | 关键词、分类、类型、状态 | 规则列表 |

#### 1.4 与其他模块的关系

```
规则配置管理
```

#### 1.5 关键规则

- 规则编码全局唯一，不可重复
- 规则版本号采用语义化版本（主版本.次版本.修订号），重大变更升主版本
- 已发布的规则不能直接编辑，必须创建新版本
- 规则审批通过后才能发布，审批驳回后可重新编辑提交
- 规则停用后，关联的计算任务使用最后生效版本，但不再接受新任务

### 模块2: 管理会计规则

> **定位升级**: 从"静态制度文档"升级为"可计算、可触发、可追溯的动态规则引擎"，直接支撑预算编制、执行监控、经营分析的核心计算场景。

#### 2.1 预算规则

##### 2.1.1 模块职责
- **做什么：** 定义预算编制、调整、控制的全套计算规则
- **不做什么：** 不替代预算管理模块的编制界面，只提供计算逻辑

##### 2.1.2 预算规则类型

| 规则类型 | 说明 | 应用场景 |
|---------|------|---------|
| **预算编制规则** | 定义预算基线的计算方法 | 年度预算编制时自动生成预算建议 |
| **预算调整规则** | 定义预算调整的触发条件和计算方式 | 中期调整时自动计算调整额度 |
| **预算控制规则** | 定义预算执行的预警阈值和控制动作 | 执行过程中超预算时自动预警/冻结 |
| **预算分摊规则** | 定义费用在不同维度间的分摊方法 | 公共费用分摊到部门/项目/产品 |

##### 2.1.3 预算编制规则示例

```yaml
规则名称: 销售收入预算编制规则
规则编码: BUDGET-001
规则类型: 预算编制

计算公式:
  预算营收 = 上年实际营收 × (1 + 行业增长率) × 市场系数 × 战略系数

参数定义:
  上年实际营收:
    来源: 核算系统.上年度实际营收
    取值方式: 自动获取
  
  行业增长率:
    来源: 外部数据.行业报告
    默认值: 0.08
    取值范围: [-0.3, 0.5]
    可编辑: true
  
  市场系数:
    来源: 手动输入
    默认值: 1.0
    取值范围: [0.5, 1.5]
    说明: 根据市场竞争态势调整，>1表示乐观，<1表示保守
  
  战略系数:
    来源: 战略意图.增长目标
    默认值: 1.2
    说明: 根据公司战略意图设定，如"增长30%"则系数为1.3

计算步骤:
  1. 获取上年实际营收
  2. 应用行业增长率调整
  3. 应用市场系数调整
  4. 应用战略系数调整
  5. 输出预算营收建议值

输出结果:
  - 预算建议值
  - 计算明细（每步的输入、公式、结果）
  - 置信度评估（基于数据质量）
```

##### 2.1.4 预算控制规则示例

```yaml
规则名称: 预算执行控制规则
规则编码: BUDGET-CTRL-001
规则类型: 预算控制

预警阈值:
  黄色预警: 预算执行率 >= 80%
  橙色预警: 预算执行率 >= 90%
  红色预警: 预算执行率 >= 100%

控制动作:
  黄色预警:
    - 发送预警通知给部门负责人
    - 在驾驶舱显示预警标识
  
  橙色预警:
    - 发送预警通知给部门负责人+财务经理
    - 要求提交超支说明
    - 冻结该科目剩余预算的50%
  
  红色预警:
    - 发送预警通知给部门负责人+财务经理+财务总监
    - 要求提交超支说明和整改计划
    - 冻结该科目全部剩余预算
    - 触发经营分析会专题讨论

例外审批:
  - 红色预警后，经财务总监审批可解冻部分预算
  - 战略专项预算不受此规则限制（单独管理）
```

| 预算规则测试 | 财务经理希望在发布前测试规则计算结果 | 测试数据、规则ID | 计算结果、计算明细 |
| 预算规则应用 | 战略运营希望在编制预算时调用规则 | 预算编制请求、组织维度、时间维度 | 预算建议值、计算明细 |
| 预算控制触发 | 系统自动在执行超阈值时触发控制 | 预算执行数据、控制规则 | 预警通知、控制动作 |
| 预算规则版本对比 | 财务经理希望对比两版预算规则的差异 | 两个版本号 | 差异报告（公式、参数、阈值变化） |

#### 2.2 基线规则

##### 2.2.1 模块职责
- **做什么：** 定义战略/经营基线的计算方法、调整规则、版本管理
- **不做什么：** 不存储基线数据本身（由执行模块存储），只存储计算规则

##### 2.2.2 基线规则类型

| 规则类型 | 说明 | 应用场景 |
|---------|------|---------|
| **基线设定规则** | 定义基线的初始计算方法 | 年度目标设定时计算基线值 |
| **基线调整规则** | 定义基线调整的触发条件和计算方式 | 外部环境变化时调整基线 |
| **基线对比规则** | 定义实际值与基线对比的分析方法 | 经营分析时生成偏差分析 |

##### 2.2.3 基线设定规则示例

```yaml
规则名称: 营收基线设定规则
规则编码: BASELINE-001
规则类型: 基线设定

基线计算方式:
  方法: 加权移动平均法
  公式: 基线 = Σ(历史实际值 × 权重系数) × 调整因子

历史数据:
  数据范围: 过去36个月
  权重系数:
    最近12个月: 0.5
    前12个月: 0.3
    最早12个月: 0.2

调整因子:
  行业趋势调整:
    来源: 外部行业数据
    说明: 根据行业增长/衰退趋势调整
  
  战略意图调整:
    来源: 战略意图.增长目标
    说明: 根据公司战略要求调整
  
  季节性调整:
    来源: 历史季节性系数
    说明: 根据业务季节性特征调整

输出:
  - 基线值
  - 基线设定明细（历史数据、权重、调整因子）
  - 置信区间（基线±X%）
```

##### 2.2.4 基线对比规则示例

```yaml
规则名称: 基线偏差分析规则
规则编码: BASELINE-COMP-001
规则类型: 基线对比

偏差计算:
  绝对偏差 = 实际值 - 基线值
  相对偏差 = (实际值 - 基线值) / 基线值 × 100%

偏差分级:
  正常: |相对偏差| <= 5%
  关注: 5% < |相对偏差| <= 10%
  预警: 10% < |相对偏差| <= 20%
  严重: |相对偏差| > 20%

分析维度:
  - 时间维度: 月度/季度/年度趋势
  - 组织维度: 部门/区域/产品线
  - 因素分解: 价格因素/销量因素/结构因素

输出:
  - 偏差值和偏差率
  - 偏差分级（颜色标识）
  - 趋势分析图
  - 因素分解报告
  - 根因分析建议（基于规则库）
```

##### 2.3.1 模块职责
- **做什么：** 定义成本归集、收入确认、费用分摊、利润核算的计算规则
- **不做什么：** 不替代财务核算系统，只提供核算口径规则

##### 2.3.2 核算规则类型

| 规则类型 | 说明 | 应用场景 |
|---------|------|---------|
| **成本归集规则** | 定义成本如何归集到部门/项目/产品 | 经营分析时计算各维度成本 |
| **收入确认规则** | 定义收入确认的时间点和金额 | 业绩统计时确认收入 |
| **费用分摊规则** | 定义公共费用如何分摊 | 公共费用分摊到受益部门 |
| **利润核算规则** | 定义利润计算的方法和口径 | 经营复盘时计算贡献利润 |

##### 2.3.3 成本归集规则示例

```yaml
规则名称: 战略专题成本归集规则
规则编码: ACCT-COST-001
规则类型: 成本归集

归集对象: 战略专题

成本类型:
  直接成本:
    - 人工成本: 参与专题的人员工时 × 工时费率
    - 物料成本: 专题直接消耗的物料
    - 外包成本: 专题外包服务费用
  
  间接成本:
    - 管理费用分摊: 按专题收入占比分摊
    - 平台费用分摊: 按专题使用平台资源占比分摊
    - 营销费用分摊: 按专题受益占比分摊

归集周期: 月度
归集方式: 自动归集 + 手动调整

输出:
  - 专题成本明细表
  - 成本结构分析（直接/间接占比）
  - 成本趋势图
```

##### 2.3.4 利润核算规则示例

```yaml
规则名称: 业务线贡献利润核算规则
规则编码: ACCT-PROFIT-001
规则类型: 利润核算

计算公式:
  贡献利润 = 确认收入 - 直接成本 - 可追溯间接成本
  
  经营利润 = 贡献利润 - 分摊管理费用
  
  净利润 = 经营利润 - 税费

核算口径:
  收入确认: 按交付完成百分比确认
  成本匹配: 收入与成本同期确认
  费用分摊: 按受益原则分摊

输出:
  - 利润表（多维度）
  - 利润结构分析
  - 利润趋势分析
  - 利润敏感性分析
```

### 模块4: 预警规则 ⭐新增

> **定位**: 预警触发条件作为规则的一种，纳入规则引擎统一管理。规则引擎负责"判断是否触发预警"，预警中心负责"触发后如何处理"。

#### 4.1 预警规则设计原则

| 原则 | 说明 |
|------|------|
| **条件即规则** | 预警触发条件（阈值、公式）作为规则定义，享受版本管理、审批流程 |
| **计算即触发** | 规则计算引擎执行预警规则，判断是否满足触发条件 |
| **输出即事件** | 触发后输出标准化预警事件，推送给预警中心处理 |
| **只定义不处理** | 预警规则只定义"何时触发"，不定义"触发后通知谁、如何处理" |

#### 4.2 预警规则类型

| 规则类型 | 说明 | 应用场景 | 输出目标 |
|---------|------|---------|---------|
| **预算预警规则** | 预算执行率、预算偏差等触发条件 | 预算执行监控 | 预警中心 |
| **基线偏差预警规则** | 实际值与基线偏差率触发条件 | 经营分析、绩效跟踪 | 预警中心 |
| **KPI预警规则** | KPI达成率、趋势变化触发条件 | KPI监控、战略评估 | 预警中心 |
| **系统监控预警规则** | 系统性能、数据质量触发条件 | 系统运维 | 预警中心 |

#### 4.3 预警规则示例

```yaml
规则名称: 预算执行率预警规则
规则编码: ALERT-BUDGET-001
规则分类: 预警规则
规则类型: ALERT-BUDGET

触发条件:
  数据源: 预算执行数据
  计算表达式: budget_execution_rate = actual_spend / budget_amount * 100%

预警分级输出:
  黄色预警:
    触发条件: execution_rate >= 80%
    输出事件:
      alert_type: BUDGET_EXECUTION_YELLOW
      alert_name: 预算执行率黄色预警
      level: YELLOW
      message: "预算执行率已达${execution_rate}%"
      trigger_value: ${execution_rate}
      threshold: 80
      suggest_action: "关注支出进度"
      target_module: 预警中心
      target_dept: ${dept_id}
      rule_code: ALERT-BUDGET-001
  
  橙色预警:
    触发条件: execution_rate >= 90%
    输出事件:
      alert_type: BUDGET_EXECUTION_ORANGE
      alert_name: 预算执行率橙色预警
      level: ORANGE
      message: "预算执行率已达${execution_rate}%，即将超支"
      trigger_value: ${execution_rate}
      threshold: 90
      suggest_action: "提交超支说明"
      target_module: 预警中心
      target_dept: ${dept_id}
      rule_code: ALERT-BUDGET-001
  
  红色预警:
    触发条件: execution_rate >= 100%
    输出事件:
      alert_type: BUDGET_EXECUTION_RED
      alert_name: 预算执行率红色预警
      level: RED
      message: "预算已超支${execution_rate}%"
      trigger_value: ${execution_rate}
      threshold: 100
      suggest_action: "立即整改并提交说明"
      target_module: 预警中心
      target_dept: ${dept_id}
      rule_code: ALERT-BUDGET-001

输出格式规范:
  - 必须符合预警中心事件标准格式
  - 必须包含rule_code用于追溯
  - 必须包含trigger_value用于展示
```

#### 4.4 基线偏差预警规则示例

```yaml
规则名称: 营收基线偏差预警规则
规则编码: ALERT-BASELINE-001
规则分类: 预警规则
规则类型: ALERT-BASELINE

触发条件:
  数据源: 经营数据
  计算表达式: |
    deviation_rate = (actual_revenue - baseline_revenue) / baseline_revenue * 100%
    absolute_deviation = ABS(deviation_rate)

预警分级输出:
  关注:
    触发条件: absolute_deviation > 5% AND absolute_deviation <= 10%
    输出事件:
      alert_type: BASELINE_DEVIATION_ATTENTION
      level: YELLOW
      message: "营收偏差率${deviation_rate}%"
      
  预警:
    触发条件: absolute_deviation > 10% AND absolute_deviation <= 20%
    输出事件:
      alert_type: BASELINE_DEVIATION_WARNING
      level: ORANGE
      message: "营收偏差率${deviation_rate}%，需关注"
      
  严重:
    触发条件: absolute_deviation > 20%
    输出事件:
      alert_type: BASELINE_DEVIATION_CRITICAL
      level: RED
      message: "营收偏差率${deviation_rate}%，严重偏离基线"
```

#### 4.6 预警规则与预警中心的分工边界

| 职责 | 规则引擎中心 | 预警中心 |
|------|-------------|---------|
| **触发条件定义** | ✅ 阈值、公式、分级条件 | ❌ 不定义 |
| **触发判断** | ✅ 计算是否满足条件 | ❌ 不判断 |
| **事件生成** | ✅ 输出标准化预警事件 | ❌ 不生成 |
| **事件接收** | ❌ 不接收 | ✅ 接收并处理 |
| **分级确认** | ❌ 只输出建议分级 | ✅ 可调整分级 |
| **通知分发** | ❌ 不处理 | ✅ 通知谁、怎么通知 |
| **工单创建** | ❌ 不处理 | ✅ 创建跟踪工单 |
| **闭环跟踪** | ❌ 不处理 | ✅ 处理状态跟踪 |
| **规则优化反馈** | ✅ 接收处理结果优化规则 | ✅ 反馈处理结果 |

### 模块5: 规则计算引擎

> **定位**: 规则引擎中心的核心执行层，负责解析规则公式、触发计算、管理计算结果。

#### 4.1 模块职责
- **做什么：** 提供规则公式的解析、计算、结果缓存和查询
- **不做什么：** 不存储业务数据（只存储计算结果），不替代业务模块的数据处理

#### 4.2 计算触发器

| 触发方式 | 说明 | 示例 |
|---------|------|------|
| **定时触发** | 按预设时间周期自动触发 | 每月1日自动计算上月预算执行率 |
| **事件触发** | 业务事件发生时触发 | 预算申请提交时自动校验预算余额 |
| **手动触发** | 用户主动触发 | 财务经理手动测试规则计算结果 |
| **批量触发** | 批量数据处理时触发 | 年度预算编制时批量计算各部门预算建议 |

#### 4.3 公式解析器

**支持的公式语法：**

```
# 基础运算
预算营收 = 上年实际营收 * (1 + 行业增长率) * 市场系数

# 条件判断
绩效系数 = IF(绩效等级 == "S", 1.5, IF(绩效等级 == "A", 1.2, 1.0))

# 聚合函数
部门平均达成率 = AVG(员工达成率)

# 时间函数
同比增长率 = (本年值 - 上年同期值) / 上年同期值

# 引用外部参数
行业增长率 = @外部数据.行业增长率

# 引用其他规则
预算基线 = @规则.预算编制规则.输出.预算建议值
```

#### 4.4 计算结果管理

| 功能 | 说明 |
|------|------|
| 结果缓存 | 计算结果缓存，避免重复计算 |
| 结果查询 | 按规则、时间、维度查询历史计算结果 |
| 结果导出 | 导出计算明细为Excel/PDF |
| 结果对比 | 对比不同时间/版本的计算结果 |

### 5.2 关键数据表设计

#### 规则定义表 (rule_definition)

```sql
CREATE TABLE rule_definition (
    rule_id         BIGINT PRIMARY KEY AUTO_INCREMENT COMMENT '规则ID',
    rule_code       VARCHAR(64) NOT NULL UNIQUE COMMENT '规则编码，全局唯一',
    rule_name       VARCHAR(128) NOT NULL COMMENT '规则名称',
    category_id     BIGINT NOT NULL COMMENT '所属分类ID',
    rule_type       VARCHAR(32) NOT NULL COMMENT '规则类型：BUDGET/BASELINE/ACCOUNTING/BONUS/COMMISSION/SPECIAL',
    sub_type        VARCHAR(32) COMMENT '子类型：如BUDGET-CTRL/BUDGET-ALLOC等',
    current_version VARCHAR(16) DEFAULT '1.0.0' COMMENT '当前生效版本号',
    status          VARCHAR(16) DEFAULT 'DRAFT' COMMENT '状态：DRAFT/UNDER_REVIEW/APPROVED/PUBLISHED/DEPRECATED',
    owner           VARCHAR(64) COMMENT '规则负责人',
    effective_date  DATE COMMENT '生效日期',
    expiry_date     DATE COMMENT '失效日期',
    description     TEXT COMMENT '规则说明',
    org_id          BIGINT COMMENT '组织ID（用于多租户隔离）',
    created_by      VARCHAR(64) NOT NULL,
    created_at      DATETIME DEFAULT CURRENT_TIMESTAMP,
    updated_by      VARCHAR(64),
    updated_at      DATETIME ON UPDATE CURRENT_TIMESTAMP,
    
    INDEX idx_category (category_id),
    INDEX idx_type (rule_type, sub_type),
    INDEX idx_status (status),
    INDEX idx_org (org_id)
) COMMENT='规则定义主表';
```

#### 规则版本表 (rule_version)

```sql
CREATE TABLE rule_version (
    version_id      BIGINT PRIMARY KEY AUTO_INCREMENT COMMENT '版本ID',

    content_json    JSON NOT NULL COMMENT '规则内容（JSON格式）',
```

## 六、非功能需求

### 6.1 性能要求

| 指标 | 要求 | 说明 |
|------|------|------|
| 规则计算响应时间 | < 500ms | 单次规则计算（含参数获取） |
| 批量计算吞吐量 | > 1000条/秒 | 批量预算编制场景 |
| 规则查询响应时间 | < 200ms | 规则列表/详情查询 |
| 并发计算支持 | > 100并发 | 月末/年末集中计算场景 |

### 6.2 安全要求

- 规则编辑权限按角色控制（财务规则只能财务角色编辑）
- 规则变更必须审批，审批记录不可篡改
- 规则执行日志保留5年，满足审计要求
- 敏感规则参数（如提成比例）加密存储

### 6.3 可用性要求

- 规则计算服务可用性 > 99.9%
- 支持规则热更新（不重启服务）
- 规则执行失败时自动重试（最多3次），失败时通知管理员

### 6.4 扩展性要求

- 支持新增规则类型无需修改代码（通过配置扩展）
- 支持自定义公式函数（插件化扩展）
- 支持多租户隔离（不同组织使用独立规则集）
