## 一、设计思维与方法论

### 1.1 核心原则

基于 software-module-design 的三大原则：

| 原则 | 含义 | 在本产品中的体现 |
|------|------|----------------|
| **单一职责** | 一个模块只做一类事 | 会议列表只读展示、通知体系只负责推送、评估只负责评分 |
| **高内聚** | 模块内部功能紧密相关 | 决议中心模块：决议记录+状态管理+KMS关联+推送，全围绕「决议闭环」 |
| **低耦合** | 模块之间通过接口通信 | 各模块通过 Meeting ID 关联，不直接操作对方数据；通知体系通过配置化消息模板解耦 |

### 1.2 文档演进说明

本文档由 `本部重点工作经营例会-功能设计文档.md`（v2.0，状态"设计中"）演进而来：

- **v1.0**（2026-05-18）：基于 frontend-design-SKILL 创建初版
- **v2.0**（2026-05-18）：重构整合功能设计 + 前端设计，状态"设计中"
- **v3.0**（2026-06-04）：**重大重构**——基于真实代码实现更新，建立与 PRD/RFC 的双向引用关系，区分「已落地」与「未来规划」

> v2.0 中规划的 **KMS 同步模块、会议记录生成器模块、批量解析模块** 在当前实现中已降级为「未来规划」，因产品方向从"KMS 双向同步"转向"DSTE 平台内闭环 + 企业微信通知推送"。

## 二、需求收集（用户故事）

```
作为 会议组织者
我希望 查看所有历史会议的时间线和关联关系
以便 快速定位某次会议的议题和决策

作为 会议组织者
我希望 按场景（片联/本部/战区/落后整改）筛选会议
以便 聚焦特定层级的经营分析会

作为 会议记录人
我希望 在会中记录决议并关联 KMS 文档
以便 会后决议有据可查、可推送通知

作为 决策执行人
我希望 查看分配给自己的行动项清单
以便 按时完成任务

作为 战略管理员
我希望 对已完成的会议进行效果评估
以便 量化会议质量、持续改进

作为 会议主持人
我希望 一键将决议或待办推送到企业微信群
以便 相关方及时收到通知

作为 高管
我希望 在日历视图中查看所有会议安排
以便 统筹时间、识别冲突
```

### M1 会议管理模块

**1. 模块职责**
- **做什么**：管理会议的生命周期（创建、编辑、删除、查看），维护会议的基本信息、议程、会议链关联
- **不做什么**：不追踪行动项进度（M2 负责）、不记录决议详情（M3 负责）、不生成推送消息（M5 负责）

**2. 涉及的数据实体**

| 数据名称 | 说明 | 示例 |
|---------|------|------|
| Meeting | 一次会议记录 | 见「六、数据模型」 |
| AgendaItem | 会议议程项 | `{ type: 'budget_finance', title: 'Q1 财务复盘', duration: 45, owner: '财务部', material_link: 'https://...' }` |

**3. 功能列表**

| 功能名称 | 状态 | 用户故事 | 输入 | 输出 | 代码位置 |
|---------|------|---------|------|------|---------|
| 创建会议 | ✅ | 组织者希望创建下次会议 | 日期、标题、场景、层级、议程列表 | 新 Meeting 对象 | `openMeetingEditor()` + `saveMeeting()` |
| 编辑会议 | ✅ | 组织者希望修改会议信息 | Meeting ID、新字段值 | 更新后的 Meeting | `renderMeetingEditPage()` |
| 删除会议 | ✅ | 组织者希望删除错误创建的会议 | Meeting ID | 删除成功/失败 | `deleteMeeting()` |
| 查看会议列表 | ✅ | 组织者希望浏览所有历史会议 | 筛选条件（月份/状态/场景/流程步骤） | 会议卡片列表 | `renderMeetings()` |
| 查看会议详情 | ✅ | 组织者希望查看某次会议的完整信息 | Meeting ID | 详情浮层（信息面板+标签页） | `renderMeetingDetail()` |
| 会议链关联 | ✅ | 用户希望看到会议之间的上下游关系 | 上游/下游 Meeting ID | 关联链接 | `upstreamMeeting` / `downstreamMeeting` |
| 智能填充 | ✅ | 系统根据会议名称自动推荐场景和日期 | 会议标题文本 | 推荐字段值 | `autoFillMeetingForm()` |
| 克隆会议 | ✅ | 快速复制已有会议结构 | Meeting ID | 克隆后的新 Meeting | `cloneMeeting()` |
| 日历视图切换 | ✅ | 用户在列表和日历视图间切换 | 视图模式 | 对应视图渲染 | `toggleMeetingsView()` |

**4. 与其他模块的关系**

```
M1 会议管理
├── 依赖：
│   ├── M2 行动项追踪：获取该会议关联的行动项
│   ├── M3 决议中心：获取该会议的决议列表
│   └── M4 评估模块：获取会议效果评分
├── 被依赖：
│   ├── M6 日历视图：获取会议时间数据
│   ├── M7 一报一会：获取会议流程状态
│   ├── M8 场景指标卡：获取场景分布统计
│   └── M5 通知体系：推送会议相关通知
└── 输出：
    └── 会议数据 → 所有下游模块
```

**5. 关键规则**
- Meeting ID 使用日期格式 `YYYYMMDD`，同一天只能有一个会议
- 删除会议时需二次确认，级联删除关联数据（行动项、决议）
- 会议场景必须为五类之一：`union_quarterly` / `hq_routine` / `region_routine` / `lagging_region` / `lagging_vertical`
- 议程项时长必须为正整数（分钟）
- 场景与层级有隐式映射关系：片联→L1，本部/战区常规→L2，落后整改→L3

### M2 行动项追踪模块

**1. 模块职责**
- **做什么**：追踪会议产生的行动项全生命周期（创建、状态更新、逾期检测）
- **不做什么**：不分配行动项（在会议编辑/详情页中分配）、不自动通知责任人（M5 负责）

**2. 涉及的数据实体**

| 数据名称 | 说明 | 示例 |
|---------|------|------|
| ActionItem | 一个行动项 | `{ id: 'A001', content: 'Q2 营收目标上调至 3.2 亿', owner: '李经理', deadline: '2026-04-30', status: 'in_progress' }` |

**3. 功能列表**

| 功能名称 | 状态 | 用户故事 | 输入 | 输出 | 代码位置 |
|---------|------|---------|------|------|---------|
| 添加行动项 | ✅ | 组织者希望分配新任务 | 内容、责任人、截止日期 | 新 ActionItem | 编辑页 `addActionItem()` |
| 状态更新 | ✅ | 责任人希望标记任务状态 | ActionItem ID、新状态 | 更新后的 ActionItem | 编辑页渲染 |
| 删除行动项 | ✅ | 组织者希望删除错误任务 | ActionItem ID | 删除成功 | 编辑页渲染 |
| 逾期检测 | ✅ | 系统需要自动标记逾期任务 | 当前日期 | 待办提醒面板展示 | `renderMeetings()` 待办提醒区 |
| 推送提醒 | ✅ | 组织者希望提醒责任人 | ActionItem + Webhook | 企微消息 | `pushTodoReminder()` |

**4. 关键规则**
- 截止日期 < 今天 且 状态 ≠ "completed" → 在待办提醒面板中展示为逾期
- 删除行动项时不需要级联删除会议
- 同一会议下允许内容相同的行动项（实际业务中可能出现）

### M3 决议中心模块

**1. 模块职责**
- **做什么**：结构化记录会议中的关键决议，支持状态管理、KMS 关联、企业微信推送
- **不做什么**：不做决议审批流（当前不设多级审批）、不自动执行决议

**2. 涉及的数据实体**

| 数据名称 | 说明 | 示例 |
|---------|------|------|
| Decision | 一条决议记录 | `{ id: 'D001', content: 'Q2 营收目标 3.2 亿', owner: '李经理', deadline: '2026-04-30', status: 'approved', kmsUrl: 'https://kms...' }` |

**3. 功能列表**

| 功能名称 | 状态 | 用户故事 | 输入 | 输出 | 代码位置 |
|---------|------|---------|------|------|---------|
| 记录决议 | ✅ | 组织者希望记录会议中的决议 | 内容、负责人、截止日期、状态 | 新 Decision | 编辑页 `addDecisionItem()` |
| 编辑决议 | ✅ | 组织者希望修改决议信息 | Decision ID、新字段值 | 更新后的 Decision | 编辑页渲染 |
| 删除决议 | ✅ | 组织者希望删除错误记录 | Decision ID | 删除成功 | 编辑页渲染 |
| KMS 关联 | ✅ | 决议需要关联 KMS 文档 | KMS URL | 可点击跳转的链接 | `kmsUrl` 字段 |
| 决议推送 | ✅ | 组织者希望将决议推送到群 | Decision + Webhook | 企微 markdown 消息 | `pushResolution()` |
| 决议中心抽屉 | ✅ | 用户希望查看全部决议 | 筛选条件（状态/搜索词） | 全量决议列表 | `openDecisionsDrawer()` |
| 状态过滤 | ✅ | 用户希望按状态查看决议 | 状态筛选 | 过滤后的决议列表 | `filterDecisions()` |

**4. 关键规则**
- 决议状态：`approved`（已批准）/ `implemented`（已执行）/ `pending`（待审批）/ `rejected`（已驳回）
- 每条决议必须有负责人和截止日期（遵循 PRD 6.9 行动项规范）
- 决议可关联 KMS 文档，实现"决议→文档"的可追溯链路

### M4 会议效果评估模块

**1. 模块职责**
- **做什么**：对已完成的会议进行四维度量化评估，支持 AI 自动推荐分 + 人工调整
- **不做什么**：不评估未完成的会议、不生成评估报告（未来规划）

**2. 涉及的数据实体**

| 数据名称 | 说明 | 示例 |
|---------|------|------|
| Effectiveness | 会议效果评估 | `{ overallScore: 92, dimensions: { preparation: 95, discussion: 90, decision: 95, execution: 88 }, feedback: ['议程合理', '数据充分'], comment: '...', evaluatedAt: '2026-06-10T08:00:00Z' }` |

**3. 功能列表**

| 功能名称 | 状态 | 用户故事 | 输入 | 输出 | 代码位置 |
|---------|------|---------|------|------|---------|
| AI 自动评分 | ✅ | 系统基于会议数据自动计算推荐分 | Meeting 对象 | 四维度评分 + 反馈标签 | `calculateAutoScore()` |
| 评估浮层 | ✅ | 参会人希望对会议进行评分 | Meeting ID | 评估表单（滑块+标签+评语） | `openMeetingEvalModal()` |
| 保存评估 | ✅ | 用户希望保存评估结果 | 评估表单数据 | 持久化的 Effectiveness | `saveMeetingEvaluation()` |
| 详情页展示 | ✅ | 用户希望查看评估结果 | Effectiveness 数据 | 综合分 + 进度条 + 标签 | `renderMeetingDetail()` eval section |
| 列表状态展示 | ✅ | 用户希望快速识别已评估会议 | Effectiveness 存在性 | ⭐ 分数+等级 或「未评估」 | 会议卡片标签页 |

**4. 评分算法说明**

```
会前准备 = min(100, 材料准时率 × 0.8 + 预评审完成 ? 20 : 0)
会议讨论 = 60 + 已召开?10 + 有议程?10 + 有纪要?10 + 纪要已定稿?10
决议质量 = min(100, 决议及时率 × 0.6 + KMS关联率 × 20 + 审批通过率 × 20)
执行落地 = min(100, 行动闭环率 × 0.8 + 已追踪?20 : 0)
综合评分 = (会前准备 + 会议讨论 + 决议质量 + 执行落地) / 4
```

系统反馈标签自动生成（如「材料充分」「数据不足」「议程合理」「时间失控」等）。

### M5 企业微信通知体系

**1. 模块职责**
- **做什么**：通过企业微信机器人 Webhook 推送会议相关通知（决议、待办、预警）
- **不做什么**：不替代企业微信的原生功能、不做消息存储（仅保留最近20条发送记录）

**2. 涉及的数据实体**

| 数据名称 | 说明 | 示例 |
|---------|------|------|
| NotificationConfig | 通知配置 | `{ webhooks: [{id, name, url}], enabledTypes: {resolution, todo, alert}, mentionAll, lastSent: [] }` |

**3. 功能列表**

| 功能名称 | 状态 | 用户故事 | 输入 | 输出 | 代码位置 |
|---------|------|---------|------|------|---------|
| 多 Webhook 配置 | ✅ | 管理员希望配置多个推送目标群 | Webhook URL + 群名称 | 配置持久化 | `openNotificationConfig()` |
| 消息模板渲染 | ✅ | 系统根据类型生成标准消息 | 类型 + 数据 | Markdown 格式消息 | `renderNotificationMessage()` |
| 决议推送 | ✅ | 组织者希望推送决议到群 | Decision + Webhook | 企微消息 | `pushResolution()` |
| 待办提醒 | ✅ | 组织者希望提醒责任人 | ActionItem + Webhook | 企微消息 | `pushTodoReminder()` |
| 会议预警 | ✅ | 系统希望提醒流程逾期 | Meeting + 逾期步骤 | 企微消息 | `pushAlert()` |
| 通知中心 | ✅ | 用户希望查看发送历史 | 配置中的 lastSent | 历史记录列表 | `openNotificationCenter()` |
| Webhook 选择器 | ✅ | 多群时选择推送目标 | Webhook 列表 | 选中的 Webhook | `openWebhookSelector()` |

**4. 关键规则**
- 兼容旧配置：单 `webhookUrl` 自动迁移为 `webhooks` 数组
- 消息格式：企业微信 markdown，支持 `<font color="info">`（蓝）/ `comment`（灰）/ `warning`（橙）
- 仅允许转发到 `qyapi.weixin.qq.com` 域名的 Webhook URL
- 发送记录保留最近 20 条，存储于 `localStorage`

### M6 日历视图模块

**1. 模块职责**
- **做什么**：以月历/年历双模式展示会议安排，支持当日浮层、筛选联动
- **不做什么**：不做日程冲突检测、不做会议预约

**2. 功能列表**

| 功能名称 | 状态 | 用户故事 | 输入 | 输出 | 代码位置 |
|---------|------|---------|------|------|---------|
| 月历视图 | ✅ | 用户希望按月查看会议分布 | 年份、月份 | 日期网格 + 会议条 | `renderCalendarMonth()` |
| 年历视图 | ✅ | 用户希望概览全年会议 | 年份 | 12个月迷你网格 | `renderCalendarYear()` |
| 当日浮层 | ✅ | 用户希望查看某天所有会议 | 日期 | 会议列表浮层 | 月历渲染中 |
| 视图切换 | ✅ | 用户在列表和日历间切换 | 按钮点击 | 对应视图显示/隐藏 | `toggleMeetingsView()` |
| 筛选联动 | ✅ | 筛选条件同时影响两个视图 | 筛选状态 | 过滤后的会议数据 | `applyFilters()` |
| 今天高亮 | ✅ | 用户快速定位当前日期 | 当前日期 | 高亮样式 | 日历渲染中 |

**3. 关键规则**
- 使用 `localStorage` 记录用户上次选择的视图模式（`dste_meetings_view: 'list' | 'calendar'`）
- 日历视图只显示符合当前筛选条件的会议
- 年历视图中，有会议的月份显示该月首场会议的场景颜色

### M7 一报一会流程模块

**1. 模块职责**
- **做什么**：为战区月度经营分析会、落后战区/垂直客群经分会提供"一报一会"流程跟踪（报告生成→预评审→召开→纪要起草→纪要定稿→行动项追踪）
- **不做什么**：不自动生成经营报告（未来规划）、不替代材料审核流程

**2. 流程步骤**

```
报告生成 → 会前评审 → 会议召开 → 纪要起草 → 纪要定稿 → 行动项追踪
```

**3. 功能列表**

| 功能名称 | 状态 | 用户故事 | 代码位置 |
|---------|------|---------|---------|
| 流程进度条 | ✅ | 用户希望查看会议流程完成度 | `renderPipeline()` |
| 步骤状态切换 | ✅ | 用户希望标记某步骤完成/未完成 | `togglePipelineStep()` |
| 前置依赖检查 | ✅ | 防止跳步完成 | 切换时检查前序步骤 |
| 后续级联取消 | ✅ | 取消某步骤时自动取消后续步骤 | 取消逻辑中的循环处理 |
| 一报一会适用范围控制 | ✅ | 仅特定场景启用 | `SCENARIO_CONFIG[].pipelineEnabled` |

**4. 关键规则**
- 仅 `region_routine`、`lagging_region`、`lagging_vertical` 三个场景启用一报一会流程
- 标记完成某步骤时，若前序步骤未完成，需弹窗确认
- 取消某步骤时，若后续有已完成步骤，需确认并级联取消

## 四、数据模型

### 4.1 Meeting（会议）

```typescript
interface Meeting {
  // === 基础信息 ===
  id: string;                    // 唯一标识，如 "20260415"（日期格式）
  title: string;                 // 会议名称
  date: string;                  // 会议日期，如 "2026-04-15"
  month: string;                 // 所属月份，如 "2026-04"
  location: string;              // 地点
  host: string;                  // 主持人
  recorder: string;              // 记录人
  status: 'planned' | 'in_progress' | 'completed' | 'cancelled';
  
  // === 场景与层级 ===
  scenario: 'union_quarterly' | 'hq_routine' | 'region_routine' | 'lagging_region' | 'lagging_vertical';
  level: 'L1' | 'L2' | 'L3';
  scope: string;                 // 范围标识，如 'union' | 'hq' | 'region_east'
  
  // === 关联 ===
  meeting_link: string;          // KMS 会议主页链接
  upstreamMeeting: string | null;// 上游会议 ID
  downstreamMeeting: string | null; // 下游会议 ID
  
  // === 一报一会流程 ===
  pipeline: {
    reportGenerated: boolean;    // 经营报告已生成
    preReviewDone: boolean;      // 会前评审已完成
    meetingHeld: boolean;        // 会议已召开
    minutesDrafted: boolean;     // 纪要已起草
    minutesApproved: boolean;    // 纪要已定稿
    actionsTracked: boolean;     // 行动项已追踪
  };
  
  // === 议程 ===
  agenda_items: AgendaItem[];
  
  // === 行动项 ===
  actions: ActionItem[];
  
  // === 决议 ===
  decisions: Decision[];
  
  // === 纪要 ===
  hasMinutes: boolean;           // 是否有纪要
  minutesStatus: 'final' | 'draft' | null; // 纪要状态
  minutes_content: string;       // 纪要正文
  
  // === 效率指标 ===
  metrics: {
    materialTimeliness: number;  // 材料准备及时率（0-100）
    resolutionTimeliness: number;// 决议录入及时率（0-100）
    actionClosure: number;       // 行动项闭环率（0-100）
    satisfaction: number;        // 会议满意度（0-5）
  };
  
  // === 效果评估 ===
  effectiveness: Effectiveness | null;
}

interface AgendaItem {
  type: 'budget_finance' | 'goal_management' | 'key_task_management' | 'business_special' | 'risk_management' | 'personnel_review';
  title: string;
  duration: number;              // 分钟
  owner: string;                 // 负责人/主讲人
  material_link?: string;        // 材料链接（可选）
  purpose?: string;              // 议题目的（可选）
  start_time?: string;           // 开始时间，如 "09:00"
  end_time?: string;             // 结束时间，如 "09:30"
}

interface ActionItem {
  id: string;                    // 如 "A001"
  content: string;               // 内容
  owner: string;                 // 负责人
  deadline: string;              // 截止日期
  status: 'pending' | 'in_progress' | 'completed';
}

interface Decision {
  id: string;                    // 如 "D001"
  content: string;               // 决议内容
  owner: string;                 // 负责人
  deadline: string;              // 截止日期
  status: 'approved' | 'implemented' | 'pending' | 'rejected';
  kmsUrl?: string;               // 关联 KMS 文档链接
}

interface Effectiveness {
  overallScore: number;          // 综合评分（0-100）
  dimensions: {
    preparation: number;         // 会前准备（0-100）
    discussion: number;          // 会议讨论（0-100）
    decision: number;            // 决议质量（0-100）
    execution: number;           // 执行落地（0-100）
  };
  feedback: string[];            // 反馈标签，如 ['议程合理', '数据充分']
  comment?: string;              // 文字评价
  evaluatedAt?: string;          // 评估时间，ISO 格式
  auto?: boolean;                // 是否为系统自动推荐分
}
```

## 五、核心交互流程

### 流程1：完整的会议生命周期（从安排到闭环）

```
开始
  │
  ▼
┌─────────────┐
│  新建/编辑会议 │ ← 填写表单 → 智能填充自动推荐字段
└──────┬──────┘
       │
       ▼
┌─────────────┐
│  会前准备      │ ← 一报一会流程：报告生成 → 预评审
└──────┬──────┘
       │
       ▼
┌─────────────┐
│  召开会议      │ ← 标记 meetingHeld = true
└──────┬──────┘
       │
       ▼
┌─────────────┐
│  记录决议+行动项│ ← 编辑页添加决议/行动项
└──────┬──────┘
       │
       ▼
┌─────────────┐
│  纪要撰写      │ ← 编辑 minutes_content
└──────┬──────┘
       │
       ▼
┌─────────────┐
│  效果评估      │ ← AI 自动推荐分 → 人工调整 → 保存
└──────┬──────┘
       │
       ▼
┌─────────────┐
│  推送通知      │ ← 决议/待办 → 企业微信群
└──────┬──────┘
       │
       ▼
┌─────────────┐
│  行动项闭环    │ ← 追踪完成状态 → 下次会议复盘
└──────┬──────┘
       │
       ▼
      结束
```

### 流程2：企业微信推送流程

```
用户点击推送按钮（决议/待办/预警）
  │
  ▼
加载通知配置（localStorage）
  │
  ├── 无 Webhook → 提示配置
  ├── 1 个 Webhook → 直接发送
  └── 多个 Webhook → 打开选择器
              │
              ▼
        用户选择目标群
              │
              ▼
        渲染 Markdown 消息
              │
              ▼
        调用 API /api/notify/proxy
              │
              ▼
        Worker 转发到企业微信
              │
              ▼
        保存发送记录 → 返回结果
              │
              ▼
        用户收到成功/失败提示
```

### 流程3：AI 自动评分流程

```
会议标记为 completed
  │
  ▼
用户打开评估浮层
  │
  ▼
系统调用 calculateAutoScore(meeting)
  │
  ├── 提取 pipeline 状态
  ├── 提取 metrics 数据
  ├── 统计决议 KMS 关联率
  ├── 统计决议审批通过率
  └── 计算四维度得分
              │
              ▼
        生成系统反馈标签
              │
              ▼
        预填评估表单（AI 推荐标记）
              │
              ▼
        用户可调整分数/标签/评语
              │
              ▼
        保存 → 持久化到 meeting.effectiveness
```

## 八、技术实现路径

### 8.1 当前实现（MVP，v0.4.0）

| 层级 | 实现 |
|------|------|
| 前端 | 单页 HTML（`src/meetings.html`，~3459行）+ CSS 变量 + 原生 JS |
| 路由 | 基于 hash 的伪路由（`#exe/meetings`, `#exe/meetings/edit`） |
| 状态管理 | 全局 `window._xxx` 变量 |
| 数据存储 | Cloudflare KV（整存整取 JSON 数组）+ `localStorage` 缓存 |
| API | Cloudflare Worker（`api-worker/worker.js`，~328行） |
| 通知 | Worker `/api/notify/proxy` 转发企业微信 Webhook |
| 认证 | CAS 单点登录 + JWT Token |
| 构建 | Vite + Playwright E2E 测试 |

### 8.2 未来演进路线

| 阶段 | 版本 | 目标 | 关键任务 |
|------|------|------|---------|
| **P1** | v0.5.0 | 体验优化 | 决议中心增强（批量操作、导出）、评估报告生成、会议纪要 AI 辅助撰写 |
| **P2** | v0.6.0 | 智能增强 | AI 纪要生成、智能报告生成、行动项跟踪看板（独立页面）、议程智能推荐 |
