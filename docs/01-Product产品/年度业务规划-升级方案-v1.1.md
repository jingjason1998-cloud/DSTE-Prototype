# 年度业务规划 — 升级设计方案 v1.1

> **文档版本**: v1.1  
> **更新日期**: 2026-06-12  
> **设计方向**: 从"年度经营计划"升级为"年度业务规划"  
> **目标版本**: v0.6.0 ~ v0.7.0  

---

## 一、背景与定位

### 1.1 为什么需要升级

当前 `bp/annual-plan` 页面已经实现了 KPI 总览、分解视图、重点工作管理等核心能力，但本质上仍是一个**目标录入和分解工具**，缺少年度业务规划的完整闭环：

| 现状问题 | 升级方向 |
|---------|---------|
| 目标手动录入，与 SP 输出脱节 | 从战略地图、战略专题、VDBD 自动承接目标与任务 |
| 只关注 KPI 数字分解 | 增加业务假设、资源预算、风险预案、能力建设的系统规划 |
| 缺少规划过程管理 | 增加规划阶段流转：准备 → 制定 → 评审 → 发布 → 执行 |
| 职能线/画布视图未实现 | 完善多视角分解，并增加规划评审与共识机制 |
| 发布到执行是演示态 | 实现真实状态流转，同步到组织绩效管理和重点工作 |

### 1.2 升级后的定位

**年度业务规划（Annual Business Planning, ABP）** 是连接 SP（战略规划）与 EXE（战略执行）的核心桥梁：

```
SP 战略规划
    ├── 战略意图（3年目标）
    ├── 战略地图（BSC 四维度目标）
    ├── 战略专题（关键战略举措）
    └── VDBD 价值驱动业务设计
            ↓ 只读引用 / 自动导入
BP 年度业务规划
    ├── 年度目标制定
    ├── KPI 组织分解
    ├── 重点工作规划
    ├── 资源配置（预算/人力）
    ├── 风险与假设
    └── 规划评审
            ↓ 发布
EXE 战略执行
    ├── 组织绩效管理
    ├── 重点工作管理
    └── 经营分析会
```

## 三、功能模块设计

### 3.1 模块总览

```
📅 年度业务规划
│
├── 📋 规划总览        ← 规划仪表盘 + 阶段状态 + 关键指标
├── 🎯 目标承接        ← 从 SP 导入战略目标 + 专题 + VDBD 关键任务
├── 📊 KPI 总览        ← 年度 KPI 目标表（现有增强）
├── 🔀 分解视图        ← 组织/职能/产品多维度分解（现有增强）
├── 🏢 职能线评价      ← 同一 KPI 在不同职能线的权重（PRD 待实现）
├── 🎨 画布视图        ← 树形可视化 + 分组框（PRD 待实现）
├── 📌 重点工作规划    ← 重点工作 CRUD + KPI 双向关联
├── 💰 资源配置        ← 预算分配 + 人力需求 + 资源冲突检查
├── ⚠️ 风险与假设      ← 关键业务假设 + 风险预案
├── ✅ 规划评审        ← 检查清单 + 评审会议 + 版本发布
└── 🚀 发布执行        ← 发布到 OMP + 重点工作 + 经营分析会
```

### 3.2 规划总览（新增）

**定位**：年度业务规划的入口页面，展示规划全貌和阶段状态。

**核心功能**：
1. **规划阶段管线**
   - 5 个阶段：准备 → 目标制定 → 分解与资源配置 → 评审 → 发布
   - 每个阶段状态：未开始 / 进行中 / 已完成
   - 阶段间依赖关系

2. **关键指标卡片**
   - 年度 KPI 数量
   - 已分解 KPI 数量 / 分解完成率
   - 重点工作数量
   - 预算分配完成率
   - 风险待确认项

3. **规划健康度评分**
   - 综合评分 0-100
   - 评分维度：目标完整性、分解合理性、资源匹配度、风险可控度

4. **版本信息**
   - 当前规划版本
   - 上一周期对比
   - 规划发布倒计时

**页面布局**：
```
┌────────────────────────────────────────────────────────────┐
│ 年度业务规划 — 2026 营销线                  [发布到执行]   │
├────────────────────────────────────────────────────────────┤
│ 规划阶段：准备 ✅  →  目标制定 🔄  →  分解配置 ⏳  →  评审 ⏳  →  发布 ⏳  │
├────────────────────────────────────────────────────────────┤
│ ┌─────────┐ ┌─────────┐ ┌─────────┐ ┌─────────┐          │
│ │ KPI 25  │ │ 已分解  │ │ 重点    │ │ 预算    │          │
│ │         │ │ 18 / 72%│ │ 工作 12 │ │ 完成 65%│          │
│ └─────────┘ └─────────┘ └─────────┘ └─────────┘          │
├────────────────────────────────────────────────────────────┤
│ 规划健康度: 68/100                                        │
│ [████████████░░░░░░░░] 目标完整 80 | 分解合理 70 | 资源匹配 55 | 风险可控 65 │
└────────────────────────────────────────────────────────────┘
```

### 3.3 目标承接（新增）

**定位**：从 SP 输出自动导入年度业务规划的输入，避免手动录入和口径不一致。

**核心功能**：
1. **从战略地图导入**
   - 选择战略地图和年度（如 2026）
   - 自动读取该年度里程碑目标
   - 映射为年度 KPI 候选
   - 用户确认后生成 KPI 实例

2. **从战略专题导入**
   - 选择当年度战略专题
   - 专题自动生成重点工作候选
   - 关联到对应的 BSC 维度

3. **从 VDBD 导入**
   - 读取 VDBD 价值驱动业务设计中的关键任务
   - 关键任务转化为重点工作
   - 价值量化数据带入资源配置

4. **导入映射规则**
   - 战略目标 → 年度 KPI
   - 战略专题 → 重点工作
   - VDBD 关键任务 → 重点工作
   - 业务设计价值量化 → 预算/收入预估

5. **差异提醒**
   - SP 有但年度规划未承接的目标
   - 年度规划新增但 SP 未覆盖的目标
   - 目标值不一致提醒

**数据模型**：
```typescript
interface AbpImportLog {
  id: string;
  cycleId: string;
  sourceType: 'strategy_map' | 'strategy_topic' | 'vdbd';
  sourceId: string;
  sourceName: string;
  importedAt: string;
  importedBy: string;
  mappings: {
    sourceItemId: string;
    sourceItemName: string;
    targetType: 'kpi' | 'key_task';
    targetId: string;
    autoMapped: boolean;
    confirmed: boolean;
  }[];
}
```

### 3.4 KPI 总览（现有增强）

**定位**：年度 KPI 目标的总表，复刻 KMS 组织绩效总目标表。

**核心功能（在现有基础上增强）**：
1. **表格展示（已有）**
   - 牵引点 × KPI指标 × 全年目标 × H1占比 × 权重 × 三档目标

2. **KTI 挂载（新增）**
   - 每个结果 KPI 可挂载多个过程指标（KTI）
   - KTI 来自战略指标库（indicatorType = process）
   - 显示 KTI 与结果 KPI 的关联关系

3. **计分规则引擎（新增）**
   - 支持简单规则：达成率线性计分
   - 支持复杂规则：阈值区间计分、同比计分、排名计分
   - 规则可视化预览

4. **KPI 关联重点工作（新增）**
   - 在 KPI 行展示关联重点工作数量
   - 点击可查看/编辑关联

5. **批量操作（新增）**
   - 批量设置 H1 占比
   - 批量设置三档比例
   - 批量设置计分规则

**KTI 挂载示例**：
```
结果 KPI: 销售额-D (178,623万)
├─ KTI: 有效商机数 ≥ 1,200 个
├─ KTI: 大客户拜访次数 ≥ 500 次
└─ KTI: 赢单率 ≥ 35%
```

### 3.5 分解视图（现有增强）

**定位**：按组织、职能、产品等维度分解年度目标。

**核心功能（在现有基础上增强）**：
1. **多维度切换（已有）**
   - 战区、行业、产品、部门

2. **分解树展示（新增）**
   - 层级展开/收起
   - 父目标与子目标的联动
   - 子目标合计与父目标对比（差异高亮）
   - 支持两种树形模式切换：
     - **组织分解树**：按组织/职能/产品层级展示
     - **BSC 因果支撑树**：按学习成长 → 内部流程 → 客户 → 财务的因果链展示

3. **BSC 支撑关联展示（新增）**
   - 在指标树中高亮 BSC 维度颜色
   - 用连线展示 KPI 之间的支撑关系
   - 支持查看某 KPI 的上游支撑 KPI 和下游被支撑 KPI
   - 自动识别断链（某财务 KPI 没有内部流程/客户 KPI 支撑）
   - 自动识别孤立的 KPI（无支撑也无被支撑）

4. **分解模板（新增）**
   - 按历史数据自动推荐分解比例
   - 按组织能力模型推荐分解
   - 自定义模板保存

4. **权重校验增强（增强）**
   - 目标值合计校验
   - 权重合计校验
   - 差异容忍度设置

5. **分解审批（新增）**
   - 子目标提交后审批
   - 审批意见记录
   - 审批通过后锁定

### 3.6 职能线评价（PRD 待实现）

**定位**：解决同一 KPI 在不同职能线中权重不同的问题。

**核心功能**：
1. **职能线配置**
   - 预置职能线：销售、售前、项目、市场、运营
   - 自定义职能线

2. **KPI × 职能线权重矩阵**
   - 行：KPI
   - 列：职能线
   - 单元格：该 KPI 在该职能线的权重

3. **权重校验**
   - 每个职能线内 KPI 权重合计 = 100%
   - 每个 KPI 在各职能线权重可独立配置

4. **职能线视角切换**
   - 销售线视角：只看销售线负责的 KPI 和权重
   - 售前线视角：只看售前线负责的 KPI 和权重

**页面布局**：
```
┌────────────────────────────────────────────────────────────────────┐
│ 职能线评价                                                          │
├────────────────────────────────────────────────────────────────────┤
│        │ 销售职能 │ 售前职能 │ 项目职能 │ 市场职能 │ 运营职能 │    │
│ ───────┼─────────┼─────────┼─────────┼─────────┼─────────┼────   │
│ 销售额-D│   10    │    5    │    2    │    0    │    0    │ 17    │
│ 回款额-K│   25    │    5    │    5    │    0    │    0    │ 35    │
│ 利润率  │    5    │    3    │    5    │    0    │    0    │ 13    │
│ ARR    │    8    │    5    │    0    │    5    │    0    │ 18    │
│ ───────┼─────────┼─────────┼─────────┼─────────┼─────────┼────   │
│ 合计    │   48    │   18    │   12    │    5    │    0    │       │
└────────────────────────────────────────────────────────────────────┘
```

**数据模型**：
```typescript
interface FunctionWeight {
  id: string;
  cycleId: string;
  kpiInstanceId: string;
  functionLine: string;        // 职能线编码
  functionLineName: string;    // 职能线名称
  weight: number;              // 在该职能线中的权重
  scoringRule?: string;        // 可覆盖默认计分规则
  owner: string;               // 职能线负责人
}
```

### 3.7 画布视图（PRD 待实现）

**定位**：以可视化方式展示年度业务规划的全景，便于理解和沟通。

**核心功能**：
1. **树形画布**
   - 中心：年度业务规划总目标
   - 第一层：BSC 四维度
   - 第二层：KPI
   - 第三层：分解维度（战区/职能/产品）
   - 第四层：重点工作/KTI
   - 支持横向（BSC 因果链）和纵向（组织分解）两种布局

2. **分组框**
   - 战区分组
   - 职能线分组
   - 产品分组
   - 自定义分组

3. **连线与关联**
   - **父子分解连线**：父 KPI → 子 KPI
   - **BSC 因果支撑连线**：学习成长 → 内部流程 → 客户 → 财务
   - **KPI 与重点工作关联线**
   - **KPI 与 KTI 关联线**
   - 连线支持不同线型/颜色区分关系类型

4. **画布操作**
   - 拖拽节点
   - 缩放/平移
   - 点击节点查看详情
   - 导出图片

5. **视图切换**
   - **战略视图（BSC 维度）**：按 BSC 四维度分组，展示因果支撑链
   - **组织视图（战区/部门）**：按组织架构展示分解树
   - **执行视图（重点工作/KTI）**：展示 KPI 与重点工作/KTI 的关联
   - **混合视图**：同时展示 BSC 维度 + 组织分解 + 重点工作关联

### 3.8 重点工作规划（现有增强）

**定位**：将战略目标转化为可执行的年度重点工作。

**核心功能（在现有基础上增强）**：
1. **重点工作 CRUD（已有）**
   - 序号、牵引方向、名称、KMS链接、负责人、状态

2. **KPI 双向关联（新增）**
   - 重点工作可关联多个 KPI
   - KPI 可查看关联的重点工作
   - 关联关系影响健康度评分

3. **里程碑规划（新增）**
   - 每个重点工作设置年度里程碑
   - 里程碑时间线展示
   - 里程碑与经营分析会议题关联

4. **资源需求（新增）**
   - 重点工作预算需求
   - 重点工作人力需求
   - 资源冲突检查

5. **来源追踪（新增）**
   - 手动创建
   - 来自战略专题
   - 来自 VDBD 关键任务
   - 来自差距分析

### 3.9 资源配置（新增）

**定位**：为年度业务规划分配预算、人力和其他资源，确保目标可实现。

**核心功能**：
1. **预算池管理**
   - 年度预算总额设定
   - 预算池分类：市场费用、研发费用、人力成本、运营成本
   - 预算池与 BSC 维度关联

2. **预算分配**
   - 按 KPI 分配预算
   - 按重点工作分配预算
   - 按组织维度分配预算
   - 分配进度可视化

3. **人力需求规划**
   - 按重点工作/项目规划人力
   - 人头数 / FTE
   - 技能要求
   - 现有能力缺口分析

4. **资源冲突检查**
   - 预算超支提醒
   - 同一负责人/团队任务过载提醒
   - 资源冲突列表与解决方案建议

5. **ROI 预估**
   - 基于 VDBD 价值量化数据
   - 自动计算各 KPI/重点工作的投入产出比
   - ROI 排序与优先级建议

**数据模型**：
```typescript
interface ResourcePlan {

  headcountPlan: {
```

### 3.10 风险与假设（新增）

**定位**：识别影响年度目标达成的关键假设和风险，提前制定预案。

**核心功能**：
1. **关键业务假设**
   - 市场环境假设
   - 客户行为假设
   - 竞争态势假设
   - 内部能力假设

2. **风险登记册**
   - 风险描述
   - 影响维度（收入/成本/交付/组织）
   - 发生概率
   - 影响程度
   - 风险等级（高/中/低）
   - 应对预案
   - 责任人

3. **假设验证计划**
   - 关键假设验证时间
   - 验证方式
   - 验证结果
   - 假设不成立时的调整方案

4. **风险与目标关联**
   - 风险关联到具体 KPI
   - 风险关联到重点工作
   - 风险对目标达成的影响量化

5. **风险看板**
   - 风险矩阵（概率 × 影响）
   - 待处理风险 TOP N
   - 风险趋势

### 3.11 规划评审（新增）

**定位**：年度业务规划的质量 gate，确保规划合理、可行、共识。

**核心功能**：
1. **评审检查清单**
   - 目标完整性：是否承接了 SP 输出
   - 分解合理性：子目标合计是否与父目标一致
   - 资源匹配度：预算/人力是否足够支撑目标
   - 风险可控度：高风险是否有应对预案
   - 职能线权重合理性
   - 重点工作与 KPI 关联度

2. **评审会议管理**
   - 评审会议安排

### 3.12 发布执行（现有增强）

**定位**：将年度业务规划正式发布到执行阶段，启动跟踪。

**核心功能（在现有基础上增强）**：
1. **发布前校验**
   - 所有 KPI 必须已分解或说明不分解原因
   - 所有重点工作必须有负责人和里程碑
   - 预算分配完成率 ≥ 阈值
   - 高风险必须有应对预案
   - 权重合计校验通过

5. **发布后归档**
   - 规划版本标记为"已发布"
   - 锁定规划数据
   - 生成发布报告

### 6.4 与组织绩效管理（OMP）的定位分工 ⭐

这是最容易出现职责重叠的边界，必须明确划分。

#### 6.4.1 核心原则

> **ABP 是年度目标、计划、资源的"制定者和发布者"；OMP 是执行数据的"跟踪者和分析者"。**
>
> **ABP 负责"定规划、发目标"；OMP 负责"录 actual、做分析、推改进"。**

#### 6.4.2 职责边界表

| 职责 | ABP | OMP | 说明 |
|------|-----|-----|------|
| 制定年度 KPI 目标值 | ✅ | ❌ | ABP 在规划期制定 |
| 分解 KPI 到组织/职能 | ✅ | ❌ | ABP 在规划期分解 |
| 配置职能线权重 | ✅ | ❌ | ABP 在规划期配置 |
| 规划重点工作 | ✅ | ❌ | ABP 在规划期制定 |
| 分配预算/人力 | ✅ | ❌ | ABP 在规划期分配 |
| 识别业务假设与风险 | ✅ | ❌ | ABP 在规划期识别 |
| 发布规划并锁定 | ✅ | ❌ | ABP 发布后数据锁定 |
| 录入/采集 actual | ❌ | ✅ | OMP 在执行期录入 |
| 计算达成率 | ❌ | ✅ | OMP 基于 ABP 目标和 actual 计算 |
| 偏差分析与预警 | ❌ | ✅ | OMP 执行期持续监控 |
| 重点工作进度跟踪 | ❌ | ✅ | OMP 跟踪实际进度 |
| 绩效报告生成 | ❌ | ✅ | OMP 生成执行报告 |
| 目标变更审批 | ✅ 发起 | ✅ 触发 | OMP 发现重大偏差后触发变更申请，ABP 审批并更新版本 |

#### 6.4.3 数据主数据边界

| 数据 | 主数据拥有者 | 说明 |
|------|------------|------|
| targetValue / baseline / challenge | **ABP** | 规划期制定，发布后只读 |
| weight / 职能线权重 | **ABP** | 规划期配置，发布后只读 |
| scoringRule | **ABP** 定义 | OMP 只读引用并执行计算 |
| actualValue | **OMP** | 执行期录入/采集 |
| achievementRate | **OMP** 计算 | 基于 ABP 目标 + OMP actual |
| budgetAllocated | **ABP** 分配 | OMP 监控执行花费 |
| actualCost | **OMP** 录入 | 执行期实际花费 |
| 重点工作计划里程碑 | **ABP** 规划 | OMP 跟踪实际完成状态 |
| 风险与假设 | **ABP** 识别 | OMP 执行期更新状态 |

#### 6.4.4 目标变更流程

执行过程中，OMP 发现重大偏差需要调整目标时，不能直接修改目标，必须走规划变更流程：

```
OMP 发现重大偏差或目标不适用
    ↓
OMP 触发"目标变更申请"（记录偏差原因、建议调整值）
    ↓
流转到 ABP 进行变更评审
    ↓
ABP 更新规划版本（如 v1.0 → v1.1）
    ↓
重新发布到 OMP
    ↓
OMP 基于新版本目标继续执行跟踪
```

变更申请的数据结构：
```typescript
interface TargetChangeRequest {
  id: string;
  cycleId: string;
  kpiInstanceId: string;
  originalTarget: number;
  proposedTarget: number;
  reason: string;                  // 变更原因
  deviationEvidence: string;       // 偏差证据
  requestedBy: string;
  requestedAt: string;
  status: 'pending' | 'approved' | 'rejected';
  reviewedBy?: string;
  reviewedAt?: string;
  comment?: string;
  newVersion?: number;             // 审批通过后生成的 ABP 版本
}
```

## 八、关键设计原则

1. **从 SP 自动承接，减少重复录入**：年度业务规划的输入应尽可能从 SP 输出自动导入。
2. **规划不仅是目标，更是资源配置**：预算、人力、风险必须与目标一起规划。
3. **多视角分解**：组织维度、职能维度、产品维度、时间维度都要支持。
4. **过程可追溯**：每个 KPI 和重点工作都能追溯到 SP 的来源。
5. **发布即锁定**：发布后的规划进入执行态，修改需走变更流程。
6. **健康度驱动**：通过健康度评分和检查清单，引导用户完成高质量的规划。

## 十、指标树与 BSC 支撑关联设计

### 10.1 设计目标

年度业务规划中的 KPI 经过分解后，会形成复杂的指标网络。用户需要一种清晰的方式来：

1. **查看层级分解关系**：公司级 KPI → 部门/战区 KPI → 团队 KPI
2. **理解 BSC 因果支撑关系**：学习成长 KPI 如何支撑内部流程 KPI，内部流程 KPI 如何支撑客户 KPI，客户 KPI 如何支撑财务 KPI
3. **发现结构问题**：断链、孤立、重复、冲突

### 10.2 指标树的两种核心视图


**设计要点**：
- 每个维度一个横向泳道
- 维度内 KPI 纵向排列
- 用带箭头的贝塞尔曲线连接支撑关系
- 箭头方向：从支撑方指向被支撑方（学习成长 → 财务）
- 不同维度用不同颜色区分
- 点击连线可查看支撑说明（为什么这个指标支撑那个指标）

### 10.3 BSC 支撑关联的数据模型

#### 10.3.1 KPI 支撑关系

```typescript
interface KpiSupportLink {
  id: string;
  cycleId: string;
  fromKpiId: string;               // 支撑方 KPI
  toKpiId: string;                 // 被支撑方 KPI
  relationType: 'drives' | 'influences' | 'supports' | 'enables';
  description: string;             // 支撑说明
  strength: number;                // 支撑强度 1-10
  autoDetected: boolean;           // 是否自动识别
  createdBy?: string;
  createdAt: string;
}
```

#### 10.3.2 BSC 因果链规则（自动识别基础）

系统默认的 BSC 维度间支撑规则：

| 支撑方向 | 关系类型 | 示例 |
|---------|---------|------|
| 学习成长 → 内部流程 | enables | 人均产出提升 → 交付周期缩短 |
| 内部流程 → 客户 | supports | 交付周期缩短 → 客户续约率提升 |
| 客户 → 财务 | drives | 客户续约率提升 → 销售额增长 |
| 内部流程 → 财务 | influences | 交付效率提升 → 回款加快 |
| 学习成长 → 客户 | enables | 销售能力 → 大客户满意度 |

自动识别逻辑：
1. 如果两个 KPI 属于不同 BSC 维度
2. 且存在上述维度对关系
3. 且用户或 SP 战略地图中未明确反对
4. 则自动建议建立支撑关联

#### 10.3.3 与战略地图因果链的衔接

ABP 中的 BSC 支撑关联可以直接复用 SP 战略地图中的因果链数据：

```
SP 战略地图
    │
    ├─ 战略目标 causal links
    │
    ▼
ABP 年度业务规划
    │
    ├─ 战略目标 → KPI 映射
    ├─ 自动生成 KPI 间支撑关联建议
    │
    ▼
用户确认/编辑支撑关系
```

### 10.5 BSC 支撑关联的校验规则

1. **断链检测**
   - 财务 KPI 没有内部流程/客户 KPI 支撑 → 警告
   - 客户 KPI 没有内部流程 KPI 支撑 → 警告
   - 内部流程 KPI 没有学习成长 KPI 支撑 → 提示

2. **孤立检测**
   - 没有任何支撑关系也没有被支撑关系的 KPI → 提示检查

3. **循环检测**
   - 支撑关系不能形成循环（A → B → C → A）

4. **支撑强度评估**
   - 对每条支撑关系评估强度
   - 弱支撑关系提示补充说明或证据

### 10.6 与画布视图的衔接

指标树的 BSC 支撑关联数据可以直接在画布视图中渲染：

```
画布视图 = 指标树 + 可视化布局引擎
```

画布视图额外提供：
- 自由拖拽布局
- 多种自动布局算法（层次布局、力导向布局、BSC 泳道布局）
- 导出图片/PDF
- 演示模式

### 10.7 数据初始化与同步

1. **从战略地图导入时**：自动带入战略目标间的因果链，映射为 KPI 支撑关联建议
2. **手动添加 KPI 时**：系统根据 BSC 维度自动推荐可能的支撑关系
3. **分解 KPI 时**：自动继承父 KPI 的支撑关系，并可根据子 KPI 维度推荐新增关系
4. **发布后锁定**：支撑关系随 KPI 目标基线一起同步到 OMP，供执行期参考
