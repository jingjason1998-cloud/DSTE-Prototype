### 2.2 架构问题

#### 问题 1：巨石文件（God File）

**症状**：`meetings.html` 3459 行，包含 UI 渲染、状态管理、业务逻辑、API 调用、通知推送全部代码。

**风险**：
- 协作冲突概率高（多人同时修改同一文件）
- 代码审查困难（单次 PR 改动难以聚焦）
- 热更新/按需加载无法实现
- 单元测试难以隔离

**规范**：遵循「单一职责原则」（SRP），一个文件不应超过 500 行逻辑代码。

**建议**：

#### 问题 2：全局状态污染

**症状**：使用 11 个 `window._xxx` 全局变量传递状态，无状态变更追踪。

```javascript
window._meetingsData
window._meetingsFilterScene
window._meetingsViewMode
window._calendarState
window._editMeetingId
window._meetingEditData
window._currentDetailMeetingId
window._evalMeetingId
window._evalFormData
window._webhookSelectorPayload
window._decisionsFilter / window._decisionsSearch
```

**风险**：
- 状态变更不可预测（任何函数都可能修改全局状态）
- 调试困难（无法追踪"谁在什么时候改了什么"）
- 测试困难（需要手动清理全局状态）
- 内存泄漏风险（状态长期挂载在 window 上）

**规范**：采用「集中式状态管理」模式。

**建议**：
```javascript
// 使用简单的 Pub/Sub Store 模式
class MeetingStore {
  constructor() {
    this.state = {
      meetings: [],
      filter: { scene: null, month: 'all', status: 'all' },
      viewMode: 'list',
      calendar: { year: new Date().getFullYear(), month: new Date().getMonth(), view: 'month' },
      editing: null,      // { id, data }
      detail: null,       // meetingId
      eval: null,         // { meetingId, formData }
    };
    this.listeners = new Set();
  }
  
  getState() { return this.state; }
  
  setState(partial) {
    const prev = { ...this.state };
    this.state = { ...this.state, ...partial };
    this.listeners.forEach(fn => fn(this.state, prev));
  }
  
  subscribe(fn) { this.listeners.add(fn); return () => this.listeners.delete(fn); }
}

const store = new MeetingStore();
store.subscribe((next, prev) => {
  if (next.filter.scene !== prev.filter.scene) renderSceneCards();
  if (next.meetings !== prev.meetings) renderMeetingList();
});
```

#### 问题 3：KV 整存整取的性能瓶颈

**症状**：`worker.js` 中 meetings 数据以完整 JSON 数组存入/取出 KV，无分页、无增量更新。

**风险**：
- 数据量增大后（>1000 条会议），读写延迟显著增加
- 并发编辑时容易产生覆盖冲突（Last Write Wins）
- 无法实现细粒度权限控制

**规范**：采用「CQRS + 事件溯源」或至少「RESTful 资源 API」模式。

**建议**（v0.6.0 迁移至 D1）：
```sql
-- Cloudflare D1 表结构
CREATE TABLE meetings (
  id TEXT PRIMARY KEY,
  title TEXT NOT NULL,
  date TEXT NOT NULL,
  scenario TEXT NOT NULL,
  status TEXT NOT NULL,
  data JSON,              -- 完整的会议数据
  created_at TEXT,
  updated_at TEXT
);

CREATE TABLE decisions (
  id TEXT PRIMARY KEY,
  meeting_id TEXT NOT NULL,
  content TEXT NOT NULL,
  owner TEXT,
  deadline TEXT,
  status TEXT,
  kms_url TEXT,
  FOREIGN KEY (meeting_id) REFERENCES meetings(id)
);

CREATE TABLE action_items (
  id TEXT PRIMARY KEY,
  meeting_id TEXT NOT NULL,
  content TEXT NOT NULL,
  owner TEXT,
  deadline TEXT,
  status TEXT,
  FOREIGN KEY (meeting_id) REFERENCES meetings(id)
);

CREATE TABLE effectiveness (
  meeting_id TEXT PRIMARY KEY,
  overall_score INTEGER,
  dimensions JSON,
  feedback JSON,
  comment TEXT,
  evaluated_at TEXT,
  FOREIGN KEY (meeting_id) REFERENCES meetings(id)
);
```

## 三、代码质量诊断

### 3.1 类型安全

**症状**：纯 JavaScript，无 TypeScript，数据结构靠约定而非类型约束。

**风险**：
- 运行时错误难以提前发现（如 `meeting.effectiveness.dimensions` 为 undefined 时访问 `.preparation`）
- IDE 智能提示缺失，开发效率低
- 重构困难（无法安全重命名字段）
- 接口契约不清晰（前后端数据格式不一致时难以发现）

**规范**：采用「渐进式 TypeScript 迁移」策略。

**建议**：
```typescript
// 创建 src/meetings/types.ts
export interface Meeting {
  id: string;
  title: string;
  date: string;
  month: string;
  scenario: ScenarioType;
  status: MeetingStatus;
  // ... 完整类型定义
}

export type ScenarioType = 
  | 'union_quarterly' 
  | 'hq_routine' 
  | 'region_routine' 
  | 'lagging_region' 
  | 'lagging_vertical';

export type MeetingStatus = 'planned' | 'in_progress' | 'completed' | 'cancelled';

// JSDoc 注释作为中间过渡（立即可用，无需构建变更）
/**
 * @param {Meeting} meeting
 * @returns {EffectivenessScore}
 */
function calculateAutoScore(meeting) { ... }
```

### 3.2 innerHTML 安全风险

**症状**：40 处 `innerHTML` 直接赋值，虽然会议数据是内部生成，但存在 XSS 隐患。

**风险**：
- 如果未来接入用户输入（如会议标题由外部系统同步），可能引入 XSS
- `minutes_content` 等字段虽然做了简单转义（`.replace(/</g, '&lt;')`），但不够系统

**规范**：采用「模板引擎」或「DOM API」替代 innerHTML。

**建议**：
```javascript
// 当前（有风险）
el.innerHTML = `<div>${meeting.title}</div>`;

// 改进：使用 DOM API
function createElement(tag, attrs = {}, children = []) {
  const el = document.createElement(tag);
  Object.entries(attrs).forEach(([k, v]) => el.setAttribute(k, v));
  children.forEach(child => {
    if (typeof child === 'string') el.appendChild(document.createTextNode(child));
    else el.appendChild(child);
  });
  return el;
}

// 或引入轻量级模板引擎（如 lit-html）
import { html, render } from 'lit-html';
render(html`<div>${meeting.title}</div>`, container);
```

### 3.3 错误处理

**症状**：仅 12 处 `try/catch`，大量异步操作无错误处理（如 `fetch` 失败、JSON 解析失败）。

**风险**：
- API 请求失败时用户无反馈，页面可能处于异常状态
- `JSON.parse(localStorage.getItem(...))` 在数据损坏时抛出异常，导致整个页面白屏

**规范**：采用「结果类型」模式（Result Type）替代异常抛出。

**建议**：
```javascript
// 当前
try {
  const data = JSON.parse(localStorage.getItem('dste_meetings'));
} catch (e) { ... }

// 改进：统一封装 Result 类型
function safeJsonParse(str, defaultValue = null) {
  try {
    return { ok: true, value: JSON.parse(str) };
  } catch (e) {
    return { ok: false, error: e.message };
  }
}

// API 调用统一封装
async function apiCall(endpoint, options = {}) {
  try {
    const resp = await fetch(API_BASE + endpoint, options);
    if (!resp.ok) return { ok: false, error: `HTTP ${resp.status}` };
    const data = await resp.json();
    return { ok: true, value: data };
  } catch (e) {
    return { ok: false, error: e.message };
  }
}
```

### 3.4 代码重复

**症状**：多处重复逻辑未提取为公共函数。

**示例**：
- 评分颜色/标签逻辑在 `getScoreColor`、`getScoreLabel` 中定义，但在多处内联使用
- 状态标签样式在 `STATUS_CONFIG` 中定义，但在渲染时多处重复构建
- 保存逻辑（`apiSave` + 重新渲染）在多个事件处理中重复

**规范**：采用「DRY 原则」（Don't Repeat Yourself），提取公共函数。

```javascript
// tests/unit/scoring.test.js
import { describe, it, expect } from 'vitest';
import { calculateAutoScore } from '../../src/meetings/utils/scoring.js';

describe('calculateAutoScore', () => {
  it('should return perfect score for completed meeting with all metrics', () => {
    const meeting = {
      pipeline: { reportGenerated: true, preReviewDone: true, meetingHeld: true, minutesDrafted: true, minutesApproved: true, actionsTracked: true },
      metrics: { materialTimeliness: 100, resolutionTimeliness: 100, actionClosure: 100 },
      agenda_items: [{}, {}],
      decisions: [{ status: 'approved', kmsUrl: 'https://...' }],
      hasMinutes: true, minutesStatus: 'final'
    };
    const result = calculateAutoScore(meeting);
    expect(result.overallScore).toBe(100);
    expect(result.feedback).toContain('材料充分');
    expect(result.feedback).toContain('议程合理');
  });

  it('should flag issues for incomplete meeting', () => {
    const meeting = {
      pipeline: {},
      metrics: {},
      agenda_items: [],
      decisions: [],
      hasMinutes: false
    };
    const result = calculateAutoScore(meeting);
    expect(result.dimensions.preparation).toBeLessThan(60);
    expect(result.feedback).toContain('数据不足');
  });
});
```
