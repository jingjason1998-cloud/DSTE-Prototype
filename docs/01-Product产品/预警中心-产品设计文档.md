## 一、产品概述

### 1.1 定位与价值主张

**预警中心**是DSTE战略管理平台的**统一预警处理中枢**，负责接收来自规则引擎、预算管理、KPI监控等模块的预警事件，进行**分级确认、多渠道通知、工单跟踪、闭环管理**。

**核心价值：**
- **预警统一入口**: 各模块预警统一汇聚，避免信息分散
- **分级智能通知**: 根据预警级别自动选择通知渠道和对象
- **闭环跟踪管理**: 预警不只是一条消息，而是可跟踪、可处理、可闭环的工单
- **历史可追溯**: 完整的预警处理历史，支持审计和分析
- **规则优化反馈**: 预警处理结果反馈给规则引擎，持续优化触发条件

```
│                                                                             │
│   关系原则:                                                                 │
│   • 规则引擎 = "判断是否该预警"（What & When）                              │
│   • 预警中心 = "预警后做什么"（Who & How）                                  │
│   • 规则变更 ──► 预警中心同步更新触发条件（订阅机制）                        │
└─────────────────────────────────────────────────────────────────────────────┘

```

### 模块1: 预警接入层

> **定位**: 预警中心的"入口网关"，负责接收来自各模块的预警事件，进行校验、转换、存储。

#### 1.1 模块职责
- **做什么：** 接收、校验、转换、存储预警事件
- **不做什么：** 不判断预警是否应该触发（由规则引擎判断），不处理通知（由通知模块处理）

#### 1.2 涉及的数据实体

| 实体 | 说明 | 关键字段 |
|------|------|---------|
| 预警事件 | 原始预警事件 | event_id, source_module, source_rule, alert_type, level, message, trigger_value, target_dept, created_at |
| 预警记录 | 转换后的预警记录 | alert_id, event_id, status, classification_result, notification_status, ticket_id |

#### 1.3 功能列表

| 功能 | 用户故事 | 输入 | 输出 |
|------|---------|------|------|
| 接收预警事件 | 系统接收来自规则引擎的预警事件 | 标准化预警事件JSON | 接收确认 |
| 校验事件格式 | 系统校验预警事件是否符合标准格式 | 预警事件 | 校验结果（通过/失败+原因） |
| 转换事件 | 系统将事件转换为内部预警记录 | 预警事件 | 预警记录 |
| 去重处理 | 系统对重复预警进行合并或丢弃 | 预警记录 | 去重后的预警记录 |
| 存储事件 | 系统存储预警记录 | 预警记录 | 存储确认 |

#### 1.4 预警事件标准格式

```json
{
  "event_id": "EVT-20260520-001",
  "timestamp": "2026-05-20T10:30:00Z",
  "source": {
    "module": "rule-engine",
    "rule_code": "ALERT-BUDGET-001",
    "rule_version": "1.2.0"
  },
  "alert": {
    "type": "BUDGET_EXECUTION_ORANGE",
    "name": "预算执行率橙色预警",
    "level": "ORANGE",
    "message": "预算执行率已达92%，即将超支",
    "trigger_value": 92,
    "threshold": 90,
    "suggest_action": "提交超支说明",
    "context": {
      "dept_id": "DEPT-001",
      "dept_name": "销售部",
      "budget_id": "BUD-2026-001",
      "budget_name": "2026年销售预算",
      "actual_spend": 9200000,
      "budget_amount": 10000000,
      "execution_rate": 0.92
    }
  }
}
```

### 模块2: 预警分级引擎

> **定位**: 对预警进行级别确认、升级、降级，确保预警得到适当的关注程度。

#### 2.1 模块职责
- **做什么：** 确认预警级别、执行升级/降级规则
- **不做什么：** 不修改预警的触发条件（由规则引擎管理）

#### 2.2 预警级别定义

| 级别 | 颜色 | 说明 | 默认处理时限 | 通知方式 |
|------|------|------|-------------|---------|
| **GREEN** | 🟢 绿色 | 正常/达成 | 无需处理 | 仅记录 |
| **YELLOW** | 🟡 黄色 | 关注 | 24小时 | 企业微信 |
| **ORANGE** | 🟠 橙色 | 预警 | 12小时 | 企业微信 + 邮件 |
| **RED** | 🔴 红色 | 严重 | 4小时 | 企业微信 + 邮件 + 短信 |

#### 2.3 升级规则

```yaml
规则名称: 预警自动升级规则
规则编码: ALERT-ESCALATION-001

升级条件:
  时间升级:
    - 黄色预警超过24小时未处理 ──► 升级为橙色
    - 橙色预警超过12小时未处理 ──► 升级为红色
    - 红色预警超过4小时未处理 ──► 升级到高管
  
  重复升级:
    - 同一规则同一部门7天内触发3次 ──► 自动升一级
    - 同一规则同一部门30天内触发5次 ──► 自动升一级
  
  关联升级:
    - 同一部门同时存在3个橙色预警 ──► 全部升级为红色
    - 关联KPI同时触发预警 ──► 按最高级别处理

升级动作:
  - 更新预警级别
  - 追加通知对象（上级领导）
  - 标记为"紧急"
  - 创建升级记录
```

#### 2.4 功能列表

| 功能 | 说明 |
|------|------|
| 级别确认 | 根据规则引擎输出的建议级别，结合当前情况确认最终级别 |
| 自动升级 | 按升级规则定时检查，对超时未处理的预警自动升级 |
| 手动升级 | 管理员手动调整预警级别 |
| 降级处理 | 预警条件消除后自动降级或关闭 |
| 升级记录 | 记录所有升级操作的历史 |

### 模块3: 通知分发中心

> **定位**: 预警中心的"通信枢纽"，负责将预警通知到正确的人，通过正确的渠道，在正确的时间。

#### 3.1 模块职责
- **做什么：** 管理通知渠道、配置通知策略、执行通知发送、跟踪通知状态
- **不做什么：** 不定义预警触发条件（由规则引擎管理）

#### 3.2 通知渠道

| 渠道 | 适用场景 | 特点 |
|------|---------|------|
| **企业微信** | 所有级别 | 即时、可交互、支持卡片 |
| **邮件** | 橙色及以上 | 正式、可存档、支持附件 |
| **短信** | 红色 | 强提醒、确保送达 |
| **站内消息** | 所有级别 | 系统内查看、历史记录 |
| **电话** | 红色（自动升级后） | 人工确认、确保知晓 |

#### 3.3 通知策略配置

```yaml
策略名称: 默认通知策略
策略编码: NOTIFY-DEFAULT

通知规则:
  黄色预警:
    即时通知:
      - 渠道: 企业微信
      - 对象: 部门负责人
      - 模板: YELLOW_WECHAT
    
  橙色预警:
    即时通知:
      - 渠道: 企业微信
      - 对象: 部门负责人
      - 模板: ORANGE_WECHAT
    延迟通知（30分钟后未处理）:
      - 渠道: 邮件
      - 对象: 部门负责人 + 财务经理
      - 模板: ORANGE_EMAIL
    
  红色预警:
    即时通知:
      - 渠道: 企业微信
      - 对象: 部门负责人 + 财务经理
      - 模板: RED_WECHAT
    延迟通知（15分钟后未处理）:
      - 渠道: 邮件 + 短信
      - 对象: 部门负责人 + 财务经理 + 财务总监
      - 模板: RED_URGENT
    升级通知（4小时后未处理）:
      - 渠道: 电话
      - 对象: 财务总监
      - 模板: RED_CALL

免打扰设置:
  - 工作时间: 9:00-18:00，所有渠道正常发送
  - 非工作时间: 仅红色预警发送短信，其他延迟到工作时间
  - 节假日: 仅红色预警发送，其他延迟到节后
```

### 模块4: 预警工单管理

> **定位**: 预警中心的"执行跟踪层"，将预警转化为可跟踪、可处理、可闭环的工单。

#### 4.1 工单状态流转

```
┌─────────┐    ┌─────────┐    ┌─────────┐    ┌─────────┐    ┌─────────┐
│  待处理  │───►│ 处理中  │───►│ 待审批  │───►│ 已完成  │───►│  已关闭  │
│  (NEW)  │    │(PROCESS)│    │(PENDING)│    │(DONE)   │    │(CLOSED) │
└────┬────┘    └────┬────┘    └────┬────┘    └────┬────┘    └─────────┘
     │              │              │              │
     │              │              │              │
     └──────────────┴──────────────┴──────────────┘
                    │
                    ▼
              ┌─────────┐
              │  已驳回  │
              │(REJECT) │
              └────┬────┘
                   │
                   └──────────────► 回到"处理中"
```

#### 4.2 工单处理流程

```
1. 预警触发 ──► 自动创建工单（状态: 待处理）
2. 负责人收到通知 ──► 查看预警详情
3. 负责人提交处理说明 ──► 工单状态变为"处理中"
   - 处理说明（必填）
   - 整改计划（可选）
   - 附件（可选）
4. 如需审批 ──► 提交审批 ──► 工单状态变为"待审批"
5. 审批人审批 ──► 通过: 工单状态变为"已完成"
                 ──► 驳回: 工单状态变为"处理中"，需重新处理
6. 验证整改效果 ──► 确认 ──► 工单状态变为"已关闭"
                ──► 未达标 ──► 工单状态变为"处理中"
```

#### 4.3 功能列表

| 功能 | 说明 |
|------|------|
| 自动创建工单 | 预警触发时自动创建对应工单 |
| 查看工单列表 | 按状态/时间/部门/类型筛选查看 |
| 提交处理说明 | 负责人提交处理说明和整改计划 |
| 上传附件 | 支持上传处理说明的附件 |
| 工单审批 | 审批人查看并审批处理结果 |
| 工单关闭 | 验证通过后关闭工单 |
| 工单转派 | 将工单转派给其他负责人 |
| 工单合并 | 将同一类型的多个预警合并处理 |
| 工单催办 | 对超时未处理的工单发送催办通知 |

### 模块5: 预警闭环跟踪

> **定位**: 确保每个预警都有始有终，不遗漏、不悬空。

#### 5.1 模块职责
- **做什么：** 跟踪预警处理全流程，确保闭环
- **不做什么：** 不替代工单管理，只负责跟踪和验证

#### 5.2 闭环验证规则

```yaml
验证规则:
  预算预警:
    验证方式: 数据验证
    验证条件: 下一周期预算执行率 < 阈值
    验证周期: 下一个统计周期
    
  基线偏差预警:
    验证方式: 数据验证
    验证条件: 偏差率恢复到正常范围
    验证周期: 下一个统计周期
    
  KPI预警:
    验证方式: 数据验证
    验证条件: KPI达成率恢复到目标范围
    验证周期: 下一个统计周期
    
  系统监控预警:
    验证方式: 状态验证
    验证条件: 系统指标恢复正常
    验证周期: 实时监控
```

#### 5.3 功能列表

| 自动升级 | 对逾期预警自动升级级别和通知对象 |

### 5.2 关键数据表设计

#### 预警事件表 (alert_event)

```sql
CREATE TABLE alert_event (
    event_id        VARCHAR(64) PRIMARY KEY COMMENT '事件ID',
    timestamp       DATETIME NOT NULL COMMENT '事件时间',
    source_module   VARCHAR(64) NOT NULL COMMENT '来源模块',
    source_rule     VARCHAR(64) COMMENT '来源规则编码',
    rule_version    VARCHAR(16) COMMENT '规则版本',
    alert_type      VARCHAR(64) NOT NULL COMMENT '预警类型',
    alert_name      VARCHAR(128) COMMENT '预警名称',
    level           VARCHAR(16) NOT NULL COMMENT '级别：GREEN/YELLOW/ORANGE/RED',
    message         TEXT NOT NULL COMMENT '预警消息',
    trigger_value   DECIMAL(18,4) COMMENT '触发值',
    threshold       DECIMAL(18,4) COMMENT '阈值',
    suggest_action  VARCHAR(256) COMMENT '建议行动',
    context_json    JSON COMMENT '上下文信息',
    created_at      DATETIME DEFAULT CURRENT_TIMESTAMP,
    
    INDEX idx_source (source_module, source_rule),
    INDEX idx_type (alert_type),
    INDEX idx_level (level),
    INDEX idx_time (timestamp)
) COMMENT='预警事件表';
```

#### 预警记录表 (alert_record)


| 预警记录 | 预警中心内部存储的预警信息 |
| 预警工单 | 预警对应的处理跟踪单据 |
| 预警级别 | GREEN/YELLOW/ORANGE/RED |
| 升级 | 预警级别提升，通知对象扩大 |
| 闭环 | 预警从触发到处理完成的全流程 |
