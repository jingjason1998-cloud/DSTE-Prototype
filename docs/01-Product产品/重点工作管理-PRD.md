### 1.3 术语表

| 术语 | 说明 |
|------|------|
| OMP | Organizational Performance Management，组织绩效管理模块 |
| 重点工作 | 战略级工作项，承接年度经营计划或经营分析会决议，在 OMP 中对应 `task` 实体 |
| 年度计划派生任务 | 由年度经营计划发布时自动生成到 OMP 的重点工作，持有 `annualPlanTaskId` |
| OMP 本地任务 | 在 OMP 内直接创建的重点工作，无 `annualPlanTaskId` |
| 子任务/子项目 | 隶属于重点工作的独立执行单元，可独立设置负责人、时间、进度、资源 |
| 父任务 | 包含子任务的顶层重点工作 |
| PersonRef | 员工唯一标识，通常为 `ldap` 或员工 ID |
| KPI 实例 | 某个战略周期内对指标库指标的具体目标设定，如 `2026 年销售部销售额` |
| 指标库 | 企业级指标定义，包含名称、口径、单位等元数据 |

## 二、产品定位与边界

### 2.1 模块定位

「重点工作管理」是 OMP 的**战略执行追踪中枢**，承接年度经营计划、经营分析会决议等上游输入，对战略举措进行项目化管理：

- **目标对齐**：通过 KPI 指标关联，回答"这项工作支撑哪些目标"
- **资源到位**：通过人员配置/子任务分配，回答"这项工作由谁来做"
- **进度可视**：通过里程碑、甘特图、进度预警，回答"这项工作进展如何"
- **风险可控**：通过风险登记与升级机制，回答"这项工作存在什么风险"

### 2.2 核心价值

1. **战略到执行的闭环**：将年度经营计划、会议决议转化为可跟踪的重点工作
2. **轻量项目管理**：聚焦战略级项目，不替代专业项目管理工具（如 Jira）
3. **资源与目标对齐**：人员、子任务、KPI 围绕重点工作统一视图

### 2.3 功能边界

| 范围 | 说明 |
|------|------|
| **做** | 重点工作的创建、编辑、删除、列表/看板/甘特图展示；里程碑管理；人员配置；KPI 关联；子任务/项目创建与资源分配；进度汇报与预警；风险登记 |
| **不做** | 替代专业项目管理工具；替代预算/成本全流程管理；跨重点工作的自动资源调度；子任务再下级拆解（本期只支持一级子任务） |
| **限制** | 年度计划派生任务的负责人、主指标由年度计划侧控制，OMP 侧只读或仅可补充 |

## 三、用户角色与场景

### 3.1 角色定义

| 角色 | 职责 | 核心场景 |
|------|------|----------|
| **组织绩效管理员** | 统筹 OMP 重点工作的创建、拆解、人员配置与进度跟踪 | 创建年度重点工作、为任务补充成员、监控整体进度 |
| **部门负责人** | 管理本部门重点工作的执行团队 | 为本部门任务分配成员、拆解子任务、更新进度 |
| **任务负责人** | 负责具体重点工作或子项目的执行 | 更新自己负责的任务进度、登记风险 |
| **高管/决策者** | 关注战略执行整体健康度 | 查看重点工作总览、识别延期风险与资源瓶颈 |

### 3.2 核心用户故事

#### 故事 1：创建并跟踪重点工作
```
作为 组织绩效管理员
我希望 在 OMP 中创建重点工作并设定负责人、时间、里程碑
以便 将年度经营计划落地为可跟踪的执行项
```

#### 故事 2：重点工作对齐 KPI
```
作为 部门负责人
我希望 为重点工作关联核心 KPI 和相关指标
以便 明确每项工作支撑的经营目标
```

#### 故事 3：为重点工作分配执行人员
```
作为 组织绩效管理员
我希望 通过拖拽快速为重点工作添加成员
以便 快速完成执行团队组建
```

#### 故事 4：拆解重点工作为子项目
```
作为 部门负责人
我希望 将大型重点工作拆分为多个子任务并分别分配资源
以便 精细化跟踪每个子项目的执行状态
```

#### 故事 5：进度汇报与风险预警
```
作为 任务负责人
我希望 每周更新重点工作进度并登记风险
以便 管理层及时掌握执行风险并介入
```

## 四、数据模型

### 4.1 重点工作实体（OMP_KeyTask）

```typescript
interface OMP_KeyTask {
  id: string;                          // 工作ID
  cycleId: string;                     // 战略周期ID
  source: 'annual_plan' | 'omp';       // 来源：年度计划 / OMP本地创建
  annualPlanTaskId?: string;           // 来源年度计划重点工作ID（仅年度计划派生任务）
  parentId?: string;                   // 父任务ID；为空表示顶层重点工作
  seq: number;                         // 显示顺序
  name: string;                        // 工作名称
  bscDimension?: string;               // BSC维度
  kmsUrl?: string;                     // 知识库链接
  owner: string;                       // 负责人 PersonRef
  ownerDept?: string;                  // 负责人部门/团队（由 owner 自动推导，不再单独维护）
  members: string[];                   // 参与人员 PersonRef 数组
  status: string;                      // planning | active | at_risk | delayed | done | closed
  description?: string;                // 详细描述
  type?: 'strategic' | 'improvement' | 'capability' | 'risk_response' | 'meeting_resolution'; // 本期不在 UI 维护，兼容旧数据
  priority?: 'P0' | 'P1' | 'P2';       // 本期不在 UI 维护，兼容旧数据
  progress: number;                    // 0-100
  dept?: string;                        // 责任部门/团队（由 owner 自动推导，不再单独维护）
  startDate: string;                   // YYYY-MM-DD
  endDate: string;                     // YYYY-MM-DD
  budget?: number;                     // 预算（万），本期不在 UI 维护
  actualCost?: number;                 // 实际成本（万），本期不在 UI 维护
  kpiAssociations: KpiAssociation[];   // 关联 KPI 实例（结构化）
  relatedKpiIds?: string[];            // 【兼容】旧字段，迁移后删除
  milestones: Milestone[];             // 里程碑列表
  risks: Risk[];                       // 风险登记
  resources: Resource[];               // 资源投入
  weight?: number;                     // 子任务权重，用于父任务聚合计算，默认1
  annualTarget?: string;               // 年度目标
  spLink?: string;                     // SP 链接
  biDashboard?: string;                // BI 看板链接
  createdAt: string;
  updatedAt: string;
}

interface KpiAssociation {
  kpiInstanceId: string;               // KPI 实例 ID
  relationType: 'primary' | 'supporting'; // 主指标 / 相关指标
}

interface Milestone {
  id: string;
  name: string;
  date: string;                        // YYYY-MM-DD
  status: 'pending' | 'completed';
}

interface Risk {
  id: string;
  description: string;
  level: 'high' | 'medium' | 'low';
  responsePlan?: string;
  createdAt: string;
}

interface Resource {
  id: string;
  type: string;                        // 人力 / 预算 / 设备 / 其他
  amount: number;
  cost?: number;
  note?: string;
}
```

### 4.2 子任务字段约束

| 字段 | 父任务 | 子任务 | 说明 |
|------|--------|--------|------|
| `parentId` | 空 | 父任务ID | 建立层级关系 |
| `source` | `annual_plan` / `omp` | 仅 `omp` | 子任务不在年度计划中直接定义 |
| `annualPlanTaskId` | 可有 | 空 | 年度计划派生任务的子任务在 OMP 内创建 |
| `owner` | 必填 | 必填 | 子任务必须有负责人 |
| `members` | 独立维护 | 独立维护 | 父子资源独立分配 |
| `progress` | 默认自动聚合 | 手动或独立跟踪 | 父任务进度按子任务加权平均 |
| `weight` | 忽略 | 必填 | 用于父任务聚合计算，默认1 |
| `dept` / `priority` / `type` / `budget` / `actualCost` | 不维护 | 不维护 | 本期不在 UI 维护；`dept` 由 `owner` 自动推导，其余字段保留兼容 |### 4.3 人员实体引用

## 六、页面交互设计

### 6.1 入口与视图切换

在 OMP 重点工作页面顶部增加视图切换区：

```
[列表] [看板] [甘特图] [👥 人员配置]
```

### 6.2 重点工作列表页

- 表格/卡片展示重点工作名称、负责人、进度、状态、主指标 Tag
- 支持关键字搜索、状态筛选
- 部门信息由负责人自动推导，不在列表中单独展示列
- 点击工作项进入详情弹窗，详情弹窗中点击「编辑」进入编辑态

### 6.3 重点工作详情弹窗（查看态）

详情弹窗采用**单面板紧凑布局**，取消多标签页，将关键信息按优先级纵向聚合在一屏内，减少用户切换成本。

**信息排序原则：**
1. **第一眼看到状态和进度**（是否健康）
2. **第二眼看到负责人与时间**（谁负责、什么时候完成）
3. **第三眼看到目标与关联**（做什么、支撑什么）
4. **最后看到执行细节**（里程碑、子任务）

```
┌─────────────────────────────────────────┐
│ 工作详情                          [编辑] │
├─────────────────────────────────────────┤
│                                         │
│  [进行中]  进度 65%                     │
│  ████████████████████░░░░░░░░░░░░░░     │
│                                         │
│  垂直细分客户运营                        │
│  通过行业深耕与区域拓展，提升大客户 ARR   │
│                                         │
│ ── 关键信息 ────────────────────────── │
│  负责人          │ 时间范围              │
│  李经理          │ 2026-01 ~ 06          │
│                                         │
│ ── 目标与关联 ──────────────────────── │
│  年度目标：Q2 大客户 ARR 提升 15%        │
│  主指标：大客户销售额（销售部）           │
│  相关指标：客户满意度（客服部）           │
│  SP 链接  │  BI 看板                      │
│                                         │
│ ── 里程碑（3） ─────────────────────── │
│  ■ 需求评审        06-01    已完成       │
│  ○ 开发完成        07-15    待办         │
│  ○ 上线验收        08-01    待办         │
│                                         │
│ ── 子任务（2） ─────────────────────── │
│  1. 行业客户拓展    王总监    60%        │
│  2. 华东客户深耕    李经理    40%        │
│                                         │
│ ── 来源 ────────────────────────────── │
│  年度经营计划  (ap_xxx)  [查看源头]       │
│                                         │
└─────────────────────────────────────────┘
```

**布局规则：**
- 顶部通栏：状态标签 + 进度条 + 工作名称 + 描述
- 「关键信息」区：2 列紧凑网格，只展示负责人、时间范围
- 「目标与关联」区：年度目标、主指标、相关指标、SP 链接、BI 看板集中展示
- 「里程碑」区：时间线形式，只展示名称、计划日期、状态，状态用图标区分
- 「子任务」区：最多展示前 5 条，超出时显示「查看全部 N 条」
- 「来源」区：仅年度计划派生任务显示，放在最底部

**不在详情弹窗展示但保留的字段：**
- 优先级、类型、部门、成员、预算、实际成本 不在查看态展示
- 上述字段仍在列表页用于筛选/展示，在编辑弹窗中维护
- 如需查看成员，点击负责人区域或进入「人员配置」视图

**交互规则：**
- 点击里程碑可展开查看详情或标记完成（若当前用户有权限）
- 点击子任务行进入子任务编辑弹窗
- 点击 SP 链接 / BI 看板在新标签页打开
- 点击「查看源头」跳转年度经营计划对应任务

### 6.8 子任务与里程碑的关系

子任务和里程碑是两种不同维度的执行追踪工具，设计目标是互补而非重复。

**职责边界：**

| 维度 | 子任务 | 里程碑 |
|------|--------|--------|
| **定义** | 重点工作按工作范围拆分的执行单元 | 重点工作按时间轴设定的关键检查点 |
| **关注点** | 谁负责、做什么、做到什么程度 | 什么时候该完成什么关键成果 |
| **负责人** | 每个子任务有独立负责人 | 里程碑通常由父任务负责人把控 |
| **进度贡献** | 直接参与父任务进度聚合 | 不直接参与聚合，用于状态校验 |
| **典型示例** | 「行业客户拓展」「华东客户深耕」 | 「需求评审完成」「上线验收通过」 |

**使用建议：**

1. **小型重点工作**：只设里程碑，不拆子任务
   - 例如：一个周期短、范围明确的改进项
   - 进度由里程碑完成比例自动驱动

2. **中大型重点工作**：既拆子任务，也设里程碑
   - 子任务用于日常执行跟踪和资源分配
   - 里程碑用于对外汇报和关键节点管控
   - 进度由子任务加权聚合驱动，里程碑状态独立展示

3. **避免混淆**：
   - 不要把子任务做成「里程碑 1」「里程碑 2」
   -  milestone 应该是可验收的节点，而不是任务名称
   - 子任务应该是可分配、可独立推进的工作包

**进度计算规则（明确优先级）：**

```
if 存在子任务:
    父任务 progress = Σ(子任务.progress × 子任务.weight) / Σ(子任务.weight)
elif 存在里程碑:
    父任务 progress = 已完成里程碑数 / 总里程碑数 × 100
else:
    父任务 progress = 手动填报值（默认 0）
```

**里程碑与子任务的 optional 关联（未来扩展）：**
- 一个里程碑可由多个子任务的完成触发
- 子任务完成后，系统提示「是否将关联里程碑标记为完成」
- 本期不强制绑定，保持灵活

**子任务是否需要自己的里程碑？**

**本期设计：子任务不设里程碑。**

原因：
1. **保持子任务轻量**：子任务是执行单元，已有负责人、起止时间、进度、成员，再加里程碑会让层级过深
2. **避免管理碎片化**：一个重点工作的里程碑应该统一在父任务层面管理，作为跨子任务的检查点
3. **本期只支持一级子任务**：如果子任务复杂到需要里程碑，说明它应该被提升为独立重点工作

**判断标准：**
- 子任务能在 1-2 周内完成 → 用子任务即可
- 子任务需要跨月度、多个检查点 → 考虑把它升级成重点工作，再拆更小的子任务

**未来若子任务支持里程碑：**
- 只在子任务详情中展示，不参与父任务进度聚合
- 父任务里程碑仍由父任务负责人统一维护
