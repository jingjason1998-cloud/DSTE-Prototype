## 一、产品概述

### 1.1 产品定位

DSTE 战略管理平台是一款**企业级战略管理 SaaS 产品**，基于华为 DSTE 方法论，融合帆软 BI 数据能力，帮助企业实现从战略制定到执行评估的完整闭环管理。

**核心价值主张**：
- 让战略从"纸上"落到"地上"——可视化、可追踪、可量化
- 打通战略-解码-执行-评估全链路
- 用数据驱动战略决策，而非经验直觉

### 1.2 DSTE 方法论简介

| 阶段 | 英文 | 中文 | 核心产出 | 时间跨度 |
|------|------|------|---------|---------|
| **SP** | Strategy Plan | 战略制定 | 3-5 年战略规划、战略意图、业务设计 | 3-5 年 |
| **BP** | Business Plan | 战略解码 | 年度经营计划、KPI 体系、重点工作 | 1 年 |
| **Execute** | Execution | 战略执行 | 项目落地、资源分配、进度监控 | 持续 |
| **Review** | Review & Improve | 战略评估 | 经营分析会、复盘改进、战略刷新 | 季度/年度 |

### 1.3 产品形态

- **Web 端**: 主要工作平台，面向所有用户
- **移动端**: 战略驾驶舱查看、预警推送、审批（二期）
- **大屏端**: 企业经营分析会展示（二期）

### 模块 1: 战略地图 (Strategy Map)

#### 1.1 模块职责
- **做什么**: 基于平衡计分卡 (BSC) 四维度构建企业战略地图，可视化展示战略目标及因果关系
- **不做什么**: 不处理具体的 KPI 数值计算（由 KPI 模块负责）、不管理项目执行（由执行模块负责）

#### 1.2 涉及的数据实体

| 实体 | 说明 | 关键字段 |
|------|------|---------|
| **战略地图** | 一个战略周期的完整地图 | 地图ID、名称、周期、版本、状态 |
| **战略主题** | 地图中的核心战略方向 | 主题ID、地图ID、名称、描述、排序 |
| **战略目标** | 具体的战略目标节点 | 目标ID、主题ID、名称、描述、维度、层级 |
| **目标关联** | 战略目标之间的因果关系 | 关联ID、源目标ID、目标目标ID、关联类型 |
| **战略维度** | BSC 四维度定义 | 维度ID、名称（财务/客户/流程/学习）、描述 |

#### 1.3 功能列表

| 功能 | 用户故事 | 输入 | 输出 |
|------|---------|------|------|
| 创建战略地图 | 战略管理员希望创建新的战略周期地图 | 地图名称、周期、模板选择 | 空白战略地图 |
| 编辑战略主题 | 战略管理员希望调整战略方向 | 主题名称、描述、颜色 | 更新后的主题 |
| 添加战略目标 | 战略管理员希望在地图中添加目标节点 | 目标名称、所属维度、层级、描述 | 地图上的新节点 |
| 建立目标关联 | 战略管理员希望设置目标间的因果关系 | 源目标、目标目标、关联说明 | 带箭头的关联线 |
| 拖拽布局 | 战略管理员希望调整地图布局 | 鼠标拖拽操作 | 新的节点位置 |
| 版本管理 | 战略管理员希望保存不同版本 | 版本说明 | 可回溯的版本历史 |
| 地图对比 | 高管希望对比不同周期的战略变化 | 两个版本ID | 差异对比视图 |
| 地图导出 | 用户希望导出地图用于汇报 | 导出格式(PDF/PNG) | 导出文件 |

#### 1.4 与其他模块的关系
- **依赖**: 无（基础模块）
- **被依赖**:
  - KPI 模块: 读取战略目标，生成对应 KPI
  - 执行模块: 读取战略目标，生成重点工作
  - 评估模块: 读取战略目标，进行达成评估

#### 1.5 关键规则
- 一个战略周期只能有一个"生效中"的战略地图
- 战略目标必须归属到 BSC 四维度之一
- 目标关联不能形成循环依赖（A→B→C→A）
- 战略地图发布前需要审批流程

### 模块 1.5: 战略专题管理 (Strategy Topic Management)

> **定位**: 战略专题是战略地图的**补充和深化**。战略地图展示的是"全景图"，而战略专题聚焦**特定战略议题的深入研究和管理**，如"数字化转型""国际化拓展""第二曲线培育"等。每个专题都是一个独立的"迷你战略项目"，有自己的目标、路径、资源和里程碑。

#### 1.5.1 模块职责
- **做什么**:
  - 发起和管理跨周期的重大战略专题
  - 为每个专题建立独立的研究框架、目标体系和行动计划
  - 跟踪专题进展，评估专题对整体战略的贡献
  - 积累专题知识和经验，形成企业战略知识资产
- **不做什么**: 不替代项目管理系统（专题下的具体项目仍由执行模块管理）、不做日常运营议题

#### 1.5.2 涉及的数据实体

| 实体 | 说明 | 关键字段 |
|------|------|---------|
| **战略专题** | 独立战略议题 | 专题ID、名称、类型、发起人、起止时间、状态 |
| **专题目标** | 专题要达成的具体目标 | 目标ID、专题ID、目标描述、衡量指标、目标值 |
| **专题路径** | 达成目标的策略路径 | 路径ID、专题ID、阶段名称、关键举措、里程碑 |
| **专题资源** | 投入的资源 | 资源ID、专题ID、资源类型、预算、负责人 |
| **专题成果** | 专题产出的成果 | 成果ID、专题ID、成果类型、描述、价值评估 |
| **专题关联** | 专题与战略地图的关联 | 关联ID、专题ID、目标ID、关联类型 |

#### 1.5.3 功能列表

| 功能 | 用户故事 | 输入 | 输出 |
|------|---------|------|------|
| **发起专题** | 高管希望发起重大战略议题 | 专题名称、背景、预期目标 | 新建专题 |
| **设计专题框架** | 战略管理员希望设计专题研究框架 | 专题ID、研究维度 | 专题框架（目标+路径+资源） |
| **关联战略目标** | 战略管理员希望将专题与战略地图关联 | 专题ID、战略目标ID | 关联关系 |
| **跟踪专题进展** | 发起人希望了解专题推进情况 | 专题ID | 进展看板 |
| **评估专题价值** | 高管希望评估专题对战略的贡献 | 专题成果数据 | 价值评估报告 |
| **专题知识沉淀** | 战略管理员希望沉淀专题经验 | 专题全过程数据 | 知识文档 |
| **专题库管理** | 用户希望查看历史专题 | 筛选条件 | 专题列表 |

#### 1.5.4 与其他模块的关系
- **依赖**:
  - 战略地图: 获取战略目标作为专题方向来源
- **被依赖**:
  - 执行模块: 专题分解为具体项目执行
  - 经营分析会: 专题进展作为会议议题
  - AI助手: 专题分析和建议

#### 1.5.5 关键规则
- 战略专题必须由高管发起，经战略委员会审批
- 每个专题必须有明确的起止时间和退出标准
- 专题数量建议控制在 3-5 个（聚焦原则）
- 专题成果必须评估对整体战略的贡献度

### 模块 2: KPI 指标体系 (KPI Management)

#### 2.1 模块职责
- **做什么**: 企业级指标库管理，支持多层级 KPI 分解、自动计算、预警提醒
- **不做什么**: 不直接采集业务数据（由数据集成模块负责）、不处理项目进度（由执行模块负责）

#### 2.2 涉及的数据实体

| 实体 | 说明 | 关键字段 |
|------|------|---------|
| **指标库** | 企业级指标定义 | 指标ID、名称、定义、计算公式、单位、频次 |
| **KPI 实例** | 具体战略周期中的 KPI | 实例ID、指标ID、战略地图ID、目标值、权重 |
| **KPI 分解** | 层级分解关系 | 分解ID、父KPIID、子KPIID、分解比例 |
| **KPI 数据** | 实际采集的数据 | 数据ID、实例ID、周期、实际值、状态 |
| **预警规则** | KPI 预警配置 | 规则ID、实例ID、阈值类型、阈值、通知方式 |

#### 2.3 功能列表

| 功能 | 用户故事 | 输入 | 输出 |
|------|---------|------|------|
| 创建指标 | 战略管理员希望定义新的指标 | 指标名称、定义、公式、单位 | 指标库新条目 |
| 从战略导出 KPI | 战略管理员希望将战略目标转为 KPI | 战略目标列表 | 初始 KPI 列表 |
| 设置目标值 | 部门负责人希望设定季度目标 | KPI 实例ID、目标值、挑战值 | 更新后的目标 |
| 分解 KPI | 部门负责人希望将 KPI 分解到下级 | 父 KPI、子部门、分解比例 | 层级 KPI 树 |
| 数据录入 | 部门负责人希望录入实际数据 | KPI 实例ID、周期、实际值 | 更新后的达成率 |
| 自动采集 | 系统希望从 BI 自动获取数据 | 数据源配置、采集周期 | 自动填充的数据 |
| 预警配置 | 战略管理员希望设置预警规则 | KPI、阈值、通知人 | 生效的预警规则 |
| KPI 看板 | 用户希望查看 KPI 达成情况 | 组织范围、时间周期 | 可视化看板 |
| 趋势分析 | 用户希望分析 KPI 历史趋势 | KPI 实例ID、时间范围 | 趋势图表 |
| 对标分析 | 用户希望与目标值对比 | KPI 实例ID、对比维度 | 差距分析图 |

#### 2.4 与其他模块的关系
- **依赖**:
  - 战略地图: 获取战略目标作为 KPI 来源
  - 数据集成: 获取自动采集的数据源
- **被依赖**:
  - 战略驾驶舱: 读取 KPI 数据进行展示
  - 执行模块: 读取 KPI 作为重点工作目标
  - 评估模块: 读取 KPI 进行达成评估

#### 2.5 关键规则
- KPI 目标值必须满足 SMART 原则（具体、可衡量、可达成、相关、有时限）
- 分解到子级的 KPI 权重之和必须等于父级 100%
- 自动采集优先于手动录入，手动录入需留痕
- 预警触发后需在 24 小时内响应

### 模块 3: BEM 战略解码 (Strategy Decoding)

#### 3.1 模块职责
- **做什么**: 基于 BEM (Business Execution Model) 六步解码法，将战略转化为可执行的行动计划
- **不做什么**: 不管理项目执行过程（由执行模块负责）、不处理日常任务（由 OA 系统负责）

#### 3.2 涉及的数据实体

| 实体 | 说明 | 关键字段 |
|------|------|---------|
| **CSF** | 关键成功因素 | CSFID、名称、描述、关联战略目标 |
| **CTQ** | 关键质量特性 | CTQID、CSFID、名称、衡量标准 |
| **重点工作** | 解码后的战略举措 | 工作ID、名称、负责人、起止时间、状态 |
| **里程碑** | 重点工作的关键节点 | 里程碑ID、工作ID、名称、计划日期、完成日期 |
| **责任矩阵** | RACI 责任分配 | 矩阵ID、工作ID、角色、责任人 |

#### 3.3 功能列表

| 功能 | 用户故事 | 输入 | 输出 |
|------|---------|------|------|
| 识别 CSF | 战略管理员希望找出关键成功因素 | 战略目标、讨论记录 | CSF 列表 |
| 导出 CTQ | 战略管理员希望将 CSF 量化为指标 | CSF 列表 | CTQ/Y 树 |
| 生成重点工作 | 部门负责人希望将 CTQ 转为项目 | CTQ 列表、资源评估 | 重点工作清单 |
| 设定里程碑 | 项目负责人希望规划关键节点 | 工作ID、里程碑名称、日期 | 里程碑计划 |
| 分配责任 | 项目负责人希望明确 RACI | 工作ID、参与人、角色 | 责任矩阵 |
| 解码评审 | 高管希望评审解码结果 | 解码完整方案 | 评审意见、审批状态 |
| 解码追踪 | 战略管理员希望追踪解码进度 | 战略周期 | 解码完成度报告 |

#### 3.4 与其他模块的关系
- **依赖**:
  - 战略地图: 获取战略目标作为解码输入
  - KPI 模块: 将 CTQ 转为可量化的 KPI
- **被依赖**:
  - 执行模块: 读取重点工作进行项目管理

#### 3.5 关键规则
- 解码必须覆盖所有战略目标，不允许遗漏
- 每个重点工作必须有明确的负责人和截止日期
- 解码结果必须经过高管评审才能生效
- 重点工作数量建议控制在 5-8 项（聚焦原则）

### 模块 4: 重点工作管理 (Key Task Management)

#### 4.1 模块职责
- **做什么**: 战略举措的项目化管理，里程碑跟踪、资源分配、风险预警
- **不做什么**: 不替代专业项目管理工具（如 Jira），聚焦战略级项目

#### 4.2 涉及的数据实体

### 模块 4.5: 业务专题管理 (Business Topic Management)

> **定位**: 业务专题是**战略执行层面的议题聚合机制**。与"战略专题"（聚焦中长期战略议题）不同，业务专题聚焦**年度或季度内的关键业务问题**，如"Q3 大客户攻坚""双十一促销专项""供应链降本"等。业务专题将相关的重点工作、KPI、资源聚合成一个"作战单元"，便于集中管理和跟踪。
>
> **与战略专题的区别**:
> | 维度 | 战略专题 | 业务专题 |
> |------|---------|---------|
> | **时间跨度** | 1-3年 | 季度-年度 |
> | **发起层级** | 高管/战略委员会 | 部门负责人/高管 |
> | **聚焦内容** | 战略方向探索 | 业务问题攻克 |
> | **关联模块** | 战略地图 | 重点工作/KPI |
> | **典型示例** | 数字化转型、国际化 | Q3大客户攻坚、降本专项 |
> | **当前实现状态** | 规划中 | **v2.1 已落地** ✅ |

#### 4.5.1 模块职责

**概念层面（产品规划）**:
- 发起和管理阶段性业务专题
- 将相关的重点工作、KPI、资源聚合到专题下统一管理
- 跟踪专题整体进展和成效
- 专题结束后进行复盘和价值评估

**当前实现层面（v2.1 已落地）**:
- `business-topics.html` 独立页面承载完整 CRUD
- 10条基于经营分析会真实专题的初始化数据（预测能力、项目群经营、折扣管理、合同质量、授权管理、应收款管理、四算专项、费控专项、收入确认、制衡关系）
- 里程碑驱动的进度自动计算
- 部门自由输入 + 筛选目录自动去重更新
- 结项总结字段支持专题复盘记录
- JSON 数据导出
- **v2.1 新增**: 议题数据池（ST/AT 导入管理）
- **v2.1 新增**: 专题-议题关联（手动关联 + 关系类型 + 统计）
- **v2.1 新增**: 知识地图 SVG 可视化
- **v2.1 新增**: AI 全局分析报告（规则引擎 + 缓存）

**不做什么**:

**前端实现数据模型（v1.2 实际落地）**:

```typescript
interface BusinessTopic {
  id: string;                    // bt_时间戳，safeId() 白名单校验
  name: string;                  // 专题名称 [必填]
  description: string;           // 一句话描述
  target: string;                // 专题目标
  priority: 'P0' | 'P1' | 'P2';  // 优先级 [必填]
  status: 'preparing' | 'in_progress' | 'done' | 'archived' | 'delayed';
  owner: string;                 // 负责人 [必填]
  department: string;            // 责任部门（自由输入，非下拉）
  progress: number;              // 整体进度 0-100（自动计算，可覆盖）
  startDate: string;             // YYYY-MM-DD
  endDate: string;               // YYYY-MM-DD
  budget: number;                // 预算（万元）
  actualCost: number;            // 实际成本（万元）
  milestones: Milestone[];       // 里程碑列表（表单内动态增删）
  tags: string[];                // 标签
  summary: string;               // 结项总结（专题结束后填写）
  linkedIssues: LinkedIssue[];    // 【v2.1】关联的议题列表
  linkedKpis: LinkedKpi[];        // 【v2.1】关联的KPI列表（预留字段激活）
  issueStats: TopicIssueStats;    // 【v2.1】议题关联统计（派生）
  dataVersion: number;            // 【v2.1】数据版本号，用于迁移检测
  // linkedStrategyTopicId: string;  // 【预留】战略专题关联字段，暂不实现
  // linkedMeetingIds: string[];     // 【预留】关联会议
  // linkedTodoIds: string[];        // 【预留】关联待办
  // linkedDecisionIds: string[];    // 【预留】关联决议
  createdAt: string;
  updatedAt: string;
}

interface Milestone {
  id: string;
  name: string;
  date: string;                  // YYYY-MM-DD
  status: 'pending' | 'completed';
}
```

**进度自动计算规则**: `progress = Math.round(已完成里程碑数 / 总里程碑数 × 100)`，无里程碑时默认为 0。

#### 4.5.6 前端技术实现设计（v1.2）

**技术栈**: 单页 HTML + 原生 JavaScript + localStorage（纯前端，无后端）

**数据持久化**:
- `STORAGE_KEY = 'dste_business_topics_v2'`
- `loadTopics()` / `saveTopics()` — localStorage 读写，返回浅拷贝防突变
- `initDefaultData()` — 10条真实专题初始化（首次访问自动填充）

**核心函数**:
- `renderTable()` / `renderStats()` — 动态渲染列表和统计卡片
- `renderDeptFilter()` — 从数据中动态提取部门去重排序
- `openFormModal(id?)` / `saveTopic()` — 新建/编辑
- `openDetailModal(id)` — 详情弹窗
- `computeProgress(milestones)` — 里程碑完成比例自动计算进度
- `generateId()` — 唯一 ID 生成

**安全防护**:
- `escapeHtml()` — 所有动态插入内容转义
- `safeId(id)` — 白名单校验 `/^bt_[a-zA-Z0-9_]+$/`，防止内联事件 XSS
- 无 `innerHTML` 直接拼接用户输入（经转义后安全）

### 模块 6: 经营分析会 (Business Review) — 战略执行核心引擎

> **定位升级**: 经营分析会不是简单的"会议管理"，而是**战略执行的中央枢纽**。它将战略解码的产出（KPI+重点工作）与执行层的日常运营连接，通过"会前数据驱动准备 → 会中结构化讨论 → 会后行动项闭环"的三段式机制，确保战略执行不偏离轨道。
>
> **核心理念**: 参考华为"一报一会"制度（经营报告+经营分析会）、IBM BLM 模型的执行闭环、以及 OKR 的 Check-in 机制，将经营分析会打造为**战略执行的节拍器**。

#### 6.1 模块职责
- **做什么**:
  - **会前**: 自动汇聚战略执行数据，生成结构化会议材料，识别异常和风险点
  - **会中**: 提供数据驱动的会议引导，支持差距分析、根因探讨、决议记录
  - **会后**: 自动生成行动项并跟踪闭环，形成"会议→行动→验证→再会议"的飞轮
  - **跨期**: 建立会议之间的关联，形成战略执行的连续叙事
- **不做什么**: 不替代会议协作工具（如飞书会议）的音视频能力，不处理非战略相关的日常运营会议

#### 6.2 经营分析会体系设计

##### 6.2.1 三层会议体系

| 会议类型 | 频率 | 参与人 | 核心议题 | 时长 | 产出 |
|---------|------|--------|---------|------|------|
| **月度经营会** | 每月 | 部门负责人+战略管理员 | 上月KPI达成、重点工作进度、风险预警、下月计划 | 2h | 月度经营报告、行动项 |
| **季度战略会** | 每季 | 高管团队+部门负责人 | 季度战略复盘、差距根因、战略调整、资源重配 | 4h | 季度战略评估报告、战略刷新 |
| **年度复盘会** | 每年 | 全管理层 | 年度战略达成、组织能力建设、下年战略方向 | 1-2天 | 年度复盘报告、新周期SP/BP |

##### 6.2.2 "一报一会"机制

参考华为实践，每个会议必须配套一份**经营报告**，形成固定节奏：

```
经营报告生成（会前3天）
    ↓
材料预审（会前1天）
    ↓
经营分析会召开
    ├── 上期行动项回顾（15min）
    ├── KPI达成分析（30min）
    ├── 重点工作进度（30min）
    ├── 差距与根因（30min）
    ├── 决议与行动（30min）
    └── 下阶段重点（15min）
    ↓
会议纪要+行动项（会后24h）
    ↓
行动项跟踪（持续至下次会议）
    ↓
闭环验证（下次会议回顾）
```

| **会议纪要** | 会议结论和待办 | 纪要ID、会议ID、决议、责任人、优先级 |

| **决议时效** | 会议纪要必须在会后24小时内发布；行动项必须在纪要发布后4小时内派发 |

| **会议纪律** | 参会人必须提前阅读材料；会议中聚焦数据和事实，减少主观描述 |
| **效果评估** | 每次会议后参会人匿名评分；季度统计会议效率指标 |

### 模块 7: AI 战略助手 (AI Strategy Assistant) — 智能决策增强引擎

> **定位升级**: AI 战略助手不是简单的"问答机器人"，而是**战略管理的"副驾驶"**——它深度理解企业战略上下文，主动发现问题、提供洞察、辅助决策，让战略管理从"人找数据"变为"数据找人"。
>
> **设计理念**: 参考 Microsoft Copilot for M365 的嵌入式 AI 体验、Palantir AIP 的数据+AI 融合、以及 Glean 的企业知识搜索，打造"无处不在、按需出现"的 AI 战略伙伴。

#### 7.1 模块职责
- **做什么**:
  - **主动洞察**: 自动扫描战略数据，主动推送异常、趋势、机会
  - **智能问答**: 用自然语言查询任何战略相关问题，获得数据支撑的回答
  - **报告生成**: 基于模板和数据自动生成各类战略报告
  - **预测模拟**: 基于历史数据预测 KPI 走势，支持 What-if 模拟
  - **根因分析**: 自动分析 KPI 偏差背后的深层原因
  - **对标学习**: 结合行业数据和最佳实践提供改进建议
- **不做什么**: 不做最终决策（仅提供建议供人参考）、不替代专业战略咨询、不处理非战略相关的日常运营问题

#### 7.2 AI 能力架构

```
┌─────────────────────────────────────────────────────────────────┐
│                      AI 战略助手能力层                           │
├─────────────┬─────────────┬─────────────┬─────────────────────┤
│   感知层     │   认知层     │   决策层     │      交互层          │
│  (Perceive) │  (Comprehend)│  (Decide)   │    (Interact)       │
├─────────────┼─────────────┼─────────────┼─────────────────────┤
│ • 数据扫描   │ • 语义理解   │ • 根因分析   │ • 嵌入式助手        │
│ • 异常检测   │ • 知识检索   │ • 预测模拟   │ • 对话式界面        │
│ • 趋势识别   │ • 模式识别   │ • 建议生成   │ • 报告生成器        │
│ • 关联发现   │ • 因果推断   │ • 方案对比   │ • 智能通知          │
└─────────────┴─────────────┴─────────────┴─────────────────────┘
         ↑              ↑              ↑              ↑
    ┌─────────────────────────────────────────────────────────┐
    │                    企业战略知识库                          │
    │  战略文档 │ KPI数据 │ 行业报告 │ 历史会议 │ 最佳实践 │ 方法论 │
    └─────────────────────────────────────────────────────────┘
```

#### 7.3 核心能力详解

##### 7.3.1 感知层 — 主动发现

**能力**: 系统自动扫描全量战略数据，主动识别值得关注的事件

| 感知类型 | 触发条件 | 示例 |
|---------|---------|------|
| **异常检测** | KPI 偏离目标 > 15% | "Q3 客户满意度 NPS 下降至 62，低于目标 70" |
| **趋势预警** | 连续 3 期下滑 | "新产品收入占比连续 3 季度下滑，当前 28.4%" |
| **里程碑风险** | 里程碑延期 > 3 天 | "'渠道拓展'里程碑延期 5 天，可能影响 Q4 目标" |
| **关联异常** | 相关 KPI 同时异常 | "营收增长但利润率下降，可能存在价格战" |
| **机会识别** | 超预期表现 | "华东区营收同比增长 35%，建议复制经验" |

**推送策略**:
- 紧急异常：即时推送（钉钉/企业微信/邮件）
- 日常洞察：每日晨报（8:00）
- 周期性总结：周报/月报自动生成

##### 7.3.2 认知层 — 深度理解

**能力**: 理解战略上下文，建立知识关联

| 功能 | 说明 | 示例 |
|------|------|------|

### 模块 6.5: 绩效与激励 (Performance & Incentive)

> **定位**: 绩效与激励是**战略评估的量化落地机制**。它将战略解码的KPI和重点工作的完成结果，转化为组织和个人绩效评价，并与激励机制挂钩，形成"战略→目标→执行→评价→激励"的完整闭环。
>
> **核心理念**: 参考华为"以奋斗者为本"的绩效文化、OKR与绩效评价的分离设计、以及平衡计分卡的绩效应用，将战略达成度与组织活力、个人成长有机结合。

#### 6.5.1 模块职责
- **做什么**:
  - **组织绩效**: 基于KPI达成率和重点工作完成度，评估部门/团队的战略贡献
  - **个人绩效**: 将组织绩效分解到个人，支持KPI+重点工作+价值观的多维评价
  - **激励设计**: 将绩效结果与薪酬、晋升、股权等激励手段关联
  - **绩效校准**: 支持跨部门绩效校准，确保评价公平性
  - **绩效改进**: 识别绩效短板，生成改进计划
- **不做什么**: 不替代HR系统的薪酬计算和发放、不处理日常考勤等非战略绩效

#### 6.5.2 涉及的数据实体

| 实体 | 说明 | 关键字段 |
|------|------|---------|
| **绩效周期** | 绩效评价的时间周期 | 周期ID、名称、起止时间、状态 |
| **组织绩效** | 部门/团队的绩效评价 | 评价ID、组织ID、周期ID、绩效等级、得分 |
| **个人绩效** | 员工的绩效评价 | 评价ID、员工ID、周期ID、绩效等级、得分 |
| **绩效指标** | 评价的具体指标 | 指标ID、评价ID、指标类型、权重、目标值、实际值 |
| **激励方案** | 激励规则配置 | 方案ID、名称、适用对象、激励类型、计算规则 |
| **激励记录** | 具体的激励发放 | 记录ID、方案ID、对象ID、金额、发放时间 |
| **绩效校准** | 跨部门校准记录 | 校准ID、周期ID、校准会议、调整记录 |

#### 6.5.3 功能列表

| 功能 | 用户故事 | 输入 | 输出 |
|------|---------|------|------|
| **绩效方案配置** | HR希望配置绩效评价规则 | 评价维度、权重、等级定义 | 绩效方案 |
| **组织绩效计算** | 系统希望自动计算部门绩效 | KPI数据、重点工作完成度 | 组织绩效得分 |
| **个人绩效评估** | 主管希望对下属进行绩效评价 | 员工数据、自评、他评 | 个人绩效等级 |
| **绩效校准会议** | HR希望组织跨部门绩效校准 | 各部门绩效分布 | 校准后的绩效结果 |
| **激励计算** | 系统希望根据绩效计算激励 | 绩效结果、激励方案 | 激励金额 |
| **绩效看板** | 高管希望查看整体绩效分布 | 组织范围、周期 | 绩效分布图 |
| **绩效改进计划** | 系统希望为低绩效员工生成改进计划 | 绩效短板分析 | 改进计划 |
| **绩效历史查询** | 用户希望查看历史绩效 | 员工ID、时间范围 | 绩效历史 |

#### 6.5.4 与其他模块的关系
- **依赖**:
  - KPI模块: 获取KPI达成数据
  - 重点工作管理: 获取工作完成度
  - 经营分析会: 获取会议决议中的绩效相关结论
- **被依赖**:
  - 干部管理: 绩效结果是干部评价的重要依据
  - 驾驶舱: 展示绩效分布

#### 6.5.5 关键规则
- 组织绩效中KPI达成率权重不低于60%
- 绩效等级建议采用强制分布（如A:20%、B+:30%、B:40%、C:10%）
- 激励发放必须与绩效等级挂钩，差距要明显
- 绩效评价周期与战略周期对齐（季度/年度）

### 模块 6.6: 干部管理 (Cadre Management)

> **定位**: 干部管理是**战略执行的组织保障**。DSTE的落地需要合格的干部梯队，干部管理模块聚焦"选、用、育、留、管"全生命周期，确保战略有合适的人来执行。
>
> **核心理念**: 参考华为"干部四力"模型（决断力、执行力、理解力、连接力）、IBM的领导力模型，将干部能力与战略要求对齐。

#### 6.6.1 模块职责
- **做什么**:
  - **干部盘点**: 定期盘点干部梯队，识别能力缺口
  - **干部选拔**: 基于战略需求选拔干部，支持竞聘、任命等流程
  - **干部培养**: 制定干部培养计划，跟踪培养进展
  - **干部考核**: 对干部进行任期考核、年度考核
  - **干部流动**: 管理干部轮岗、晋升、降级、退出
- **不做什么**: 不替代HR系统的日常人事管理、不处理非干部级员工管理

#### 6.6.2 涉及的数据实体

| 实体 | 说明 | 关键字段 |
|------|------|---------|
| **干部档案** | 干部基本信息 | 档案ID、员工ID、现任职务、任职时间、干部级别 |
| **干部能力** | 能力评估 | 能力ID、档案ID、能力维度、评估得分、评估时间 |
| **干部绩效** | 历史绩效 | 绩效ID、档案ID、周期、绩效等级 |
| **干部培养** | 培养计划 | 计划ID、档案ID、培养内容、起止时间、状态 |
| **干部任免** | 任免记录 | 任免ID、档案ID、任免类型、原职务、新职务、时间 |
| **干部梯队** | 梯队规划 | 梯队ID、岗位ID、后备干部、准备度 |

#### 6.6.3 功能列表

| 功能 | 用户故事 | 输入 | 输出 |
|------|---------|------|------|
| **干部盘点** | HR希望定期盘点干部梯队 | 盘点周期、范围 | 干部盘点报告 |
| **能力评估** | 主管希望评估干部能力 | 评估维度、评分 | 能力雷达图 |
| **选拔流程** | HR希望组织干部选拔 | 岗位需求、候选人 | 选拔结果 |
| **培养计划** | HR希望制定干部培养计划 | 能力缺口、培养目标 | 培养计划 |
| **任期考核** | 系统希望提醒干部任期考核 | 任期到期时间 | 考核提醒 |
| **梯队规划** | 高管希望规划关键岗位后备 | 关键岗位、后备人选 | 梯队地图 |
| **干部流动** | HR希望管理干部任免 | 任免申请、审批 | 任免记录 |
| **干部看板** | 高管希望查看干部整体情况 | 组织范围 | 干部概况 |

#### 6.6.4 与其他模块的关系
- **依赖**:
  - 绩效与激励: 获取干部绩效数据
  - 经营分析会: 获取会议中对干部的评价
- **被依赖**:
  - 战略地图: 干部能力与战略要求对齐

#### 6.6.5 关键规则
- 关键岗位必须有后备干部（AB角）
- 干部任期一般为3年，到期必须考核
- 干部晋升必须有绩效支撑（连续2年B+以上）
- 干部轮岗周期建议2-3年

## 七、全面预算管理概览（业财融合视角）

> **设计原则**: 全面预算管理不作为一个独立模块，而是**融入各业务模块的业财融合能力**。预算数据与战略数据同源同口径，实现"战略-业务-财务"三位一体。

### 7.1 预算在DSTE全流程中的位置

```
┌─────────────────────────────────────────────────────────────────────────┐
│                        全面预算管理贯穿DSTE全流程                         │
├─────────────────┬─────────────────┬─────────────────┬─────────────────────┤
│    SP 战略制定   │   BP 战略解码    │   Execute 执行   │    Review 评估      │
├─────────────────┼─────────────────┼─────────────────┼─────────────────────┤
│ • 战略投资预算   │ • 年度经营预算   │ • 预算执行监控   │ • 预算偏差分析       │
│ • 资本支出预算   │ • 部门预算分解   │ • 滚动预测       │ • 预算执行率评估     │
│ • 并购/合作预算  │ • 项目预算编制   │ • 预算调整       │ • 下年预算基线       │
│ • 研发战略预算   │ • 人力预算       │ • 费用控制       │ • 预算考核           │
└─────────────────┴─────────────────┴─────────────────┴─────────────────────┘
```

### 7.2 预算数据模型

| 预算类型 | 说明 | 关联模块 | 时间跨度 |
|---------|------|---------|---------|
| **战略投资预算** | 支撑战略目标的重大投资 | 战略地图、战略专题 | 3-5年 |
| **年度经营预算** | 年度收入、成本、费用预算 | KPI模块（预算即目标） | 1年 |
| **部门预算** | 各部门的费用、人力预算 | 组织维度分解 | 1年 |
| **项目预算** | 重点工作/专题的专项预算 | 重点工作、业务专题 | 项目周期 |
| **滚动预测** | 基于实际执行的动态预测 | 经营分析会 | 季度更新 |

### 7.3 预算与模块的融合设计

#### 7.3.1 战略地图 + 战略投资预算

| 融合点 | 设计说明 |
|--------|---------|
| **战略意图关联预算** | 每个战略意图可关联战略投资预算，如"国际化拓展"关联海外市场开拓预算 |
| **目标节点预算** | 战略目标可设置达成目标所需的资源预算 |
| **投资优先级** | 基于战略重要性对投资项目进行优先级排序 |

#### 7.3.2 KPI模块 + 年度经营预算

| 融合点 | 设计说明 |
|--------|---------|
| **预算即目标** | KPI的目标值可直接来自年度预算，如"营收目标=预算收入" |
| **预算达成率** | KPI实例增加"预算达成率"指标，=实际值/预算值 |
| **预算分解** | 年度预算按组织层级分解，与KPI分解同步 |
| **预算调整联动** | 预算调整时自动更新关联KPI的目标值 |

**KPI实例增加预算字段**:
```
kpi_instance 表增加:
- budget_value: 预算值（与target_value区分）
- budget_completion_rate: 预算达成率
- budget_adjustment_log: 预算调整记录
```

#### 7.3.3 重点工作/业务专题 + 项目预算

| 融合点 | 设计说明 |
|--------|---------|
| **工作预算编制** | 创建重点工作时同步编制项目预算 |
| **预算执行跟踪** | 工作进度更新时同步更新预算执行进度 |
| **成本预警** | 项目成本超预算时自动预警 |
| **专题预算聚合** | 业务专题自动聚合关联工作的预算 |

**重点工作表增加预算字段**:
```
key_task 表增加:
- budget: 项目预算
- actual_cost: 实际成本
- cost_completion_rate: 成本执行率
- budget_variance: 预算偏差
```

#### 7.3.4 经营分析会 + 预算分析

| 融合点 | 设计说明 |
|--------|---------|
| **预算执行回顾** | 经营分析会标准议程增加"预算执行回顾" |
| **预算偏差分析** | 自动分析预算偏差，识别超支/节支原因 |
| **滚动预测讨论** | 基于实际执行调整年度预测 |
| **预算调整决议** | 会议决议可直接触发预算调整流程 |

**经营报告增加预算章节**:
| 章节 | 内容 |
|------|------|
| 预算执行总览 | 收入/成本/费用预算达成率 |
| 预算偏差分析 | 重大偏差项及根因 |
| 滚动预测 | 基于实际的年度预测更新 |
| 预算调整建议 | AI生成的调整建议 |

#### 7.3.5 战略驾驶舱 + 预算看板

| 融合点 | 设计说明 |
|--------|---------|
| **预算达成组件** | 驾驶舱增加预算达成率卡片 |
| **预算趋势图** | 预算vs实际vs预测的趋势对比 |
| **预算预警组件** | 超预算风险预警 |
| **部门预算视图** | 按组织维度查看预算执行 |

### 7.4 预算管理关键流程

#### 7.4.1 年度预算编制流程

```
战略地图确定 → 战略目标量化 → KPI目标值设定 → 预算目标分解
     ↓                                              ↓
部门预算编制 ← 部门KPI承接 ← 部门目标对齐 ← 预算模板下发

     ↓

     ↓

```

#### 7.4.2 预算执行监控流程

```
日常业务发生 → 财务系统记录 → 数据集成同步 → KPI模块更新
                                              ↓
                                         预算达成率计算
                                              ↓
                                    偏差分析（AI辅助）
                                              ↓
                                    预警/正常 → 经营分析会讨论

```

#### 7.4.3 预算调整流程

```

     ↓

     ↓
影响分析（AI辅助：对KPI/其他部门的影响）
     ↓
预算评审会
     ↓
审批（按权限：分管领导/CEO/董事会）
     ↓
预算更新 → KPI目标同步更新 → 通知相关方
```

### 7.5 预算核心指标

| **预算调整频次** | 年度调整次数 | < 3次 |

## 八、网页信息架构设计

>

### 8.1 设计原则

采用**混合架构**：独立页面承载列表/总览，弹窗承载详情/预览，AI助手以浮窗形式全局可用。

| 交互模式 | 适用场景 | 优势 |
|---------|---------|------|
| **独立页面** | 列表展示、复杂数据表格、长时间操作 | 空间大、可收藏链接、浏览器历史 |
| **弹窗(Modal)** | 详情查看、快速预览、表单填写 | 上下文不丢失、操作轻量、演示聚焦 |
| **AI浮窗** | 全局随时唤起、对话式交互 | 不打断工作流、即时响应 |
