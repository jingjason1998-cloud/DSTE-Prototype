# 会议材料智能审核助手 — 统一产品设计文档

> **文档版本**：v1.0  
> **产品模块**：DSTE 战略管理平台 · 执行阶段 · 会议材料审核（`reviewer.html`）  
> **关联场景**：垂直客群-落后述职、落后战区业绩承诺会、负责人年度述职、通用议题材料审核  
> **状态**：部分实现，持续迭代中

## 一、产品定位与价值主张

### 1.1 一句话定位

企业经营分析会议的「材料预审官」，在会议召开前自动完成材料质量把关，让每一份上会的材料都经得起问、对得上数、说得清逻辑。

### 1.2 核心价值

| 维度 | 定义 |
|------|------|
| **核心用户** | 汇报人（业务负责人）、会议组织者（财务BP/战略部） |
| **解决痛点** | 会议材料质量参差 → 会议效率低 → 决策质量差 |
| **价值主张** | 让每一份上会的材料，都经得起问、对得上数、说得清逻辑 |
| **成功标准** | 材料一次通过率达到 80% 以上；上会追问率下降 50% |

### 1.3 产品边界

| 范围内 | 范围外 |
|--------|--------|
| 材料质量预审与评分 | 会议纪要与决议跟踪（由经营分析会模块负责） |
| 单材料/批量材料审核 | 材料内容的自动修改/重写 |
| 审核历史与版本对比 | 材料真实性的深度核查（仅做表面一致性检查） |
| 跨页面得分共享（cockpit 议程展示） | 审批流程与工作流引擎 |

## 二、用户角色与核心场景

### 2.1 角色地图

```
┌─────────────────────────┐     ┌─────────────────────────┐
│        汇报人            │     │      会议组织者         │
│    （业务负责人）         │     │   （财务BP / 战略部）   │
└───────────┬─────────────┘     └───────────┬─────────────┘
            │                               │
            ▼                               ▼
  · 写材料前看标准                  · 创建会议，配置审核规则
  · 写完一键自检                    · 批量审核各部门材料
  · 按反馈修改完善                  · 质量把关，不通过则打回
  · 提交最终版                      · 汇总状态，确认准入
```

### 2.2 核心场景

| 场景 | 用户 | 任务 | 期望结果 |
|------|------|------|----------|
| **自检** | 汇报人 | 材料写完后，想先自己过一遍 | 5分钟知道哪里要补、哪里要改 |
| **预审把关** | 会议组织者 | 收到多个部门材料，要统一审核质量 | 批量处理，自动标红问题，优先处理致命错误 |
| **会前确认** | 会议组织者 | 确认所有材料是否达到上会标准 | 一眼看清：几份已过、几份待改、几份未交 |

## 三、审核场景体系

### 3.1 场景总览

| 场景 ID | 场景名称 | 材料类型 | 审核侧重点 | 当前状态 |
|---------|---------|---------|-----------|---------|
| `vertical-segment-review` | 垂直客群-落后述职 | 述职材料 | 业绩差距、根因深度、计划闭环、SP对齐、态度反思 | 已配置 6 维标准 |
| `lagging-region-review` | 落后战区业绩承诺会 | 业绩承诺材料 | 目标承诺、行动具体化、材料规范度、数据质量 | 已配置 4 维标准（R级制+一票否决） |
| `annual-leader-review` | 负责人年度述职 | 年度述职材料 | 完整性、SP关联、态度反思、下一步计划 | 已配置 4 维标准 |
| `general-topic-review` | 通用议题材料审核 | 议题决策材料 | 目标-方案对齐、决策支撑、行动具体化、材料规范度 | 已配置 4 维标准 |

### 3.2 场景选择逻辑

```
用户输入 KMS 链接
    │
    ▼
选择审核场景（下拉框）
    │
    ├── 垂直客群-落后述职 → 6维述职模型
    ├── 落后战区业绩承诺会 → 4维承诺模型（R级制）
    ├── 负责人年度述职 → 4维述职模型
    └── 通用议题材料审核 → 4维议题决策模型
    │
    ▼
调用对应场景审核标准 → AI 生成报告
```

## 四、审核标准矩阵

### 4.1 垂直客群-落后述职（6维 / 100分）

| 维度 | 权重 | 满分 | 核心拷问 | 红线/底线 |
|------|------|------|---------|----------|
| **完整性** | 35% | 35 | 6大模块是否齐全？ | 缺少市场现状/业绩回顾 → 严重扣分 |
| **差距与根因分析** | 20% | 20 | 差距是否量化？根因是否深入本质？ | **< 12分 → 单项底线，直接待修改** |
| **下一步计划** | 20% | 20 | 措施是否闭环、SMART？ | 措施脱节/无责任人/无时间节点 → 打回 |
| **业绩预测达成概率分析** | 10% | 10 | 未来预测是否有逻辑和概率支撑？ | 无概率评估 → 打回完善 |
| **SP战略关联度** | 10% | 10 | 是否承接公司SP战略方向？ | 未提及战略 → 打回补充 |
| **态度与反思** | 5% | 5 | 是否坦诚反思、不推诿？ | 推诿/回避 → 扣分 |

**通过标准**：总分 ≥ 80分，且差距与根因分析 ≥ 12分。

### 4.2 落后战区业绩承诺会（4维 / 100分）

| 维度 | 权重 | 满分 | 核心拷问 | 一票否决 |
|------|------|------|---------|---------|
| **目标承诺** | 20% | 20 | 业绩承诺是否明确、完整、可衡量？ | **目标承诺 = 0 分 → R5 退回重写** |
| **行动具体化** | 30% | 30 | 行动方案是否符合 SMART 原则？ | **行动具体化 < 10 分 → R5 退回重写** |
| **材料规范度** | 35% | 35 | 必需章节是否齐全？建议是否差异化？ | 缺少必需章节 → 严重扣分 |
| **数据质量** | 15% | 15 | 数据是否准确、逻辑自洽？ | **数据质量 < 5 分 → R5 退回重写** |

**R 级制**：
- R1 (90-100)：材料达标，可直接上会
- R2 (80-89)：基本达标，微调后上会
- R3 (70-79)：明显不足，修改后需重新评审
- R4 (60-69)：较多问题，大幅修改后重新评审
- R5 (0-59)：严重不达标，退回重写

### 4.3 负责人年度述职（4维 / 100分）

| 维度 | 权重 | 满分 | 核心关注 |
|------|------|------|---------|
| **完整性** | 25% | 25 | 年度目标、关键战役、团队建设、能力提升、下年规划 |
| **SP战略关联度** | 25% | 25 | 个人/团队目标与公司SP对齐 |
| **态度与反思** | 25% | 25 | 坦诚反思、领导力成长 |
| **下一步计划** | 25% | 25 | 下年度规划具体可执行 |

### 4.4 通用议题材料审核（4维 / 100分）

| 维度 | 权重 | 满分 | 核心拷问 | 一票否决 |
|------|------|------|---------|---------|
| **目标-解决方案对齐度** | 30% | 30 | 为什么要上会？方案对不对症？ | **零决策请求 → 总分封顶60分** |
| **决策支撑度** | 30% | 30 | 数据够不够硬？选项够不够全？ | **零数据支撑 → 总分封顶60分** |
| **行动具体化（SMART）** | 25% | 25 | 谁来干？怎么衡量？什么时候干完？ | **零责任主体/零时间计划 → 总分封顶60分** |
| **材料规范度** | 15% | 15 | 结构顺不顺？篇幅是否可控？ | 空话套话占比过高 → 大幅扣分 |

## 五、AI 审核报告规范

### 5.1 报告必备模块

```
[AI] XX 材料审核报告
材料：XXX
场景：XXX
总分：XX/100  判定：[通过/待修改/退回重写]

一、评分总览（维度表格）
二、详细审核（每个维度：得分+打分理由+问题清单）
三、红线检查 / 一票否决检查
四、审核结论（总分+定性+最大硬伤+上会风险+追问预判）
五、修改清单（可直接发给当事人的待办列表）
```

### 5.2 高质量写作标准

| 原则 | 要求 | 差示例 | 好示例 |
|------|------|--------|--------|
| **分数校准** | 必须说明"不是满分因为…不是0分因为…所以取…" | "材料内容基本齐全，得分25分" | "6大模块中3个到位、2个半吊子、1个缺位。取中间偏下：21分" |
| **用"零"强调缺失** | 完全缺失时直接用"零" | "缺少主观归因分析" | "零主观归因（主观关键词0个）" |
| **定性判断** | 敢于给材料定性 | "根因分析还有提升空间" | "这不是根因分析，这是找借口" |
| **材料引用具体** | 每个判断必须有数字/原文支撑 | "差距拆解不够详细" | "4661万缺口未按客户/产品/区域/时间任何维度拆解" |
| **问题含后果** | 说明"如果不改会怎样" | "4661万缺口未按维度拆解" | "4661万缺口未拆解，意味着负责人并没有真正想清楚问题出在哪里。上会参谋部一定会追问" |
| **建议有可交付物** | 建议必须包含"做什么动作+做到什么程度" | "建议加强根因分析深度" | "补主观原因分析至少占40%，使用至少1种分析工具（建议5Why）" |
| **结论5要素** | 总分/定性/最大硬伤/上会风险/追问预判 | "总分67分，待修改" | "总分67分，根因分析10分踩了单项底线。最大硬伤：零工具、零主观归因。上会风险：高。参谋部最可能追问：①'4661万按维度拆一下？'②'你自己做错了什么？'③'1199万硬缺口从哪来？'" |

### 5.3 问题分级标准

| 级别 | 含义 | 适用场景 | 处理方式 |
|------|------|---------|---------|
| 🔴 致命 / P0 | 必须修改 | 缺少必需模块、零业绩承诺、行动无责任人、数据重大错误 | 不上会，打回重写 |
| 🟡 警告 / P1 | 建议修改 | 分析不深入、计划不具体、格式不规范 | 修改后上会 |
| 🔵 建议 / P2 | 可选优化 | 排版建议、表述优化 | 视情况处理 |

## 六、页面与交互设计

### 6.1 页面结构（`reviewer.html`）

```
┌─────────────────────────────────────────────────────┐
│ 顶部导航栏（与 cockpit 一致）                        │
├──────────┬──────────────────────────────────────────┤
│ 侧边栏    │  内容区                                   │
│          │  ┌─────────────────────────────────────┐  │
│ 战略执行  │  │ 输入区：场景选择 + KMS链接 + 审核按钮 │  │
│ 重点工作  │  └─────────────────────────────────────┘  │
│ 业务专题  │  ┌─────────────────────────────────────┐  │
│ 经营分析会 │  │ 评审原则卡片（按场景动态渲染）        │  │
│ 会议材料审核│  └─────────────────────────────────────┘  │
│          │  ┌─────────────────────────────────────┐  │
│          │  │ 审核报告区（默认隐藏，审核后显示）    │  │
│          │  │ · 总分卡片                           │  │
│          │  │ · 分项评分表格                       │  │
│          │  │ · 问题清单                           │  │
│          │  │ · 改进建议                           │  │
│          │  │ · 审核结论                           │  │
│          │  │ · 原始报告（可折叠）                 │  │
│          │  └─────────────────────────────────────┘  │
│          │  ┌─────────────────────────────────────┐  │
│          │  │ 批量审核 + 历史记录 + 配置面板        │  │
│          │  └─────────────────────────────────────┘  │
└──────────┴──────────────────────────────────────────┘
```

### 6.2 核心交互流程

| 步骤 | 用户操作 | 系统反馈 |
|------|---------|---------|
| 1 | 选择审核场景 | 左侧评审原则卡片动态切换 |
| 2 | 输入 KMS 链接 | 校验 URL 格式（http/https） |
| 3 | 点击「AI 直接审核」 | 按钮进入 loading 状态，状态栏显示"正在获取内容并调用 AI 审核" |
| 4 | AI 返回报告 | 渲染评分总览、分项评分、问题清单、审核结论 |
| 5 | 用户查看/复制/导出 | 支持复制报告、导出 PDF、查看原始报告 |
| 6 | 切换场景再审核 | 若已有一份报告，提示"场景已切换，请重新执行审核" |

### 6.3 批量审核流程

```
点击「批量审核」→ 展开输入区
    │
    ▼
粘贴多个 KMS 链接（每行一个）
    │
    ▼
点击「开始批量审核」→ 后台依次审核
    │
    ▼
显示进度条 + 每条结果状态
    │
    ▼
完成后显示「生成对比矩阵」按钮
    │
    ▼
点击后弹出模态框，展示多材料对比表格
```

## 七、数据流与接口

### 7.1 单材料审核数据流

```
reviewer.html
    │
    ├── 用户输入：KMS URL + scene_id
    │
    ▼
POST /api/review
    │
    ├── proxy_server.py 接收请求
    ├── 抓取 KMS 页面内容
    ├── 拼接场景 prompt（含审核标准）
    ├── 调用 Kimi API
    │
    ▼
返回：{ success, report, dimension_scores, total_score, scene_id }
    │
    ▼
前端 parseDimensionScores() 提取分数
    │
    ▼
POST /api/history 保存审核记录
    │
    ▼
同时写入 localStorage: dste_review_scores[url] = { maxScore, lastReviewAt }
```

### 7.2 核心接口

| 接口 | 方法 | 用途 |
|------|------|------|
| `GET /api/scenes` | GET | 获取审核场景列表 |
| `POST /api/review` | POST | 提交材料进行 AI 审核 |
| `GET /api/history` | GET | 查询审核历史（支持按 URL 过滤） |
| `POST /api/history` | POST | 保存审核记录 |
| `GET /api/batch/:taskId/compare` | GET | 获取批量审核对比矩阵 |

### 7.3 跨页面数据共享

| 共享方式 | Key | 用途 | 更新时机 |
|---------|-----|------|---------|
| localStorage | `dste_review_scores` | 存储 URL → 最高分的映射 | reviewer 每次审核后自动更新 |
| localStorage | `dste_theme` | 主题同步 | 切换主题时 |

## 附录：审核要点速查表

### 通用检查清单（适用于所有场景）

1. 材料是否包含清晰的问题/目标定义？
2. 关键数据是否量化、可追溯？
3. 根因分析是否深入本质（不止外部环境）？
4. 改进措施是否具体、可执行、有责任人、有时间节点？
5. 建议/计划是否与问题根因一一对应？
6. 材料是否与 SP 战略方向对齐？
7. 数据计算是否正确、逻辑是否自洽？
8. 材料结构是否清晰、表达是否精简？
