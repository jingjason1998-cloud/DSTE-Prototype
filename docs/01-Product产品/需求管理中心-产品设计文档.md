## 一、产品概述

### 1.1 定位与价值主张

**需求管理中心**是 DSTE 战略管理平台自身的「产品需求池」。它面向 DSTE 产品团队、技术团队及内部用户，统一收集与管理 DSTE 平台的功能需求、体验优化、缺陷修复、技术债和竞品对标，最终将需求落到 RoadMap、版本发布和开发计划中。

**核心价值：**
- **统一产品需求入口**: 避免需求散落在微信、邮件、会议、Excel 中
- **需求可追溯**: 从需求提出到版本发布，全流程留痕
- **RoadMap 与版本对齐**: 需求直接关联到开发路线图和版本规划
- **闭环验证**: 发布后验证效果，形成完整产品迭代闭环
- **数据驱动迭代**: 按来源、类型、模块、版本统计需求分布

## 二、用户角色与故事

### 2.1 角色定义

| 角色 | 职责 | 核心诉求 |
|------|------|---------|
| **需求提出者** | 提交产品需求或反馈 | 提交简单、状态可追踪 |
| **需求分析者** | 评估需求价值与可行性 | 信息完整、决策有据 |
| **需求规划者** | 将需求排入版本和 RoadMap | 关联清晰、进度可控 |
| **需求实现者** | 开发实现需求 | 验收标准明确 |
| **需求验证者** | 验证需求上线效果 | 可追溯、可验收 |

### 2.2 用户故事

**需求提出者**
```
作为 DSTE 内部用户
我希望 在系统中提交对 DSTE 平台的优化建议
以便 产品团队能看到并评估我的诉求

作为 产品经理
我希望 把客户高频反馈录入需求池
以便 规划版本时参考真实用户声音
```

**需求分析者**
```
作为 产品经理
我希望 按优先级和来源筛选待分析需求
以便 高效组织需求评审

作为 技术负责人
我希望 看到需求的价值说明、验收标准和影响模块
以便 快速评估技术成本
```

**需求规划者**
```
作为 产品经理
我希望 将已分析需求关联到目标版本和 RoadMap 阶段
以便 形成从需求到发布的闭环

作为 项目经理
我希望 看到各版本的需求清单和进度
以便 跟踪开发节奏
```

### 模块2: 需求分析

> **定位**: 对需求池中的需求进行价值、可行性、成本的评估，输出分析结论。

#### 2.1 状态推进

```
已收集 → 分析中 → 已规划
            ↓
      已拒绝 / 已挂起
```

#### 2.2 价值/成本/紧急评分

分析阶段填写：
- 业务价值评分 1-5
- 技术成本评分 1-5
- 紧急程度评分 1-5

系统自动计算综合得分，辅助优先级判断。

#### 2.3 评审意见记录

每次状态变更必须记录：
- 评审人
- 评审时间
- 评审意见
- 状态变更（从 → 到）

#### 2.4 伪需求/拒绝/挂起

- 产品经理可标记为「已拒绝」，记录原因
- 可标记为「已挂起」，说明复评条件
- 拒绝/挂起的需求进入回收站，可重新激活为「已收集」

---

### 模块3: 版本规划

> **定位**: 将已分析的需求排入目标版本和 RoadMap 阶段。

#### 3.1 目标版本关联

- 目标版本下拉从 RoadMap 数据读取
- 支持多选/单选（Phase 1 单选）

#### 3.2 RoadMap 阶段关联

- 关联到 RoadMap 的具体阶段或条目
- 在 RoadMap 页面展示该阶段需求数量

#### 3.3 版本视图

- 按目标版本分组展示需求
- 统计每个版本的需求数量、类型分布、完成进度

---

### 模块4: 实现跟踪

> **定位**: 跟踪需求从规划到发布的实现过程。

#### 4.1 开发任务链接

- 支持填写开发任务链接（如内部工单系统）
- 可记录开发负责人

#### 4.2 状态同步

状态流转：
```
已规划 → 开发中 → 待发布 → 已发布
```

#### 4.3 发布说明

- 需求发布后填写发布说明
- 作为版本发布说明的素材

---

### 模块5: 需求验证

> **定位**: 验证需求上线后的实际效果，完成闭环。

#### 5.1 验收标准

- 新建需求时填写验收标准
- 验证阶段对照验收标准确认

#### 5.2 验证结果

- 验证人、验证时间、验证结论
- 验证结论：通过 / 部分通过 / 不通过

#### 5.3 闭环归档

验证通过后状态变为「已关闭」，记录关闭时间。

---

### 模块6: 需求分析

> **定位**: 为产品团队提供需求数据的统计与分析。

#### 6.1 来源分布

按需求来源统计数量与占比。

#### 6.2 类型分布

按需求类型统计数量与占比。

#### 6.3 模块分布

按影响模块统计数量，识别各模块迭代压力。

#### 6.4 版本分布

按目标版本统计需求数量与完成率。

## 四、核心页面流程

### 4.1 需求提交流程

```
用户进入需求管理中心
  │
  ▼
点击「+ 新建需求」
  │
  ▼
填写需求表单（标题/类型/来源/优先级/问题/价值/描述/验收标准/影响模块）
  │
  ▼
保存 → 状态变为「已收集」
  │
  ▼
通知产品经理
```

### 4.2 需求分析流程

```
产品经理进入需求池
  │
  ▼
筛选「已收集」需求
  │
  ▼
查看需求详情
  │
  ▼
点击「开始分析」，状态变为「分析中」
  │
  ▼
填写价值/成本/紧急评分、分析意见
  │
  ▼
点击「完成分析」，状态变为「已规划」或「已拒绝」/「已挂起」
```

### 4.3 版本规划流程

```
已分析需求
  │
  ▼
产品经理在需求表单中选择目标版本（从 RoadMap 读取）
  │
  ▼
系统自动填充 RoadMap 阶段
  │
  ▼
状态变为「已规划」
  │
  ▼
进入「版本规划」视图，需求出现在对应版本卡片中
  │
  ▼
开发团队领取后开始开发
```

**版本规划视图操作流程：**

```
用户进入「版本规划」Tab
  │
  ▼
查看按版本分组的需求卡片
  │
  ▼
点击「分配需求」或拖拽未规划需求到版本
  │
  ▼
需求 targetVersion 更新，卡片数量实时变化
  │
  ▼
点击版本卡片「查看全部」打开版本详情弹窗
  │
  ▼
查看该版本全部需求、完成进度、批量操作
```

### 4.4 实现与验证流程

```
已规划需求
  │
  ▼
开发中 → 待发布 → 已发布
  │
  ▼
验证人按验收标准验证
  │
  ▼
验证通过 → 已关闭
```

### 5.2 关键数据表设计

#### 需求表 (requirement)

```sql
CREATE TABLE requirement (
    req_id          BIGINT PRIMARY KEY AUTO_INCREMENT COMMENT '需求ID',
    req_code        VARCHAR(32) NOT NULL UNIQUE COMMENT '需求编号，如 REQ-2026-001',
    title           VARCHAR(256) NOT NULL COMMENT '需求标题',
    description     TEXT COMMENT '详细描述',
    type            VARCHAR(32) NOT NULL COMMENT '类型：FEATURE/OPTIMIZATION/BUG/TECH_DEBT/COMPETITOR/SECURITY',
    source          VARCHAR(32) NOT NULL COMMENT '来源：USER_FEEDBACK/INTERNAL_USER/PRODUCT_PLAN/TECH_TEAM/COMPETITOR/VERSION_AUDIT/DATA_DRIVEN',
    priority        VARCHAR(8) NOT NULL COMMENT '优先级：P0/P1/P2/P3',
    status          VARCHAR(32) DEFAULT 'COLLECTED' COMMENT '状态：COLLECTED/ANALYZING/PLANNED/DEVELOPING/PENDING_RELEASE/RELEASED/VERIFIED/CLOSED/REJECTED/SUSPENDED',
    affected_modules JSON COMMENT '影响模块列表',
    target_version  VARCHAR(32) COMMENT '目标版本',
    roadmap_phase   VARCHAR(64) COMMENT 'RoadMap 阶段',
    product_owner   VARCHAR(64) COMMENT '产品经理',
    tech_owner      VARCHAR(64) COMMENT '技术负责人',
    reporter        VARCHAR(64) NOT NULL COMMENT '提出人',
    reporter_dept   VARCHAR(64) COMMENT '提出部门',
    problem         TEXT COMMENT '问题/痛点',
    value           TEXT COMMENT '价值说明',
    acceptance_criteria TEXT COMMENT '验收标准',
    business_value  TINYINT COMMENT '业务价值 1-5',
    technical_effort TINYINT COMMENT '技术成本 1-5',
    urgency         TINYINT COMMENT '紧急程度 1-5',
    dev_task_link   VARCHAR(512) COMMENT '开发任务链接',
    release_note    TEXT COMMENT '发布说明',
    verified_by     VARCHAR(64) COMMENT '验证人',
    verified_at     DATETIME COMMENT '验证时间',
    attachments     JSON COMMENT '附件列表',
    created_by      VARCHAR(64) NOT NULL,
    created_at      DATETIME DEFAULT CURRENT_TIMESTAMP,
    updated_at      DATETIME ON UPDATE CURRENT_TIMESTAMP,
    closed_at       DATETIME COMMENT '关闭时间',

    INDEX idx_status (status),
    INDEX idx_priority (priority),
    INDEX idx_type (type),
    INDEX idx_source (source),
    INDEX idx_target_version (target_version),
    INDEX idx_created_at (created_at)
) COMMENT='DSTE 产品需求表';
```

#### 需求评审记录表 (requirement_review)

```sql
CREATE TABLE requirement_review (
    review_id       BIGINT PRIMARY KEY AUTO_INCREMENT COMMENT '评审ID',
    req_id          BIGINT NOT NULL COMMENT '需求ID',
    reviewer        VARCHAR(64) NOT NULL COMMENT '评审人',
    action          VARCHAR(32) NOT NULL COMMENT '操作：SUBMIT/ANALYZE/PLAN/REJECT/SUSPEND/DEVELOP/RELEASE/VERIFY/CLOSE',
    comment         TEXT COMMENT '评审意见',
    created_at      DATETIME DEFAULT CURRENT_TIMESTAMP,

    INDEX idx_req (req_id),
    INDEX idx_action (action),
    INDEX idx_time (created_at)
) COMMENT='需求评审记录表';
```

## 六、非功能需求

### 6.1 性能要求

| 指标 | 要求 | 说明 |
|------|------|------|
| 需求列表查询 | < 300ms | 支持分页、筛选、排序 |
| 新建需求响应 | < 500ms | 含附件上传除外 |
| 看板统计 | < 500ms | 实时聚合 |
| 导出 | < 3s | 1000 条以内 |

### 6.2 安全要求

- 所有用户输入使用 `escapeHtml` 转义，防止 XSS
- 附件上传限制类型和大小
- 删除操作需二次确认

### 6.3 可用性要求

- 表单支持自动保存草稿
- 状态推进需记录评审意见
- 列表支持批量操作（后续扩展）

### 模块7: 版本规划与 RoadMap 关联

> **定位**: 把需求落到具体版本和 RoadMap 阶段，形成「需求 → 版本 → 发布」的规划视图。

#### 7.1 版本规划视图

按版本分组展示需求，一眼看到每个版本的需求数量、进度和清单。

**版本卡片信息：**

| 元素 | 说明 |
|------|------|
| 版本号 | 如 v0.6.0，来自 RoadMap 数据 |
| 计划/发布日期 | 从 RoadMap 读取 |
| 版本状态 | 已发布 / 开发中 / 已规划 |
| 需求列表 | 该版本下前 N 条需求，可展开查看全部 |
| 完成进度 | 已关闭数 / 总数，带进度条 |
| 操作 | 查看全部、分配需求 |

**视图布局：**

```
版本规划
[全部] [已发布] [开发中] [已规划] [未规划]

┌──────────────┐  ┌──────────────┐  ┌──────────────┐
│  v0.6.0      │  │  v0.5.6      │  │  未规划       │
│  📅 2026-07  │  │  📅 2026-06  │  │              │
│  状态: 已规划 │  │  状态: 开发中 │  │              │
│  ─────────── │  │  ─────────── │  │  ─────────── │
│  REQ-001     │  │  REQ-003     │  │  REQ-005     │
│  REQ-002     │  │  REQ-004     │  │              │
│  ─────────── │  │  ─────────── │  │  ─────────── │
│  进度 1/2    │  │  进度 0/2    │  │  共 1 条     │
└──────────────┘  └──────────────┘  └──────────────┘
```

#### 7.2 版本筛选

| 筛选 | 说明 |
|------|------|
| 全部 | 所有版本 + 未规划 |
| 已发布 | RoadMap 中 status === released 的版本 |
| 开发中 | 当前有需求处于 DEVELOPING / PENDING_RELEASE 状态 |
| 已规划 | 已排期但尚未进入开发 |
| 未规划 | targetVersion 为空的需求 |

#### 7.3 版本详情弹窗

点击「查看全部」打开：
- 版本基本信息（版本号、日期、状态、CHANGELOG 摘要）
- 该版本下全部需求列表（按状态分组）
- 完成进度条
- 批量操作：批量推进状态、批量移除版本

#### 7.4 RoadMap 关联

**需求表单中的目标版本：**
- 下拉选项从 `src/data/roadmap-data.json` 动态读取
- 选项包含版本号和版本状态，如 `v0.6.0（已规划）`
- 选择版本后自动填充「RoadMap 阶段」
- 若需求状态为「已收集」/「分析中」，建议推进为「已规划」

**版本规划视图 ↔ RoadMap：**
- 版本卡片提供「在 RoadMap 中查看」链接
- 跳转到 `cockpit.html#dashboard/roadmap`
- RoadMap 页面未来可展示每个版本关联的需求数量
