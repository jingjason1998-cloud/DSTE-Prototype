## 一、产品概述

### 1.1 定位与价值主张

**需求管理中心**是 DSTE 战略管理平台的**统一需求入口**，负责承接来自市场、客户、高管、部门员工等多元角色的战略需求，通过标准化评审、分级、跟踪流程，将零散需求转化为 SP/BP 的可执行输入。

**核心价值：**
- **需求统一归口**: 避免需求散落在邮件、会议、Excel 中，形成可追溯的需求资产
- **战略输入结构化**: 将原始需求与战略地图、年度经营计划、重点工作建立关联
- **评审流程透明化**: 明确需求的评审状态、责任人、决策依据，支持审计
- **闭环跟踪**: 从需求提出到战略落地，全程跟踪需求是否被采纳、如何落地
- **决策数据支撑**: 按来源、类型、优先级、处理效率分析需求分布，辅助资源分配

## 二、用户角色与故事

### 2.1 角色定义

| 角色 | 职责 | 核心诉求 |
|------|------|---------|
| **需求提出者** | 提交战略需求 | 提交简单、状态可追踪 |
| **需求评审者** | 评估需求价值与可行性 | 信息完整、决策有据 |
| **需求处理者** | 将需求转化为战略/计划输入 | 关联清晰、状态同步 |
| **需求管理员** | 维护需求池、配置规则 | 流程可控、数据完整 |
| **需求分析者** | 分析需求趋势与效率 | 维度丰富、导出方便 |

### 2.2 用户故事

**需求提出者**
```
作为 部门负责人
我希望 在系统中提交本部门年度战略需求
以便 高层在制定战略时能看到我们的诉求

作为 产品经理
我希望 将客户高频反馈作为需求录入系统
以便 战略制定时参考真实市场声音
```

**需求评审者**
```
作为 战略运营
我希望 按优先级和来源筛选待评审需求
以便 高效组织评审会议

作为 高管
我希望 看到需求的价值评估、预算影响、风险说明
以便 快速做出批准或拒绝决策
```

**需求处理者**
```
作为 战略运营
我希望 将已批准需求关联到战略专题或年度经营计划
以便 形成从需求到执行的闭环

作为 BP 负责人
我希望 看到已批准需求在年度经营计划中的落地情况
以便 检查需求是否被真正执行
```

**需求管理员**
```
作为 系统管理员
我希望 配置需求类型、优先级、评审流程
以便 适应不同组织的管理规范

作为 需求管理员
我希望 查看所有需求的处理时效
以便 发现流程瓶颈
```

### 模块1: 需求池

> **定位**: 需求管理中心的入口页面，展示所有需求的汇总列表，支持完整的 CRUD 与查询能力。

#### 1.1 需求列表

| 字段 | 说明 |
|------|------|
| 需求编号 | 系统自动生成，如 REQ-2026-001 |
| 标题 | 需求一句话描述 |
| 提出人 | 需求提交者 |
| 来源 | 市场/客户/高管/部门/其他 |
| 类型 | 战略/产品/运营/组织/其他 |
| 优先级 | P0/P1/P2/P3 |
| 状态 | 待评审/评审中/已批准/已拒绝/已挂起/已落地 |
| 创建时间 | 提交时间 |
| 期望解决时间 | 提出者期望的解决时间 |

#### 1.2 需求筛选与搜索

| 筛选维度 | 选项 |
|---------|------|
| 状态 | 待评审 / 评审中 / 已批准 / 已拒绝 / 已挂起 / 已落地 |
| 优先级 | P0 / P1 / P2 / P3 |
| 来源 | 市场 / 客户 / 高管 / 部门 / 其他 |
| 类型 | 战略 / 产品 / 运营 / 组织 / 其他 |
| 提出时间 | 最近 7 天 / 30 天 / 90 天 / 自定义 |
| 关键词搜索 | 标题、描述、提出人 |

#### 1.3 需求排序与分页

- 默认排序：创建时间倒序
- 支持排序字段：创建时间、优先级、期望解决时间
- 分页：默认 20 条/页，支持 10/50/100 条切换

#### 1.4 新建需求

| 字段 | 必填 | 说明 |
|------|:----:|------|
| 标题 | ✅ | 一句话描述需求 |
| 详细描述 | ✅ | 背景、问题、期望价值 |
| 来源 | ✅ | 市场/客户/高管/部门/其他 |

| 优先级 | ✅ | 提出者自评 P0-P3 |
| 提出人 | ✅ | 默认当前用户 |
| 责任部门 | ⚠️ | 需求归属部门 |
| 期望解决时间 | ⚠️ | 期望完成时间 |

#### 1.5 需求详情

详情弹窗/页面展示：
- 基础信息（编号、标题、描述、来源、类型、优先级、状态）
- 提出信息（提出人、部门、创建时间、期望解决时间）
- 评审记录（评审人、意见、时间）
- 关联信息（关联的战略专题、年度计划、重点工作）
- 操作按钮：编辑、提交评审、关联、关闭

### 模块2: 需求评审

> **定位**: 对需求池中的需求进行价值、可行性、资源影响的评估，输出评审决策。

#### 2.1 评审状态流转

```
┌─────────┐    ┌─────────┐    ┌─────────┐    ┌─────────┐
│  待评审  │───►│ 评审中  │───►│ 已批准  │───►│  已落地  │
└────┬────┘    └────┬────┘    └────┬────┘    └─────────┘
     │              │              │
     │              │              │
     └──────────────┴──────────────┘
                    │
                    ▼
              ┌─────────┐
              │ 已拒绝  │
              └────┬────┘
                   │
                   └──────────────► 可重新激活为「待评审」
```

| 状态 | 说明 |
|------|------|
| 待评审 | 刚提交，等待评审 |
| 评审中 | 已分配给评审人，正在评估 |
| 已批准 | 评审通过，可进入 SP/BP/重点工作 |
| 已拒绝 | 评审不通过，记录原因 |
| 已挂起 | 暂不具备条件，后续再评估 |
| 已落地 | 已关联到具体执行项并推进中 |

#### 2.2 评审意见记录

每次状态变更必须记录：
- 评审人
- 评审时间
- 评审意见
- 状态变更（从 → 到）
- 预算影响评估（可选）
- 风险提示（可选）

#### 2.3 批量评审

支持在列表页选择多条需求进行批量操作：
- 批量提交评审
- 批量批准/拒绝/挂起
- 批量分配评审人

#### 2.4 评审日历/会议关联

- 支持将需求加入经营分析会或专项评审会议议程
- 与 `meetings.html` 的经营分析会模块打通（数据关联，后续实现）

### 模块3: 需求跟踪

> **定位**: 确保被批准的需求能落地到 SP/BP/Execute 的具体执行项中，形成端到端闭环。

#### 3.1 关联战略专题

已批准的战略类需求可关联到 `sp/insights-topics` 中的战略专题。

#### 3.2 关联年度经营计划

已批准的年度类需求可关联到 `bp/annual-plan` 中的 KPI 或重点工作。

#### 3.3 关联重点工作

已批准需求可直接生成或关联到 `exe/tasks` 中的重点工作。

#### 3.4 落地状态跟踪

- 自动同步关联执行项的状态
- 当关联的专题/计划/工作完成时，需求状态自动变为「已落地」
- 长期未落地的需求触发预警（依赖预警中心）

---

### 模块4: 需求分析

> **定位**: 为管理层和战略运营提供需求数据的统计、分析与报告能力。

#### 4.1 来源分布分析

按需求来源统计数量与占比：市场、客户、高管、部门、其他。

#### 4.2 优先级分布分析

按 P0/P1/P2/P3 统计数量，识别高优先级需求处理瓶颈。

#### 4.3 处理效率分析

- 平均评审周期
- 各状态停留时长
- 各部门需求处理效率
- 评审通过率/拒绝率

#### 4.4 需求报告导出

支持导出 PDF/Excel 报告，包含：
- 需求总览
- 来源分布
- 优先级分布
- 处理效率
- 落地情况

---

### 模块5: 系统配置

> **定位**: 适配不同组织的管理规范，配置需求管理的元数据与流程。

#### 5.1 需求类型配置

默认类型：战略、产品、运营、组织、其他。支持自定义扩展。

#### 5.2 优先级配置

默认 P0/P1/P2/P3，支持配置颜色、说明、默认处理时限。

#### 5.3 评审流程配置

- 配置不同优先级需求的评审人
- 配置是否需要多级审批
- 配置自动升级规则（如 P0 需求超过 3 天未评审自动升级）

#### 5.4 权限配置

| 权限 | 说明 |
|------|------|
| 需求查看 | 查看需求列表与详情 |
| 需求提交 | 新建需求 |
| 需求编辑 | 编辑自己或全部需求 |
| 需求评审 | 执行评审操作 |
| 需求关联 | 将需求关联到 SP/BP/Execute |
| 配置管理 | 维护需求类型、优先级、流程 |

## 四、核心页面流程

### 4.1 需求提交流程

```
用户进入需求管理中心
  │
  ▼
点击「+ 新建需求」
  │
  ▼
填写需求表单（标题/描述/来源/类型/优先级/期望解决时间）
  │
  ▼
保存 → 状态变为「待评审」
  │
  ▼
通知需求管理员和默认评审人
```

### 4.2 需求评审流程

```
评审人进入需求池
  │
  ▼
筛选「待评审」/「评审中」需求
  │
  ▼
查看需求详情
  │
  ▼
填写评审意见
  │
  ├─ 批准 ──► 状态变为「已批准」，可关联 SP/BP/Execute
  │
  ├─ 拒绝 ──► 状态变为「已拒绝」，记录原因
  │
  └─ 挂起 ──► 状态变为「已挂起」，说明复评条件
```

### 4.3 需求落地流程

```
已批准需求
  │
  ▼
战略运营将需求关联到：
  ├─ 战略专题（SP）
  ├─ 年度经营计划 KPI/重点工作（BP）
  └─ 重点工作（Execute）
  │
  ▼
同步关联执行项的进度
  │
  ▼
关联项完成 ──► 需求状态自动变为「已落地」
```

### 5.2 关键数据表设计

#### 需求表 (requirement)

```sql
CREATE TABLE requirement (
    req_id          BIGINT PRIMARY KEY AUTO_INCREMENT COMMENT '需求ID',
    req_code        VARCHAR(32) NOT NULL UNIQUE COMMENT '需求编号，如 REQ-2026-001',
    title           VARCHAR(256) NOT NULL COMMENT '需求标题',
    description     TEXT NOT NULL COMMENT '详细描述',
    source          VARCHAR(32) NOT NULL COMMENT '来源：MARKET/CUSTOMER/EXECUTIVE/DEPT/OTHER',
    type            VARCHAR(32) NOT NULL COMMENT '类型：STRATEGY/PRODUCT/OPERATION/ORG/OTHER',
    priority        VARCHAR(8) NOT NULL COMMENT '优先级：P0/P1/P2/P3',
    status          VARCHAR(32) DEFAULT 'PENDING' COMMENT '状态：PENDING/REVIEWING/APPROVED/REJECTED/SUSPENDED/LANDED',
    proposer        VARCHAR(64) NOT NULL COMMENT '提出人',
    dept_id         BIGINT COMMENT '责任部门ID',
    expect_date     DATE COMMENT '期望解决时间',
    attachments     JSON COMMENT '附件列表',
    reviewer        VARCHAR(64) COMMENT '当前评审人',
    landed_at       DATETIME COMMENT '落地时间',
    created_by      VARCHAR(64) NOT NULL,
    created_at      DATETIME DEFAULT CURRENT_TIMESTAMP,
    updated_at      DATETIME ON UPDATE CURRENT_TIMESTAMP,
    
    INDEX idx_status (status),
    INDEX idx_priority (priority),
    INDEX idx_source (source),
    INDEX idx_type (type),
    INDEX idx_dept (dept_id),
    INDEX idx_time (created_at)
) COMMENT='需求表';
```

#### 需求评审记录表 (requirement_review)

```sql
CREATE TABLE requirement_review (
    review_id       BIGINT PRIMARY KEY AUTO_INCREMENT COMMENT '评审ID',
    req_id          BIGINT NOT NULL COMMENT '需求ID',
    reviewer        VARCHAR(64) NOT NULL COMMENT '评审人',
    action          VARCHAR(32) NOT NULL COMMENT '操作：SUBMIT/APPROVE/REJECT/SUSPEND/REACTIVATE',
    comment         TEXT COMMENT '评审意见',
    budget_impact   VARCHAR(256) COMMENT '预算影响评估',
    risk_note       VARCHAR(512) COMMENT '风险提示',
    created_at      DATETIME DEFAULT CURRENT_TIMESTAMP,
    
    INDEX idx_req (req_id),
    INDEX idx_action (action),
    INDEX idx_time (created_at)
) COMMENT='需求评审记录表';
```

#### 需求关联表 (requirement_link)

```sql
CREATE TABLE requirement_link (
    link_id         BIGINT PRIMARY KEY AUTO_INCREMENT COMMENT '关联ID',
    req_id          BIGINT NOT NULL COMMENT '需求ID',
    target_type     VARCHAR(32) NOT NULL COMMENT '目标类型：STRATEGY_TOPIC/ANNUAL_PLAN/KPI/KEY_TASK',
    target_id       VARCHAR(64) NOT NULL COMMENT '目标ID',
    target_title    VARCHAR(256) COMMENT '目标标题（冗余）',
    created_by      VARCHAR(64) NOT NULL,
    created_at      DATETIME DEFAULT CURRENT_TIMESTAMP,
    
    INDEX idx_req (req_id),
    INDEX idx_target (target_type, target_id),
    UNIQUE KEY uk_req_target (req_id, target_type, target_id)
```

## 六、非功能需求

### 6.1 性能要求

| 指标 | 要求 | 说明 |
|------|------|------|
| 需求列表查询 | < 300ms | 支持分页、筛选、排序 |
| 新建需求响应 | < 500ms | 含附件上传除外 |
| 报告导出 | < 5s | 5000 条以内 |
| 并发提交 | > 50并发 | 季度规划高峰期 |

### 6.2 安全要求

- 需求数据按组织隔离
- 敏感需求（如高管提出的战略级需求）可设置查看权限
- 所有评审操作记录审计日志
- 附件上传限制类型和大小，防止恶意文件

### 6.3 可用性要求

- 需求表单支持自动保存草稿
- 评审提醒通过通知中心/企业微信发送
- 支持批量操作，提升运营效率

### 6.4 扩展性要求

- 支持自定义需求字段（后续版本）
- 支持与企业微信/钉钉集成，接收需求提交
- 支持从 CRM/工单系统自动导入需求
