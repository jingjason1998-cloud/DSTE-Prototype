# 第一部分：功能设计

## 一、设计思维与方法论

### 1.1 核心原则

基于 software-module-design 的三大原则：

| 原则 | 含义 | 在本产品中的体现 |
|------|------|----------------|
| **单一职责** | 一个模块只做一类事 | Dashboard只读展示、KMS同步只读写、生成器只生成文本 |
| **高内聚** | 模块内部功能紧密相关 | 决策记录模块：记录+编辑+类型推荐+标签，全围绕「决策」 |
| **低耦合** | 模块之间通过接口通信 | 各模块通过 Meeting ID / Todo ID 关联，不直接操作对方数据 |

### 1.2 四步设计法

```
Step 1: 需求收集    → 用"用户故事"描述每个功能
Step 2: 功能归类    → 把相似功能放在一起，命名模块
Step 3: 模块定义    → 明确每个模块的职责、数据、功能、关系
Step 4: 关系梳理    → 画出模块之间的调用依赖图
```

## 二、需求收集（用户故事）

```
作为 会议组织者
我希望 查看所有历史会议的时间线
以便 快速定位某次会议的议题和决策

作为 会议组织者
我希望 安排下次会议的议题、主讲人和时长
以便 提前通知参会人准备

作为 会议组织者
我希望 在会中记录决策和TODO
以便 会后可以追踪执行

作为 决策执行人
我希望 查看分配给自己的TODO清单
以便 按时完成任务

作为 决策执行人
我希望 标记TODO为已完成
以便 会议组织者知道进度

作为 参会负责人
我希望 追溯3个月前的某个决策依据
以便 向上级解释当时的判断

作为 会议组织者
我希望 自动生成标准化的会议记录
以便 一键写入KMS存档

作为 会议组织者
我希望 从KMS已有的会议记录中导入数据
以便 历史数据不丢失
```

| 模块名 | 包含功能 | 职责概括 |
|--------|---------|---------|

## 四、模块详细定义

### 模块1：会议管理模块

**1. 模块职责**
- **做什么：** 管理会议的生命周期（创建、编辑、删除、查看），维护会议的基本信息和议题列表
- **不做什么：** 不追踪TODO进度（TODO模块负责）、不记录决策详情（决策模块负责）、不写KMS（KMS同步模块负责）

**2. 涉及的数据（实体）**

| 数据名称 | 说明 | 示例 |
|---------|------|------|
| Meeting | 一次会议记录 | id: "20260518", date: "2026-05-18", title: "本部重点工作经营例会", status: "completed" |
| AgendaItem | 会议议题 | title: "上期TODO复盘", duration: 10, speaker: "marks" |

**3. 功能列表**

| 功能名称 | 用户故事 | 输入 | 输出 |
|---------|---------|------|------|
| 创建会议 | 组织者希望创建下次会议 | 日期、标题、议题列表 | 新Meeting对象 |
| 编辑会议 | 组织者希望修改会议信息 | Meeting ID、新字段值 | 更新后的Meeting |
| 删除会议 | 组织者希望删除错误创建的会议 | Meeting ID | 删除成功/失败 |
| 查看会议列表 | 组织者希望浏览所有历史会议 | 筛选条件（月份/状态） | 会议列表 |
| 查看会议详情 | 组织者希望查看某次会议的完整信息 | Meeting ID | 会议详情（议题/TODO/决策） |

**4. 与其他模块的关系**

- **依赖谁：**
  - TODO追踪模块：获取该会议关联的TODO列表
  - 决策记录模块：获取该会议的决策列表
- **被谁依赖：**
  - Dashboard模块：获取统计数据
  - 会议记录生成器：生成会议内容时引用

**5. 关键规则/限制**
- Meeting ID 使用日期格式 `YYYYMMDD`，同一天只能有一个会议
- 删除会议时，级联删除关联的TODO（需二次确认）
- 议题时长必须是正整数（分钟）

### 模块2：TODO追踪模块

**1. 模块职责**
- **做什么：** 追踪所有TODO任务的全生命周期（创建、完成、逾期检测、删除）
- **不做什么：** 不分配TODO（会议管理模块在会中记录时分配）、不通知责任人（未来由通知模块负责）

**2. 涉及的数据（实体）**

| 数据名称 | 说明 | 示例 |
|---------|------|------|
| Todo | 一个待办任务 | id: "20260518_001", content: "营销线IOC增加指标", assignee: "Alex.Fang", deadline: "2026-05-25", status: "pending" |

**3. 功能列表**

| 功能名称 | 用户故事 | 输入 | 输出 |
|---------|---------|------|------|
| 添加TODO | 组织者希望分配新任务 | 内容、责任人、截止日期、关联会议ID | 新Todo对象 |
| 完成TODO | 责任人希望标记任务已完成 | Todo ID | 状态变为"completed" |
| 撤销完成 | 责任人误标记，希望恢复 | Todo ID | 状态恢复为"pending" |
| 删除TODO | 组织者希望删除错误任务 | Todo ID | 删除成功/失败 |
| 筛选TODO | 用户希望按状态查看任务 | 状态筛选条件 | 过滤后的TODO列表 |
| 逾期检测 | 系统需要自动标记逾期任务 | 当前日期 | 逾期任务状态更新为"overdue" |

**4. 与其他模块的关系**

- **依赖谁：**
  - 会议管理模块：获取TODO关联的会议信息
- **被谁依赖：**
  - Dashboard模块：获取TODO统计数据
  - 会议管理模块：在会议详情页展示关联TODO

**5. 关键规则/限制**
- 截止日期 < 今天 且 状态 ≠ "completed" → 自动标记 "overdue"
- 删除TODO时不需要级联删除会议（会议可以独立存在）
- 同一会议下，TODO内容相同视为重复（导入时去重）

---

### 模块3：决策记录模块

**1. 模块职责**
- **做什么：** 结构化记录会议中的关键决策，支持分类、标签、追溯
- **不做什么：** 不做决策审批流（当前一期不设审批）、不通知相关方

**2. 涉及的数据（实体）**

| 数据名称 | 说明 | 示例 |
|---------|------|------|
| Decision | 一条决策记录 | id: "dec_001", content: "26年培训预算200万", type: "strategic", tags: ["预算"], basis: "Q1支出分析", impact: "所有部门", date: "2026-04-13" |

**3. 功能列表**

| 功能名称 | 用户故事 | 输入 | 输出 |
|---------|---------|------|------|
| 记录决策 | 组织者希望记录会议中的决策 | 内容、类型、标签、依据、影响范围、关联会议ID | 新Decision对象 |
| 编辑决策 | 组织者希望修改决策信息 | Decision ID、新字段值 | 更新后的Decision |
| 删除决策 | 组织者希望删除错误记录 | Decision ID | 删除成功/失败 |
| 智能类型推荐 | 系统根据决策内容推荐类型 | 决策内容文本 | 推荐类型（strategic/execution/pending） |

**4. 与其他模块的关系**

- **依赖谁：**
  - 会议管理模块：获取决策关联的会议信息
- **被谁依赖：**
  - Dashboard模块：获取最近决策列表
  - 会议管理模块：在会议详情页展示关联决策

**5. 关键规则/限制**
- 决策类型只能是三选一：strategic（战略）、execution（执行）、pending（待定）
- 智能推荐逻辑：含"预算/编制/战略/HC"关键词 → strategic；含"完成/制作/提交/复盘" → execution
- 标签使用英文逗号分隔，去重存储

### 模块4：会议记录生成器模块

**1. 模块职责**
- **做什么：** 通过表单交互，生成标准化的会议记录区块文本，支持回写KMS
- **不做什么：** 不直接操作KMS（由KMS同步模块负责）、不存储生成的内容（仅临时生成）

**2. 涉及的数据（实体）**

| 数据名称 | 说明 | 示例 |
|---------|------|------|
| MeetingForm | 生成器表单数据 | date: "2026-05-18", type: "hq", title: "本部例会", topics: [...], todos: [...], decisions: [...] |

**3. 功能列表**

| 功能名称 | 用户故事 | 输入 | 输出 |
|---------|---------|------|------|
| 填写会议信息 | 组织者希望录入会议基本信息 | 日期、类型、标题、记录人 | 表单状态更新 |
| 动态添加议题 | 组织者希望增加会议议题 | 标题、主讲人、时长 | 议题列表更新 |
| 动态添加TODO | 组织者希望增加TODO | 内容、责任人、截止日期 | TODO列表更新 |
| 动态添加决策 | 组织者希望增加决策 | 类型、内容、依据、影响范围、标签 | 决策列表更新 |
| 实时预览 | 组织者希望看到生成的标准格式 | 表单全部数据 | 标准区块文本（[MEETING]/[TOPIC]/[TODO]/[DECISION]） |
| 回写KMS | 组织者希望自动写入KMS | Page ID、生成的标准区块文本 | 写入成功/失败提示 |

**4. 与其他模块的关系**

- **依赖谁：**
  - KMS同步模块：执行实际的KMS写入操作
- **被谁依赖：**
  - （无，生成器是独立的入口模块）

**5. 关键规则/限制**
- 议题/TODO/决策可以动态添加删除，但至少要有1个议题
- 标准区块格式严格遵循 `[BLOCK]`...`[END]` 结构
- 回写KMS前必须检测本地代理服务是否运行

### 模块5：KMS同步模块

**1. 模块职责**
- **做什么：** 作为看板与KMS（Confluence）之间的桥梁，实现数据的双向读写和格式转换
- **不做什么：** 不解析业务逻辑（由批量解析模块负责）、不展示数据（由Dashboard等模块负责）

**2. 涉及的数据（实体）**

| 数据名称 | 说明 | 示例 |
|---------|------|------|
| KmsPage | KMS页面内容 | pageId: "310905380", title: "营销线重点工作经营例会", body: "<html>...", version: 1187 |
| StandardBlock | 标准区块文本 | "[MEETING]\n日期: 2026-05-18\n[END]" |

**3. 功能列表**

| 功能名称 | 用户故事 | 输入 | 输出 |
|---------|---------|------|------|
| 读取KMS页面 | 系统需要获取当前页面内容 | Page ID、API Token | KmsPage对象（含body和version） |
| 追加会议记录 | 系统需要在页面末尾追加内容 | Page ID、标准区块文本 | 更新后的KmsPage |
| 解析KMS内容 | 系统需要从页面内容中提取结构化数据 | KMS页面文本 | 解析结果（meeting/topics/todos/decisions） |
| 格式转换 | 系统需要将标准区块转为XHTML | 标准区块文本 | XHTML片段（转义后包在<pre>中） |

**4. 与其他模块的关系**

- **依赖谁：**
  - （无，KMS同步是底层服务，通过Confluence REST API直接操作）
- **被谁依赖：**
  - 会议记录生成器：调用回写功能
  - 批量解析模块：调用解析功能
  - 会议管理模块：未来可扩展为自动同步

**5. 关键规则/限制**
- 每次更新必须递增 version.number，否则Confluence拒绝写入
- 追加内容时放在 `<pre>` 标签内，保持格式原样
- 读取失败时返回明确错误码（401=Token失效，404=页面不存在）

### 模块6：Dashboard模块

**1. 模块职责**
- **做什么：** 聚合展示全局关键指标和最近动态，作为用户打开系统的第一屏
- **不做什么：** 不做数据修改（只读）、不做复杂分析（仅基础统计）

**2. 涉及的数据（实体）**

| 数据名称 | 说明 | 示例 |
|---------|------|------|
| DashboardMetrics | 仪表盘指标 | totalMeetings: 7, totalTopics: 32, totalTodos: 6, completionRate: "50%" |

**3. 功能列表**

| 功能名称 | 用户故事 | 输入 | 输出 |
|---------|---------|------|------|
| 统计指标 | 用户希望看到全局数据 | 全部Meeting和Todo数据 | 4个指标卡片 |
| 最近会议 | 用户希望快速查看最新会议 | 全部Meeting（取最近3条） | 会议卡片列表 |
| 关键TODO预警 | 用户希望看到需要关注的TODO | 全部Todo（pending/overdue） | 预警卡片列表 |
| 最近决策 | 用户希望快速查看最新决策 | 最近2次会议的决策 | 决策卡片列表 |

**4. 与其他模块的关系**

- **依赖谁：**
  - 会议管理模块：获取会议列表和议题数
  - TODO追踪模块：获取TODO列表和状态
  - 决策记录模块：获取决策列表
- **被谁依赖：**
  - （无，Dashboard是只读的展示层）

**5. 关键规则/限制**
- 最近会议按日期倒序，最多3条
- 关键TODO只显示pending和overdue，最多5条
- 最近决策取最近2次会议的决策，最多5条

### 模块8：会议效果评估模块

**1. 模块职责**
- **做什么：** 对每次会议进行多维度量化评估，生成评分报告和改进建议，追踪会议质量趋势
- **不做什么：** 不替代人工判断（仅辅助评估）、不修改会议数据（只读分析）

**2. 涉及的数据（实体）**

| 数据名称 | 说明 | 示例 |
|---------|------|------|
| MeetingEvaluation | 会议评估报告 | id: "eval_001", meetingId: "20260413", overallScore: 78, dimensions: [...], suggestions: [...] |
| DimensionScore | 维度得分 | dimension: "TODO闭环", score: 85, weight: 0.25, details: "3个TODO已完成2个" |
| Suggestion | 改进建议 | id: "sug_001", dimension: "议题效率", priority: "high", content: "建议控制议题数量", action: "将议题精简至3-5个" |

**3. 功能列表**

| 功能名称 | 用户故事 | 输入 | 输出 |
|---------|---------|------|------|
| 查看评估列表 | 组织者希望了解各次会议的质量评分 | 全部MeetingEvaluation | 评估报告列表（按时间倒序） |
| 评估单次会议 | 组织者希望对某次会议进行打分 | Meeting ID | MeetingEvaluation报告 |
| 查看评估详情 | 组织者希望了解评分细节和建议 | Evaluation ID | 各维度分数+改进建议 |
| 查看历史趋势 | 组织者希望看到会议质量变化 | 全部MeetingEvaluation | 趋势图表（时间轴×总分） |
| 查看最佳实践 | 组织者希望学习高分会议的特征 | 高分会议数据（≥85分） | 最佳实践总结 |
| 对比评估 | 组织者希望对比两次会议的质量 | 两个Meeting ID | 并排对比（各维度分数+差异） |

**4. 与其他模块的关系**

- **依赖谁：**
  - 会议管理模块：获取会议基本信息和议题数据
  - TODO追踪模块：获取TODO完成情况
  - 决策记录模块：获取决策质量数据
- **被谁依赖：**
  - Dashboard模块：展示最近评分和质量趋势

**5. 评估维度与打分规则**

| 维度 | 权重 | 满分条件 | 扣分规则 |
|------|------|---------|---------|
| **议题效率** | 20% | 议题3-5个，单个15-30min | 议题<2个或>8个扣20%；平均时长>40min扣15% |
| **TODO闭环** | 25% | 所有TODO在截止日期前完成 | 完成率直接换算分数（如完成2/3=67分） |
| **决策质量** | 20% | 所有决策有依据+影响范围+标签 | 缺失一项扣20%；待定决策过多扣10% |
| **会议效率** | 15% | 有议题/TODO/决策记录 | 每缺失一类扣30%；额外有会议记录人加分 |
| **记录完整** | 20% | 会议信息完整（日期/标题/议题/记录人） | 基础分60%；有TODO+10%；有决策+10%；有类型标签+10%；≥90分额外+10% |

**总分计算**：各维度分数 × 权重后求和，四舍五入取整

**6. 建议生成规则**

| 维度分数 | 优先级 | 建议类型 |
|---------|--------|---------|
| < 60 | high | 必须改进，给出具体行动 |
| 60-80 | medium | 有待提升，给出优化方向 |
| 80-100 | low | 保持或表扬 |

**7. 内置建议模板**

| 维度 | 低分场景 | 建议内容 | 具体行动 |
|------|---------|---------|---------|
| 议题效率 | 议题过多 | 议题数量超标，建议精简 | 将议题控制在3-5个，合并相似议题 |
| 议题效率 | 议题时长不均 | 议题时长分配不合理 | 重点议题分配20-30min，常规议题10-15min |
| TODO闭环 | 完成率低 | TODO闭环率偏低 | 会中明确责任人和截止日期，会后3天内首次跟进 |
| TODO闭环 | 责任人模糊 | 多个TODO责任人为TBD | 会中必须指定具体责任人，禁止用"TBD" |
| 决策质量 | 依据缺失 | 决策缺少依据说明 | 记录决策时填写"为什么做这个决策" |
| 决策质量 | 影响范围不明 | 决策未标注影响范围 | 明确标注影响哪些团队/业务线 |
| 会议效率 | 缺少TODO | 会议未产生TODO | 每次会议至少产生1-3个可执行的TODO |
| 会议效率 | 缺少决策 | 会议未记录决策 | 会议中达成的共识必须记录为决策 |
| 记录完整 | 信息不全 | 会议基本信息不完整 | 确保填写日期、标题、记录人、会议类型 |

### 模块9：会议原则管理模块

**1. 模块职责**
- **做什么：** 管理会议质量原则（规则），根据原则自动扫描所有会议数据，发现问题项目并生成统计报告
- **不做什么：** 不修改会议数据（只标记问题，不自动修复）、不替代人工判断（仅辅助检查）

**2. 涉及的数据（实体）**

|---------|------|------|

| RuleViolation | 原则违反记录 | id: "vio_001", ruleId: "rule_001", meetingId: "20260413", itemType: "todo", itemId: "20260413_148", message: "TODO未设置截止日期" |

**3. 功能列表**

| 功能名称 | 用户故事 | 输入 | 输出 |
|---------|---------|------|------|
| 查看原则列表 | 组织者希望了解当前有哪些检查规则 | 无 | 原则列表（含启用状态） |
| 添加原则 | 组织者希望新增一条质量检查规则 | 原则名称、描述、分类、条件类型、严重程度 | 新MeetingRule对象 |
| 编辑原则 | 组织者希望修改现有规则 | Rule ID、新字段值 | 更新后的MeetingRule |
| 删除原则 | 组织者希望删除不再需要的规则 | Rule ID | 删除成功/失败 |
| 启用/禁用原则 | 组织者希望临时关闭某条规则 | Rule ID、enabled状态 | 更新后的状态 |
| 运行扫描 | 组织者希望检查所有会议数据的质量 | 全部Meeting/Todo/Decision数据 | 违规列表 RuleViolation[] |
| 查看问题报告 | 组织者希望查看扫描发现的问题 | 违规列表 | 按严重程度分组的问题报告 |
| 问题统计 | 组织者希望看到问题趋势 | 历史违规数据 | 统计图表（问题数量/趋势/分类占比） |

**4. 与其他模块的关系**

- **依赖谁：**
  - 会议管理模块：获取全部会议数据进行扫描
  - TODO追踪模块：获取TODO数据进行规则检查
  - 决策记录模块：获取决策数据进行规则检查
- **被谁依赖：**
  - Dashboard模块：获取问题统计数据展示

**5. 关键规则/限制**
- 内置原则不可删除（只能禁用），如：TODO必须有截止日期、决策必须记录依据
- 扫描时只检查 enabled=true 的原则
- 同一项目可能被多条原则命中，需去重展示
- 问题报告按严重程度排序：fatal > warning > suggestion

**6. 内置原则清单（预设）**

## 六、数据模型

### 6.1 Meeting（会议）

```typescript
interface Meeting {
  id: string;           // YYYYMMDD 格式，如 "20260518"
  date: string;         // "2026-05-18"
  month: string;        // "2026-05"
  type: 'hq' | 'weekly' | 'special';
  title: string;        // "本部重点工作经营例会"
  status: 'planned' | 'completed';
  agenda_items: AgendaItem[];
  decisions: Decision[];
}

interface AgendaItem {
  title: string;
  duration: number;     // 分钟
  speaker: string;
}
```

### 6.2 Todo

```typescript
interface Todo {
  id: string;           // 如 "20260518_001"
  content: string;
  assignee: string;
  deadline: string;     // "2026-05-25"
  status: 'pending' | 'completed' | 'overdue';
  source_meeting: string;  // 关联会议ID
}
```

### 6.3 Decision（决策）

```typescript
interface Decision {
  id: string;
  content: string;
  type: 'strategic' | 'execution' | 'pending';
  tags: string[];       // ['预算', '人力']
  basis: string;        // 决策依据
  impact: string;       // 影响范围
  date: string;         // 记录日期
}
```

# 第二部分：前端设计

## 九、设计思维（基于 frontend-design-SKILL）

### 9.1 目的（Purpose）

| 问题 | 回答 |
|------|------|
| **解决什么问题** | 经营例会「会前无准备、会中无记录、会后无跟踪」的三无困境 |
| **核心用户** | ① 会议组织者（Jason）② 参会负责人（各条线负责人）③ 决策执行人（TODO责任人） |
| **使用场景** | 每周/双周的固定经营例会，从议题安排 → 会议召开 → TODO追踪 → 决策归档的全周期 |

### 9.2 基调（Tone）

**方向选择：工业实用（Industrial）+ 编辑杂志（Editorial）的融合**

| 维度 | 定义 |
|------|------|
| **核心关键词** | 功能优先、数据密集、无装饰、工具感 + 大标题、信息层级清晰 |
| **为什么选这个** | 经营例会看板是**高频使用的生产力工具**，不是营销页面。用户需要的是：一眼看到关键数据、快速操作、高效决策。工业实用的工具感建立专业信任，编辑杂志的信息层级确保复杂数据不乱 |
| **差异化记忆点** | 「一条决策线」——每个决策卡片左侧的彩色竖线，从页面顶部贯穿到底部，形成强烈的视觉记忆 |

### 9.3 约束（Constraints）

| 约束类型 | 说明 |
|---------|------|
| **技术栈** | 单页 HTML（当前）→ 未来可迁移至 React/Vue |
| **数据源** | KMS（Confluence）双向读写 + 本地 localStorage 缓存 |
| **部署方式** | 静态页面 + 本地代理服务（kms_proxy.py） |
| **性能要求** | 首屏 < 1s，交互响应 < 100ms |
| **无障碍** | WCAG 2.1 AA 标准，键盘可全操作 |

### 9.4 差异化（Differentiation）

**用户离开后，脑子里记住的那一样东西是什么？**

> **「那条紫色的决策线」** —— 每个决策卡片左侧的彩色竖线（战略=紫、执行=蓝、待定=橙），在深色背景上形成强烈的视觉锚点。这是其他会议工具没有的设计语言。

## 十二、视觉设计方向

### 12.1 设计关键词

**专业 · 紧凑 · 可信 · 高效**

> 基于 frontend-design-SKILL 的「工业实用」方向，辅以「编辑杂志」的信息层级感。

### 12.2 色彩系统

```css
:root {
  /* 背景层 */
  --bg-primary: #0f172a;      /* 深蓝灰 - 主背景 */
  --bg-card: #1e293b;         /* 卡片背景 */
  --bg-input: #0f172a;        /* 输入框背景 */
  
  /* 边框 */
  --border-default: #334155;  /* 默认边框 */
  --border-hover: #475569;    /* hover 边框 */
  
  /* 文字 */
  --text-primary: #e2e8f0;    /* 主文字 */
  --text-secondary: #94a3b8;  /* 次要文字 */
  --text-muted: #64748b;      /* 辅助说明 */
  
  /* 功能色 */
  --color-blue: #3b82f6;      /* 主操作/议题 */
  --color-amber: #fbbf24;     /* 待处理/警告 */
  --color-green: #10b981;     /* 已完成/成功 */
  --color-red: #ef4444;       /* 逾期/错误 */
  --color-purple: #a855f7;    /* 战略决策 */
  --color-violet: #8b5cf6;    /* 回写按钮 */
}
```

### 12.3 排版系统

| 层级 | 字号 | 字重 | 用途 |
|------|------|------|------|
| H1 | 20px | 700 | 页面标题 |
| H2 | 18px | 600 | 模块标题 |
| H3 | 16px | 600 | 卡片标题 |
| Body | 14px | 400 | 正文内容 |
| Small | 12px | 400 | 辅助信息/标签 |
| Mono | 13px | 400 | 代码块/数据 |

**字体选择**：
- 中文：`Noto Sans SC`（思源黑体）— 干净、专业、跨平台一致
- 英文/数字：`Inter` — 现代、清晰、屏幕优化
- 等宽数据：`Menlo / Monaco` — 数据对齐

> 注：前端-design-SKILL 建议避免 Inter，但对于「工业实用」方向的数据密集工具界面，Inter 的屏幕优化特性非常合适。这不是营销页面，不需要「过目不忘」的字体性格，需要的是「长时间阅读不疲劳」的功能性。

### 12.4 间距系统

| Token | 值 | 用途 |
|-------|-----|------|
| xs | 4px | 图标与文字间距 |
| sm | 8px | 紧凑内边距 |
| md | 16px | 卡片内边距 |
| lg | 24px | 模块间距 |
| xl | 32px | 页面边距 |

### 12.5 差异化视觉元素

**「决策线」设计语言** —— 整个产品的视觉记忆点：

```
每个决策卡片左侧有一条 3px 的彩色竖线：

┃ 战略决策  ← 紫色 #a855f7
┃ 执行决策  ← 蓝色 #3b82f6  
┃ 待定      ← 橙色 #f59e0b

这条线从卡片顶部贯穿到底部，与卡片圆角形成"切角"效果。
在 Dashboard 的「最近决策」区域，这条线成为整个列表的视觉节奏。
```
