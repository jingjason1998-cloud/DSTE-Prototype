## 一、产品概述

### 1.1 模块定位

**组织绩效管理模块**是 DSTE 战略管理平台在**战略执行阶段**的核心监控枢纽。它承接战略解码（BP）产出的 KPI 体系和重点工作清单，通过**"目标基线管理 → 执行跟踪 → 偏差分析 → 改进闭环"**的四步循环，确保组织战略目标可量化、可追踪、可达成。

**核心价值主张**：
- **从"结果考核"到"过程管理"** —— 不仅看最终达成率，更关注执行过程中的偏差、趋势和风险
- **从"静态报表"到"动态追踪"** —— KPI 数据自动采集、重点工作进度实时更新、偏差自动预警
- **从"单点指标"到"关联分析"** —— KPI 与重点工作关联打通，指标异常自动定位到执行动作
- **从"月度回顾"到"持续监控"** —— 支持日/周/月/季度多频次跟踪，及时发现并纠偏

## 四、子系统1: KPI 管理

> **定位**: KPI 管理是组织绩效的**量化引擎**。它将战略目标转化为可衡量、可追踪的指标体系，通过自动采集和智能分析，让组织绩效"看得见、说得清、管得住"。
>
> **与 PRD 模块2 (KPI 指标体系) 的区别**: PRD 模块2 侧重**指标体系建设**（战略解码层的定义和分解），本子系统侧重**指标执行跟踪**（战略执行层的采集、分析、改进）。

### 4.1 模块1.1: 指标库管理

#### 4.1.1 模块职责
- **做什么**: 维护企业级指标库，统一指标定义、计算公式、数据口径
- **不做什么**: 不设定具体战略周期的目标值（由年度业务规划模块负责）、不采集数据（由数据采集模块负责）

#### 4.1.2 涉及的数据实体

| 实体 | 说明 | 关键字段 |
|------|------|---------|
| **指标定义** | 企业级指标的标准定义 | 指标ID、名称、定义、计算公式、单位、统计频次、数据来源、责任部门、业务类型 |
| **指标分类** | 指标的分类体系 | 分类ID、名称、父分类ID、排序、描述 |
| **指标版本** | 指标定义的变更历史 | 版本ID、指标ID、版本号、变更说明、变更时间、变更人 |
| **指标关联** | 指标之间的关联关系 | 关联ID、源指标ID、目标指标ID、关联类型（因果/相关/父子） |

#### 4.1.3 指标分类体系

基于平衡计分卡 (BSC) 四维度，结合业务类型进行二级分类：

```
📊 财务指标
   ├── 收入类（营收、合同额、回款额）
   ├── 利润类（毛利、净利润、利润率）
   ├── 成本类（运营成本、人力成本、营销成本）
   └── 投资回报类（ROI、EVA）

👤 客户指标
   ├── 客户满意度（NPS、CSAT、CES）
   ├── 客户增长（新增客户、客户留存率）
   ├── 客户价值（客单价、客户生命周期价值）
   └── 市场份额（市场占有率、品牌认知度）

⚙️ 内部流程指标
   ├── 运营效率（交付周期、人均产出）
   ├── 质量指标（缺陷率、一次通过率）
   ├── 创新指标（新产品收入占比、专利申请数）
   └── 合规指标（合规率、审计问题数）

📚 学习与成长指标
   ├── 人才发展（人均培训时长、关键岗位胜任率）
   ├── 组织活力（员工敬业度、离职率）
   ├── 数字化能力（系统使用率、数据覆盖率）
   └── 知识管理（知识沉淀数、最佳实践复用率）
```

#### 4.1.4 功能列表

| 功能 | 用户故事 | 输入 | 输出 |
|------|---------|------|------|
| 创建指标 | 绩效管理员希望定义新的指标 | 指标名称、定义、公式、单位、分类、频次 | 指标库新条目 |
| 编辑指标 | 绩效管理员希望修改指标定义 | 指标ID、变更字段、变更说明 | 更新后的指标（生成新版本） |
| 查看指标库 | 用户希望浏览所有可用指标 | 筛选条件（分类/频次/部门） | 指标列表 |
| 指标详情 | 用户希望了解指标的完整定义 | 指标ID | 指标详情（定义+公式+历史版本） |
| 指标关联 | 绩效管理员希望设置指标间关系 | 源指标、目标指标、关联类型 | 关联关系图 |
| 指标导入 | 绩效管理员希望批量导入指标 | Excel/CSV 文件 | 导入结果（成功/失败明细） |
| 指标导出 | 用户希望导出指标库 | 筛选条件 | Excel/CSV 文件 |

#### 4.1.5 关键规则
- 指标名称必须唯一，不可重复
- 指标计算公式变更时，必须创建新版本，保留历史公式
- 已被使用的指标（已有 KPI 实例）不可删除，只能禁用
- 指标分类支持自定义扩展，但 BSC 四维度为系统内置，不可删除

### 4.2 模块1.2: KPI 目标基线管理 / 目标变更

#### 4.2.1 模块职责

- **做什么**: 
  - 接收年度业务规划（ABP）发布后同步的 KPI 目标基线（targetValue、baseline、challenge、weight、分解关系、职能线权重等）
  - 在执行期维护 KPI 目标基线的只读引用
  - 发起目标变更申请，当实际执行出现重大偏差需要调整目标时
  - 审批通过后，接收 ABP 新版本的目标基线
- **不做什么**: 
  - **不制定年度 KPI 目标值**（由 ABP 年度业务规划模块负责）
  - **不做 KPI 分解**（由 ABP 负责）
  - **不做资源分配**（由 ABP 负责）
  - **不直接修改已发布的 ABP 目标**（必须通过目标变更流程）

> **定位澄清**: OMP 是 KPI 目标基线的"消费者"和"变更申请的触发者"，ABP 是 KPI 目标基线的"制定者"和"发布者"。

#### 4.2.2 与 ABP 的边界

| 数据/操作 | ABP（规划期） | OMP（执行期） |
|----------|--------------|--------------|
| 制定年度 KPI 目标值 | ✅ | ❌ |
| 分解 KPI 到组织/职能 | ✅ | ❌ |
| 配置职能线权重 | ✅ | ❌ |
| 发布并锁定目标 | ✅ | ❌（只读引用） |
| 录入 actual | ❌ | ✅ |
| 计算达成率 | ❌ | ✅ |
| 发起目标变更申请 | ❌（审批） | ✅（触发） |
| 审批目标变更 | ✅ | ❌ |

#### 4.2.3 涉及的数据实体

| 实体 | 说明 | 关键字段 |
|------|------|---------|
| **KPI 实例（只读引用）** | ABP 发布后同步的 KPI 目标基线 | 实例ID、指标ID、周期ID、目标值、挑战值、权重、责任人、责任部门、ABP 版本号 |
| **KPI 分解（只读引用）** | ABP 发布后同步的层级分解关系 | 分解ID、父KPI实例ID、子KPI实例ID、分解比例 |
| **职能线权重（只读引用）** | ABP 发布后同步的职能线权重 | KPI实例ID、职能线、权重 |
| **目标调整申请** | 执行期目标变更的申请记录 | 调整ID、KPI实例ID、原目标值、新目标值、调整原因、偏差证据、审批状态 |
| **目标调整审批记录** | 审批历史 | 审批ID、调整申请ID、审批人、审批意见、审批时间 |

#### 4.2.4 KPI 目标基线的来源与同步

```
ABP 年度业务规划
    │
    │ 发布并锁定
    ▼
OMP 组织绩效管理
    │
    ├─ 接收 KPI 目标树（只读）
    ├─ 接收权重配置（只读）
    ├─ 接收计分规则（只读）
    ├─ 接收职能线权重（只读）
    └─ 接收分解关系（只读）
```

同步规则：
- ABP 发布后，OMP 自动同步目标基线，生成"执行期 KPI 实例"
- OMP 中的目标基线默认只读，不可直接编辑
- 如需修改，必须发起目标变更申请，流转回 ABP 审批

#### 4.2.5 目标变更流程

```
OMP 发现重大偏差
    │
    ▼
OMP 用户发起"目标变更申请"
    │
    ▼
填写：变更原因、建议目标值、偏差证据
    │
    ▼
提交到 ABP 进行变更评审
    │
    ▼
ABP 审批通过 → 更新规划版本 → 重新发布
    │
    ▼
OMP 接收新版本目标基线
    │
    ▼
OMP 基于新目标继续执行跟踪
```

变更申请的数据结构：
```typescript
interface TargetChangeRequest {
  id: string;
  cycleId: string;
  kpiInstanceId: string;
  originalTarget: number;
  proposedTarget: number;
  reason: string;                  // 变更原因
  deviationEvidence: string;       // 偏差证据
  requestedBy: string;
  requestedAt: string;
  status: 'pending' | 'approved' | 'rejected';
  reviewedBy?: string;
  reviewedAt?: string;
  comment?: string;
  newAbpVersion?: number;          // 审批通过后生成的 ABP 版本
}
```

#### 4.2.6 功能列表

| 功能 | 用户故事 | 输入 | 输出 |
|------|---------|------|------|
| 查看目标基线 | 绩效管理员希望查看已发布的 KPI 目标 | 周期ID | 只读的 KPI 目标树 |
| 同步 ABP 目标 | ABP 发布后系统自动同步 | ABP 发布事件 | 执行期 KPI 实例 |
| 发起目标变更 | 部门负责人发现目标不合理，申请调整 | KPI实例ID、新目标值、原因、偏差证据 | 目标变更申请 |
| 查看变更申请 | 审批人希望查看待审批的变更 | 变更申请ID | 变更详情 |
| 接收新版目标 | ABP 审批通过后，OMP 自动更新 | 新版本 ABP 数据 | 更新后的目标基线 |
| 查看 KPI 树 | 用户希望查看 KPI 的层级结构 | KPI 实例ID | 层级分解树视图 |

#### 4.2.7 目标基线管理流程

```
ABP 年度业务规划完成并发布
    │
    ▼
OMP 自动同步 KPI 目标基线
    │
    ▼
执行期：录入 actual、计算达成率、分析偏差
    │
    ▼
发现重大偏差或目标不适用
    │
    ▼
OMP 发起目标变更申请
    │
    ▼
ABP 审批变更 → 更新规划版本 → 重新发布
    │
    ▼
OMP 接收新版本目标基线
    │
    ▼
继续执行跟踪
```

### 4.3 模块1.3: KPI 数据采集

#### 4.3.1 模块职责
- **做什么**: 采集 KPI 的实际数据，支持自动采集和手动录入两种方式
- **不做什么**: 不修改指标定义（由指标库管理负责）、不分析数据（由达成分析模块负责）

#### 4.3.2 采集方式对比

| 采集方式 | 适用场景 | 优先级 | 数据质量 | 成本 |
|---------|---------|--------|---------|------|
| **BI 系统自动采集** | 已有 BI 系统的指标数据 | 最高 | 高 | 低 |
| **API 接口对接** | 业务系统（ERP/CRM/HR）数据 | 高 | 高 | 中 |
| **Excel 批量导入** | 周期性批量数据 | 中 | 中 | 低 |
| **手动录入** | 无系统来源的指标 | 最低 | 依赖人 | 高 |

#### 4.3.3 涉及的数据实体

| 实体 | 说明 | 关键字段 |
|------|------|---------|
| **KPI 数据记录** | 单期实际数据 | 记录ID、KPI实例ID、周期、实际值、采集方式、采集时间、录入人 |
| **采集任务** | 定时采集配置 | 任务ID、KPI实例ID、采集方式、采集周期、下次执行时间、状态 |
| **采集日志** | 采集执行记录 | 日志ID、任务ID、执行时间、执行结果、错误信息 |

#### 4.3.4 功能列表

| 功能 | 用户故事 | 输入 | 输出 |
|------|---------|------|------|
| 配置自动采集 | 绩效管理员希望配置 BI 数据源 | KPI 实例ID、数据源配置、采集周期 | 采集任务 |
| 手动录入数据 | 数据录入员希望录入实际值 | KPI 实例ID、周期、实际值 | 数据记录 |
| 批量导入数据 | 数据录入员希望批量导入 | Excel/CSV 文件 | 导入结果（成功/失败明细） |
| 查看数据历史 | 用户希望查看某 KPI 的历史数据 | KPI 实例ID、时间范围 | 数据列表/趋势图 |
| 数据审核 | 绩效管理员希望审核录入的数据 | 记录ID、审核意见 | 审核结果（通过/驳回） |
| 数据修正 | 录入员希望修正错误数据 | 记录ID、修正值、修正原因 | 修正记录（保留原值） |

#### 4.3.5 关键规则
- 自动采集优先于手动录入
- 手动录入的数据需经审核才能用于绩效计算
- 数据修正必须保留原值和修正记录，不可直接覆盖
- 同一 KPI 同一周期只能有一条有效数据记录

### 4.4 模块1.4: KPI 达成分析

#### 4.4.1 模块职责
- **做什么**: 计算 KPI 达成率，进行多维度分析（趋势、对比、排名）
- **不做什么**: 不诊断偏差根因（由偏差诊断模块负责）、不生成改进方案（由 AI 助手负责）

#### 4.4.2 核心计算公式

| 指标 | 公式 | 说明 |
|------|------|------|
| **达成率** | 实际值 / 目标值 × 100% | 正向指标（越高越好） |
| **偏差率** | (实际值 - 目标值) / 目标值 × 100% | 正值=超额，负值=落后 |
| **完成状态** | 达成率 ≥ 100%: 达成; ≥ 80%: 预警; < 80%: 落后 | 阈值可配置 |
| **同比** | (本期实际值 - 同期实际值) / 同期实际值 × 100% | 与去年同期对比 |
| **环比** | (本期实际值 - 上期实际值) / 上期实际值 × 100% | 与上一期对比 |

#### 4.4.3 分析维度

```
KPI 达成分析
├── 时间维度
│   ├── 趋势分析：近 N 期的达成率走势
│   ├── 同比分析：与去年同期对比
│   └── 环比分析：与上一期对比
│
├── 组织维度
│   ├── 部门排名：同一 KPI 跨部门达成率排名
│   ├── 层级对比：公司级 vs 部门级 vs 团队级
│   └── 标杆对比：最佳实践部门 vs 落后部门
│
└── 关联维度
    ├── KPI 关联：相关 KPI 的同步变化分析
    └── 工作关联：KPI 与重点工作的关联影响分析
```

#### 4.4.4 功能列表

| 功能 | 用户故事 | 输入 | 输出 |
|------|---------|------|------|
| 达成率计算 | 系统希望自动计算各 KPI 达成率 | KPI 数据记录、目标值 | 达成率、完成状态 |
| 趋势分析 | 用户希望查看 KPI 历史趋势 | KPI 实例ID、时间范围 | 趋势图表 |
| 部门排名 | 用户希望查看跨部门排名 | KPI 实例ID、部门范围 | 排名列表/柱状图 |
| 同比环比 | 用户希望对比不同时期 | KPI 实例ID、对比周期 | 对比数据 |
| 达成率汇总 | 用户希望查看部门/公司整体达成情况 | 组织范围、KPI 范围 | 汇总数据/热力图 |
| 异常识别 | 系统希望自动识别异常 KPI | 全部 KPI 数据 | 异常清单（偏差过大/趋势恶化） |

#### 4.5.1 模块职责
- **做什么**: 对达成率异常的 KPI 进行根因分析，定位问题来源
- **不做什么**: 不替代业务专家的判断（仅提供分析框架和数据支撑）、不自动执行改进动作

#### 4.5.2 偏差诊断框架

```
KPI 偏差诊断框架（参考 5Why + 鱼骨图）

│  ├─ 根因分类：人员/流程/系统/资源/外部环境                    │

│                                                              │
│  Step 4: 生成建议                                            │
│  ├─ 短期措施：立即执行的纠偏动作                              │

└─────────────────────────────────────────────────────────────┘
```

#### 4.5.3 功能列表

| 功能 | 用户故事 | 输入 | 输出 |
|------|---------|------|------|
| 偏差识别 | 系统希望自动识别偏差 KPI | 全部 KPI 达成率数据 | 偏差清单（含偏差类型和程度） |
| 根因分析 | 系统希望对偏差进行初步分析 | KPI 实例ID、偏差数据 | 分析建议（分层诊断结果） |
| 关联分析 | 系统希望分析与偏差 KPI 关联的因素 | KPI 实例ID | 关联因素列表（相关工作、关联 KPI、外部事件） |
| 诊断记录 | 用户希望记录根因分析结论 | KPI 实例ID、根因、改进措施 | 诊断记录 |
| 改进跟踪 | 系统希望跟踪改进措施的执行效果 | 改进措施、跟踪周期 | 改进效果评估 |

## 五、子系统2: 重点工作管理

> **定位**: 重点工作管理是组织绩效的**执行引擎**。它将战略解码产出的战略举措转化为可追踪、可管理的工作清单，通过里程碑驱动和进度跟踪，确保战略举措落地执行。
>
> **与 PRD 模块4 (重点工作管理) 的区别**: PRD 模块4 是通用的项目化管理，本子系统增加了**组织绩效视角**的关联设计：每项重点工作必须关联到具体的 KPI，工作完成度直接影响 KPI 达成率。

### 5.1 模块2.1: 重点工作台账

#### 5.1.1 模块职责
- **做什么**: 管理战略级重点工作的全生命周期（创建、跟踪、结项）
- **不做什么**: 不替代专业项目管理工具（如 Jira），聚焦战略级工作的宏观管理；不做详细的任务级管理

#### 5.1.2 涉及的数据实体

| 实体 | 说明 | 关键字段 |
|------|------|---------|
| **重点工作** | 战略级工作项 | 工作ID、名称、类型、优先级、状态、进度、负责人、责任部门、关联KPI、起止时间、预算 |
| **工作类型** | 工作分类 | 战略举措、业务改进、能力建设、风险应对、其他 |
| **工作变更** | 工作的变更记录 | 变更ID、工作ID、变更字段、原值、新值、变更原因、变更人、变更时间 |

#### 5.1.3 重点工作类型定义

| 类型 | 说明 | 典型示例 | 关联模块 |
|------|------|---------|---------|
| **战略举措** | 直接支撑战略目标的行动 | "华东区渠道下沉"、"海外市场拓展" | 战略地图 |
| **业务改进** | 针对业务问题的专项改进 | "客户响应时效优化"、"库存周转率提升" | 业务专题 |
| **能力建设** | 组织能力短板补齐 | "销售团队赋能培训"、"CRM系统升级" | 干部管理 |
| **风险应对** | 针对已识别风险的对策 | "大客户流失风险应对"、"供应链中断预案" | 预警中心 |

### 5.2 模块2.2: 里程碑管理

#### 5.2.1 模块职责
- **做什么**: 为重点工作设定和管理关键里程碑节点
- **不做什么**: 不做详细的任务分解（任务级管理由 OA/项目管理工具负责）

#### 5.2.2 涉及的数据实体

| 实体 | 说明 | 关键字段 |
|------|------|---------|
| **里程碑** | 关键节点 | 里程碑ID、工作ID、名称、计划日期、实际日期、状态、完成说明 |
| **里程碑依赖** | 里程碑之间的前后关系 | 依赖ID、前序里程碑ID、后续里程碑ID |

#### 5.2.3 里程碑状态

| 状态 | 说明 | 触发条件 |
|------|------|---------|
| **未开始** | 里程碑尚未到达计划日期 | 当前日期 < 计划日期 |
| **进行中** | 里程碑已到达计划日期，尚未完成 | 当前日期 ≥ 计划日期，且无实际完成日期 |
| **已完成** | 里程碑已按时或提前完成 | 实际完成日期 ≤ 计划日期 |
| **已延期** | 里程碑未在计划日期完成 | 当前日期 > 计划日期，且无实际完成日期 |

#### 5.2.4 进度自动计算

```
工作整体进度 = 已完成里程碑数 / 总里程碑数 × 100%

当存在里程碑依赖时：
- 前序里程碑未完成，后续里程碑自动标记为"阻塞"
- 阻塞状态的里程碑不计入进度计算
```

#### 5.2.5 功能列表

| 功能 | 用户故事 | 输入 | 输出 |
|------|---------|------|------|

```sql
    progress        INT COMMENT '当期进度百分比',
    completed_content TEXT COMMENT '完成内容',
    problems        TEXT COMMENT '遇到的问题',
    next_plan       TEXT COMMENT '下期计划',
    reported_by     BIGINT COMMENT '汇报人ID',
    created_at      TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    
    INDEX idx_task (task_id),
    INDEX idx_date (report_date)
) COMMENT='进度记录表';
```

#### 风险/问题表 (omp_risk_issue)

```sql
CREATE TABLE omp_risk_issue (
    item_id         BIGINT PRIMARY KEY AUTO_INCREMENT COMMENT 'ID',
    task_id         BIGINT NOT NULL COMMENT '工作ID',
    item_type       ENUM('risk','issue') COMMENT '类型：风险/问题',
    title           VARCHAR(200) COMMENT '标题',
    description     TEXT COMMENT '描述',
    level           ENUM('high','medium','low') COMMENT '等级',
    probability     DECIMAL(3,2) COMMENT '概率（仅风险）',
    impact          TEXT COMMENT '影响说明',
    solution        TEXT COMMENT '解决方案/应对措施',
    responsible_id  BIGINT COMMENT '责任人ID',
    status          ENUM('open','in_progress','resolved','closed') DEFAULT 'open',
    resolved_at     TIMESTAMP COMMENT '解决时间',
    created_at      TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    
    INDEX idx_task (task_id),
    INDEX idx_type (item_type),
    INDEX idx_level (level),
    INDEX idx_status (status)
) COMMENT='风险/问题表';
```

### 9.1 页面架构

```
组织绩效管理
├── 📊 绩效看板（首页）
│   ├── 组织绩效总览
│   ├── 部门绩效对比
│   ├── KPI 达成热力图
│   └── 重点工作进度视图
│
├── 📈 KPI 管理
│   ├── 指标库
│   │   └── 指标详情弹窗
│   ├── KPI 目标基线管理
│   │   └── 目标基线查看/变更申请弹窗
│   ├── KPI 数据采集
│   │   └── 数据录入/导入弹窗
│   ├── KPI 达成分析
│   │   └── 偏差诊断弹窗
│   └── 部门 KPI 看板
│
├── ✅ 重点工作管理
│   ├── 重点工作台账
│   │   └── 工作详情弹窗
│   ├── 甘特图视图
│   ├── 进度汇报
│   │   └── 汇报弹窗
│   ├── 风险/问题管理
│   │   └── 风险/问题详情弹窗
│   └── 资源管理
│
└── 📑 绩效报告
    ├── 周期报告列表
    │   └── 报告详情弹窗
    ├── 偏差分析报告
    └── 趋势预测报告
```

### 9.2 核心交互流程

**流程1: KPI 目标基线查看与变更申请**

```
进入 KPI 目标基线管理页面
    │
    ▼
选择战略周期
    │
    ▼
从 ABP 同步已发布的 KPI 目标基线（自动/手动）
    │
    ▼
查看 KPI 目标树（只读）
    │
    ▼
执行期：录入 actual、计算达成率、分析偏差
    │
    ▼
发现重大偏差或目标不适用
    │
    ▼
发起目标变更申请
    │    ┌────────────────────────┐
    └───►│ 变更申请信息：          │
         │ • 变更的 KPI           │
         │ • 建议目标值           │
         │ • 变更原因             │
         │ • 偏差证据             │
         └────────────────────────┘
    │
    ▼
提交到 ABP 进行变更评审
    │
    ▼
ABP 审批通过后，OMP 自动同步新版本目标基线
    │
    ▼
基于新目标基线继续执行跟踪
```

**流程2: 重点工作全生命周期**

```
创建重点工作
    │
    ▼
设定基本信息（名称/类型/负责人/起止时间）
    │
    ▼
关联 KPI（必须关联至少一个 KPI）
    │
    ▼
设定里程碑（至少1个关键节点）
    │
    ▼
进入执行阶段
    ├── 定期进度汇报（按优先级频率）
    ├── 里程碑完成时更新状态
    ├── 登记风险和问题
    └── 记录资源投入
    │
    ▼
工作完成 → 提交结项报告
    │
    ▼
绩效管理员审核结项
    │
    ▼
工作关闭，更新关联 KPI 达成数据
```

**流程3: 偏差发现与闭环**

```
系统自动检测 KPI 偏差（或手动发现）
    │
    ▼
偏差分级（轻微/中等/严重）
    │
    ▼
生成偏差预警（输出到预警中心）
    │
    ▼
负责人收到通知，查看偏差详情
    │
    ▼
系统提供根因分析建议（AI 辅助）
    │
    ▼
负责人记录根因分析和改进措施
    │
    ▼
创建改进工作（关联到原 KPI）
    │
    ▼
跟踪改进措施执行
    │
    ▼
验证改进效果（下一周期数据）
    │
    ▼
偏差闭环，归档记录
```

## 十、非功能需求

### 10.1 性能要求

| 指标 | 要求 | 说明 |
|------|------|------|
| 绩效看板首屏加载 | < 2s | 包含所有图表和数据卡片 |
| KPI 数据查询 | < 500ms | 单 KPI 多期数据查询 |
| 部门对比查询 | < 1s | 跨部门 KPI 对比 |
| 重点工作列表查询 | < 500ms | 支持筛选和分页 |
| 甘特图渲染 | < 1s | 20 个工作项以内 |
| 报告生成 | < 5s | 月度报告自动生成 |

### 10.2 安全要求

- KPI 数据按组织架构隔离（只能查看本部门及下级部门数据）
- 目标值调整需审批流程，留痕审计
- 数据修正必须保留原值和修正记录
- 手动录入数据需审核后才能用于绩效计算
- 敏感绩效数据脱敏展示（如个人绩效排名）

### 10.3 可用性要求

- 支持 KPI 数据批量导入（Excel/CSV）
- 支持自定义指标计算公式
- 支持自定义预警阈值
- 支持自定义报告模板
- 支持数据导出（Excel/PDF）

### 10.4 扩展性要求

- 支持新增指标分类和维度
- 支持自定义分解规则
- 支持第三方系统数据对接（BI/ERP/CRM）
- 支持多租户 SaaS 架构
