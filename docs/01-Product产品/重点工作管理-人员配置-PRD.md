## 二、产品定位与边界

### 2.1 模块定位

「人员配置」是 OMP 重点工作管理子模块的**资源分配能力**，解决“重点工作由谁来做”的问题。它连接：
- **上游**：人员与组织目录（员工、部门/团队、组织架构）
- **下游**：重点工作台账（`task.members`）
- **约束**：年度经营计划对负责人（`owner`）的锁定关系

### 2.2 核心价值

1. **可视化分配**：通过任务池 + 通讯录双栏视图，直观查看“哪些工作缺人”；支持从通讯录拖拽人员到任务卡片
2. **轻量交互**：任务卡片和人员条目均无需点选/复选框，直接拖拽完成分配
3. **来源一致**：年度计划派生任务的负责人保持与年度经营计划同步，避免双向修改导致数据不一致

### 2.3 功能边界

| 范围 | 说明 |
|------|------|
| **做** | 为重点工作（OMP 本地创建 + 年度计划派生）分配/移除**成员** |
| **不做** | 修改重点工作**负责人**（由年度经营计划模块负责）；替代预算/成本管理；替代项目级任务管理 |
| **限制** | 年度计划派生任务的负责人不可在 OMP 内移除或重新分配 |

## 三、用户角色与场景

### 3.1 角色定义

| 角色 | 职责 | 核心场景 |
|------|------|----------|
| **组织绩效管理员** | 统筹 OMP 重点工作的人员配置 | 为多个重点工作补充成员 |
| **部门负责人** | 管理本部门重点工作的执行团队 | 为本部门任务分配合适成员 |
| **高管/决策者** | 关注资源投入合理性 | 查看重点工作人员覆盖情况、识别资源瓶颈 |

### 3.2 用户故事

#### 故事1：组织绩效管理员为多个任务补充成员
```
作为 组织绩效管理员
我希望 通过拖拽快速为多个重点工作添加成员
以便 快速完成人员配置
```

#### 故事2：部门负责人直观分配成员
```
作为 部门负责人
我希望 通过拖拽把人员直接分配到重点工作
以便 快速完成执行团队组建
```

#### 故事3：同一人参与多个任务
```
作为 部门负责人
我希望 同一个人可以参与多个重点工作
以便 合理利用人力资源，不必重复添加同一人到不同任务
```

#### 故事4：保持年度计划负责人一致
```
作为 组织绩效管理员
我希望 年度计划派生任务的负责人只能随年度经营计划变更
以便 OMP 与年度经营计划保持来源一致，避免数据冲突
```

## 四、数据模型

### 4.1 重点工作实体扩展

在 OMP 重点工作 `OMP_KeyTask` 基础上，增加/明确以下字段：

```typescript
interface OMP_KeyTask {
  id: string;                          // 工作ID
  name: string;                        // 工作名称
  source: 'annual_plan' | 'omp';       // 来源：年度计划 / OMP本地创建
  annualPlanTaskId?: string;           // 来源年度计划重点工作ID（非空为派生任务）
  owner: string;                       // 负责人 PersonRef
  ownerDept?: string;                  // 负责人部门/团队
  members: string[];                   // 参与人员 PersonRef 数组（新增）
  dept: string;                        // 责任部门/团队
  type: 'improvement' | 'capability' | 'risk_response' | 'meeting_resolution';
  progress: number;                    // 0-100
  startDate: string;                   // YYYY-MM-DD
  endDate: string;                     // YYYY-MM-DD
  budget?: number;
  actualCost?: number;
  relatedKpiIds?: string[];
  cycleId?: string;                    // 战略周期ID
}
```

人员配置视图消费人员与组织目录的 `Employee` 数据：

interface Employee {

## 五、功能需求

### 5.1 任务池展示

#### FR-1 任务列表
- 显示当前战略周期内的重点工作
- 每条任务卡片展示：
  - 工作名称
  - 类型（业务改进/能力建设/风险应对/会议决议）
  - 责任部门
  - 截止日期
  - 负责人头像 + 姓名
  - 已有成员头像（最多展示3个，超出显示 `+N`）

#### FR-2 任务筛选
- 按部门筛选（动态取当前周期任务涉及的所有部门）
- 按类型筛选
- 按关键字搜索（工作名称、负责人姓名）

#### FR-3 任务卡片作为拖拽接收区
- 任务卡片本身不需要点选，也不需要复选框
- 任务卡片仅作为人员拖拽的放置（drop）目标
- 当人员条目被拖到某个任务卡片上并松开时，该人员被添加为该任务成员
- 由于当前任务数量不多，给多个任务分配同一人员时，通过重复拖拽实现

### 5.2 通讯录/资源人选

#### FR-4 组织架构树
- 基于人员与组织目录的组织架构树展示
- 支持展开/折叠组织节点
- 点击组织节点过滤右侧人员列表
- 显示每个组织节点下的人员数量

#### FR-5 人员搜索
- 支持按姓名、工号、部门搜索
- 搜索时优先于组织树过滤
- 支持一键清除搜索

#### FR-6 人员条目
- 人员按部门/团队组织，**名字挂在部门下边**
- 每条人员展示：头像、姓名、部门/团队
- **人员条目可长按/拖拽**：拖动到左侧任务卡片完成成员分配
- 拖拽时不从通讯录移除，人员条目仍保留在列表中
- 不需要复选框，也不依赖框选来触发分配

#### FR-7 人员排序
- 人员列表默认按姓名升序展示
- 智能推荐作为下一阶段功能，本期不显示推荐分、不做推荐排序

### 5.3 分配与移除

#### FR-8 添加成员
- **唯一主交互：拖拽人员条目到任务卡片**
  - 在右侧通讯录中点击并按住人员姓名/条目
  - 拖动到左侧目标任务卡片上方
  - 松开（drop）后，该人员即被添加为对应任务的成员
  - 通讯录中该人员不消失，可继续拖拽到其他任务
- 同一人可参与多个重点工作，系统不做上限限制
- 给多个任务添加同一人员时，直接重复拖拽即可

#### FR-9 拖拽反馈
- 拖拽人员条目时，条目半透明跟随鼠标移动
- 拖拽经过可接收的任务卡片时，任务卡片高亮（边框变为主色、背景变为浅色主色）
- 拖拽经过不可接收区域时，任务卡片恢复默认样式
- 释放后，若目标为有效任务卡片，则执行分配并显示成功提示

#### FR-10 移除成员
- 在任务卡片上直接点击已分配成员 chip 的 × 移除
- 移除后立即持久化并刷新视图

#### FR-11 分配冲突处理
- 若目标人员已是某任务的负责人，则跳过该任务，提示「该人员已是这些任务的负责人」
- 若目标人员已是某任务成员，则去重，不重复添加

### 5.4 年度计划派生任务约束

#### FR-12 负责人锁定
- 年度计划派生任务（`annualPlanTaskId` 非空）的负责人由年度经营计划同步
- OMP 人员配置台**不展示**年度计划派生任务的负责人修改入口
- 年度计划派生任务的成员可以正常配置

## 七、业务规则

### 7.1 分配规则

| 规则编号 | 规则内容 | 优先级 |
|----------|----------|--------|
| BR-1 | 一个重点工作可同时拥有 1 名负责人和 N 名成员 | P0 |
| BR-2 | 同一人不能同时作为同一任务的负责人和成员 | P0 |
| BR-3 | 成员分配以 PersonRef 为唯一标识 | P0 |
| BR-4 | 重复拖拽同一人员到同一任务时，已存在的人员自动去重 | P0 |
| BR-5 | 负责人为空的任务可在年度经营计划或 OMP 编辑弹窗中指定（不在人员配置台处理） | P0 |
| BR-6 | 从通讯录拖拽人员到任务后，通讯录中该人员不消失，可继续用于其他分配 | P0 |
| BR-7 | 同一人可参与多个重点工作，系统不做数量上限限制 | P0 |

### 7.2 年度计划派生任务规则

| 规则编号  | 规则内容                                 | 优先级 |
| ----- | ------------------------------------ | --- |
| BR-8  | 年度计划派生任务的 `owner` 由年度经营计划同步，OMP 只读   | P0  |
| BR-9  | 年度计划派生任务的 `members` 可在 OMP 人员配置台正常维护 | P0  |
| BR-10 | 年度经营计划修改负责人后，发布/保存时同步更新 OMP 派生任务     | P0  |
| BR-11 | 删除 OMP 派生任务不反向删除年度计划源头               | P1  |

### 7.3 数据一致性规则

| 规则编号  | 规则内容                                             | 优先级 |
| ----- | ------------------------------------------------ | --- |
| BR-12 | 人员配置变更后通过 API 保存到服务器端，刷新或切换页面后数据保持一致 | P0  |
| BR-13 | 当人员从人员与组织目录删除/禁用时，历史任务中仍保留其 PersonRef，仅渲染为“未知人员” | P1  |
| BR-14 | 跨周期任务仅在当前周期人员配置视图中可见                             | P0  |

## 八、非功能需求

### 8.1 性能

| 指标 | 要求 | 说明 |
|------|------|------|
| 人员配置视图首屏加载 | < 1.5s | 当前周期任务 + 人员数据 |
| 人员搜索响应 | < 200ms | 本地搜索，无需服务端 |
| 连续拖拽分配 | < 300ms/次 | 将同一人连续拖到多个任务，每次 drop 处理 ≤ 300ms |
| 人员列表渲染 | 默认 Top 80 | 超过 80 人时按姓名截断或分页，避免 DOM 过大 |

### 8.2 可用性

- 操作成功后显示 Toast 提示
- 操作失败时给出明确原因（如「该人员已是负责人，无法添加为成员」）
- 空状态友好提示：无任务、无人员、无搜索结果

### 8.3 兼容性

- 主题切换：使用 CSS 变量（`--primary`, `--success`, `--warning`, `--danger` 等）
- 响应式：移动端隐藏组织架构树，通讯录宽度 100%

### 8.4 安全

- 人员数据按组织架构隔离，用户只能查看有权限的组织/人员
- 所有操作记录变更到 `task.members`，支持审计

## 九、接口与依赖

### 9.1 依赖上游

| 来源模块 | 接口/函数 | 数据 | 说明 |
|----------|-----------|------|------|
| 人员与组织目录 | `getEmployees()` | 员工列表 | 获取全部可用人员 |
| 人员与组织目录 | `buildOrgHierarchy(employees)` | 组织架构树 | 渲染左侧组织树 |
| 人员与组织目录 | `searchEmployees(query, limit)` | 搜索结果 | 通讯录搜索 |
| 公共工具 | `window.renderPerson(personRef)` | 人员显示名 | 将 PersonRef 渲染为姓名 |
| 公共工具 | `window.renderAvatar(personRef)` | 头像 HTML | 人员头像渲染 |

### 9.2 数据持久化

成员配置变更通过服务端 API 保存，不再仅依赖浏览器 `localStorage`。

| 操作 | 方法 | 接口 | 请求体 | 说明 |
|------|------|------|--------|------|
| 查询任务列表 | GET | `/api/omp/tasks?cycleId={cycleId}` | - | 获取当前周期重点工作列表（含 `members`） |
| 添加/移除成员 | PATCH | `/api/omp/tasks/{taskId}/members` | `{ members: string[] }` | 全量更新任务成员列表 |
| 批量更新成员 | PATCH | `/api/omp/tasks/members/batch` | `[{ taskId, members }]` | 批量更新多个任务成员（可选） |

**关键规则：**
- 前端拖拽完成后立即调用 PATCH 接口保存
- 接口返回成功后刷新本地任务列表，保持前端状态与服务器一致
- 网络失败时显示错误提示，不自动回滚 UI，用户可重试
- 保留本地乐观更新以提升交互流畅度，但以服务端数据为最终一致来源

**本地缓存策略（仅作为降级）：**
- 服务端不可用时，可降级读取本地缓存，但保存操作仍需排队等待恢复后同步
- 本地缓存不作为主数据源，避免多设备/多用户场景下数据不一致

## 十一、风险与应对

| 风险 | 概率 | 影响 | 应对 |
|------|------|------|------|
| 人员数据量大导致渲染卡顿 | 中 | 中 | 默认按姓名排序展示，必要时分页或虚拟滚动 |
| 年度计划负责人同步延迟 | 低 | 高 | 年度规划保存/发布/初始化时强制同步到 OMP 派生任务 |
| 人员与组织目录字段变更 | 中 | 中 | 通过 `Employee` 接口抽象，隐藏内部字段差异 |
| 服务端保存失败 | 中 | 高 | 保存失败时明确提示用户，支持重试；本地保留乐观更新，待恢复后同步 |
| 用户误移除成员 | 中 | 中 | 移除操作即时生效但保留操作日志，支持审计追溯 |

## 十二、下一阶段计划

### 12.1 智能推荐（下一版本）

本期人员配置以「姓名排序 + 拖拽分配」为最小可用交互。下一版本引入智能推荐，帮助用户快速找到合适人员。

**推荐维度（暂定）**：

| 维度 | 权重 | 计算逻辑 |
|------|------|----------|
| 部门匹配 | 40% | 人员所属部门/团队与任务部门的匹配度 |
| 历史合作 | 30% | 人员曾负责/参与同部门、同类型任务的记录 |
| 当前负载 | 30% | 人员当前负责/参与的任务数量，避免过载 |

**表现形式**：
- 人员条目右侧显示「推荐分」
- 悬停或拖拽到任务卡片时，人员列表按与该任务的匹配度排序
- 高推荐人员视觉上突出显示

**前置条件**：
- 需要积累一定历史任务数据
- 需要人员与组织目录字段稳定
