## 一、背景与目标

### 1.1 背景

经营分析会是 DSTE 战略执行阶段最核心的闭环场景之一，覆盖：

- **会前**：议题收集、材料准备、议程编排
- **会中**：讨论辅助、数据查询、决议记录
- **会后**：纪要生成、行动项分发、督办跟踪
- **持续**：逾期预警、效果评估、知识沉淀

当前已落地部分单点 AI 能力，但尚未形成「感知-规划-工具-执行」的 Agent 闭环。

### 1.2 目标

1. 把经营分析会的 AI 能力从「零散工具」升级为「会议管理 Agent」。
2. 覆盖会前/会中/会后/督办全链路，实现至少 80% 高频场景 AI 辅助。
3. 建立人在回路机制：AI 生成草案，用户确认后再写入系统。
4. 与 RFC-008 的 **1+N+统一工具层** 架构保持一致。

### 3.1 会前：让会议更有目标

**场景 1：自动生成建议议程**

> 用户：帮我准备下周经营分析会

AI 执行：
1. `queryKpiStatus`：扫描当前周期 KPI，识别落后项
2. `queryOverdueActions`：查未闭环行动项
3. `queryOpenResolutions`：查待审批决议
4. `queryRelatedMeetings`：查历史同类型会议议程
5. 生成建议议程草案 → 用户确认 → 写入会议

**场景 2：会前准备度检查**

> 用户：这个会能开了吗？

AI 执行：
1. 检查议程是否为空
2. 检查材料审核是否通过
3. 检查决议/行动项是否已闭环
4. 输出准备度分数和卡点清单

### 3.2 会中：让讨论更有依据

**场景 3：自然语言查询数据**

> 用户：华东区回款本月完成了多少？

AI 执行：
1. `queryKpiStatus(kpiName="华东区回款")`
2. 返回实际值、目标值、完成率、趋势
3. 给出简要解读

**场景 4：补充 KMS 资料**

> 用户：这个议题有没有相关制度？

AI 执行：
1. `searchKms(query="回款管理制度")`
2. 返回相关文档片段和链接

**场景 5：快速创建行动项**

> 用户：这件事交给李四跟进，下周三前给我结果

AI 执行：
1. `createActionItem(content, owner="李四", deadline="下周三")`
2. 生成草案卡片 → 用户确认 → 写入会议

### 3.3 会后：让闭环更有效

**场景 6：从纪要提取决议和行动项**

> 用户：根据这份纪要生成决议和行动项

AI 执行：
1. 读取会议纪要文本
2. 提取决议 → `createResolution` 草案
3. 提取行动项 → `createActionItem` 草案
4. 批量展示 → 用户逐项确认 → 写入系统

**场景 7：自动生成会议纪要**

> 用户：生成一份会议纪要

AI 执行：
1. 汇总会议议程、讨论要点、决议、行动项
2. 生成结构化纪要文本
3. 用户编辑确认后可导出/分发

### 3.4 持续督办：让执行更有力

**场景 8：逾期行动项催办**

> 用户：帮我催一下逾期的行动项

AI 执行：
1. `queryOverdueActions`：查逾期行动项
2. 生成催办文案
3. `sendNotification`：推送到企业微信

**场景 9：主动洞察**

AI 周期性扫描：
- KPI 连续落后
- 行动项逾期
- 决议长期未闭环
- 会议材料未审核

在驾驶舱/会议列表展示洞察卡片：「华南区营收连续 2 周落后，建议纳入下周经分会」。

## 四、技术架构

### 4.1 整体架构

与 RFC-008 保持一致：**1 个中枢 Agent + N 个专家 Agent + 统一工具层**。

```
┌─────────────────────────────────────────┐
│           前端：会议 AI 助手             │
│     （侧边栏对话 + 议程推荐 + 洞察卡片）  │
└─────────────────────────────────────────┘
                    │
┌─────────────────────────────────────────┐
│        战略中枢 Agent（Orchestrator）     │
│  意图识别 / 任务分解 / Agent 路由 / 汇总   │
└─────────────────────────────────────────┘
                    │
        ┌───────────┼───────────┐
        ▼           ▼           ▼
   ┌─────────┐ ┌─────────┐ ┌─────────┐
   │ 会议管理 │ │ 经营分析 │ │ 风险预警 │
   │ Agent   │ │ Agent   │ │ Agent   │
   └────┬────┘ └────┬────┘ └────┬────┘
        └───────────┼───────────┘
                    ▼
        ┌─────────────────────┐
        │      统一工具层       │
        │ queryMeetingAgenda   │
        │ queryMeetingActions  │
        │ queryMeetingResolutions│
        │ queryKpiStatus       │
        │ createActionItem     │
        │ createResolution     │
        │ sendNotification     │
        │ searchKms            │
        │ generateMinutes      │
        └─────────────────────┘
```

### 4.2 会议管理 Agent 职责

当前最小闭环聚焦在「会议管理 Agent」，负责：

- 议程推荐
- 会中问答（查询议程/行动项/决议/KPI/KMS）
- 行动项/决议草案生成
- 会前准备度检查
- 会议纪要提取/生成

## 六、Phase 1 详细设计

### 6.1 工具执行迁到 Worker

当前问题：
- 工具在 `AIClient._executeTool` 里执行，依赖 `window.findMeetingById`
- 写入工具放在前端有安全风险
- 无法访问后端实时 KPI/BI 数据

方案：

```
前端 → /api/ai/agent (或 /api/ai/chat) → Worker
                                      ↓
                                   ToolExecutor
                                      ↓
                                   KV / BI / KMS
```

Worker 端 `ToolExecutor` 职责：
- 根据 tool name 调用对应 handler
- 校验用户权限（读/写）
- 记录审计日志
- 返回执行结果

### 6.2 createResolution 工具

参数：
```json
{
  "meetingId": "string",
  "content": "string",
  "owner": "string",
  "deadline": "YYYY-MM-DD"
}
```

返回：
```json
{
  "draft": true,
  "type": "resolution",
  "meetingId": "string",
  "resolution": { ... }
}
```

### 6.3 queryKpiStatus 工具

参数：
```json
{
  "kpiName": "string",
  "dept": "string",
  "period": "string"
}
```

返回：
```json
{
  "success": true,
  "kpi": {
    "name": "华东区回款",
    "target": 1000,
    "actual": 780,
    "completionRate": 0.78,
    "trend": "down",
    "period": "2026-06"
  }
}
```

### 6.4 checkMeetingReadiness 工具

返回：
```json
{
  "ready": false,
  "score": 65,
  "items": [
    { "name": "议程是否非空", "pass": true },
    { "name": "材料是否审核通过", "pass": false, "reason": "3 份材料未审核" },
    { "name": "上期决议是否闭环", "pass": false, "reason": "2 项决议 pending" }
  ]
}
```

### 6.5 会议纪要提取

输入：会议纪要的纯文本

输出：
```json
{
  "resolutions": [...],
  "actions": [...],
  "summary": "string"
}
```

前端展示草案卡片，用户逐项确认后写入。

## 八、风险与应对

| 风险 | 影响 | 应对 |
|---|---|---|
| AI 幻觉导致写入错误数据 | 高 | 所有写入类工具必须先返回草案，用户确认后再执行 |
| 工具调用延迟高 | 中 | 优先在 Worker 端执行；对慢查询加缓存 |
| KPI 数据不准确 | 高 | 工具返回必须标注数据来源和更新时间 |
| 用户不接受 AI 建议 | 中 | 保留手动入口，AI 建议以「草案」形式呈现，不强制 |
| 多 Agent 编排复杂度高 | 中 | Phase 1 只做一个会议管理 Agent，后续再拆分 |

## 九、下一步行动

建议从 **Phase 1 的「工具执行迁到 Worker」** 开始，原因：

1. 这是所有后续写入类工具的安全基础
2. 当前前端执行工具已经有功能验证，迁到 Worker 风险可控
3. 迁完后，`createResolution`、`sendNotification` 等工具可以统一在后端实现

具体下一步：

1. 在 `api-worker/worker.js` 新增 `ToolExecutor`
2. 把 `queryMeetingAgenda/Actions/Resolutions` 和 `createActionItem` 迁移到 Worker 执行
3. 前端 `AIClient._executeTool` 改为向 Worker 发起工具执行请求
4. 验证本地和生产环境
