## 一、产品定位与边界

### 1.1 一句话定位

**KPI 管理**是 DSTE 战略管理平台在战略执行阶段的「量化追踪引擎」。它把 ABP 年度经营计划发布的 KPI 目标基线转化为可录入、可计算、可追踪、可分析的实例数据，支撑组织绩效的偏差发现与改进闭环。

### 1.2 在 DSTE 流程中的位置

```
战略解码 (BP)
    │
    ▼ 发布并锁定
ABP 年度经营计划 ──────────────┐
    │                         │
    │ 只读同步                │ 目标修订后重新发布
    ▼                         ▼
OMP — KPI 管理              ABP
    │
    ├── 接收 KPI 目标基线（只读）
    ├── 录入/导入 actual
    ├── 计算达成率与状态
    ├── 识别偏差并输出预警
    └── 支撑经营分析会讨论
```

### 1.3 核心边界（做什么 / 不做什么）

| 维度 | KPI 管理做 ✅ | KPI 管理不做 ❌（由其他模块负责） |
|------|--------------|----------------------------------|
| **目标制定** | 接收 ABP 发布的 KPI 目标基线，只读引用 | 制定、分解、调整年度 KPI 目标值 |
| **目标更新** | ABP 重新发布后自动同步新版目标基线 | 在 OMP 内发起或审批目标变更 |
| **数据采集** | 手动录入、Excel 批量导入 | BI/ERP 系统自动采集（P1 再对接） |
| **数据分析** | 达成率、状态、同比/环比、趋势 | AI 根因分析、预测（P2 再接入 AI） |
| **偏差闭环** | 识别偏差，输出到预警中心和经营分析会 | 生成改进工作由重点工作管理负责 |
| **指标定义** | 维护企业级指标库（MVP 内简化版） | 战略解码层指标体系的建设逻辑 |

> **关键原则**: KPI 管理是 ABP 目标基线的「消费者」。目标基线的制定、修订、审批均在 ABP 完成；OMP 只接收已发布的目标基线。

## 二、用户角色与核心场景

### 2.1 角色定义

| 角色 | 核心诉求 | 高频场景 |
|------|---------|---------|
| **组织绩效管理员** | 统一指标口径、维护目标基线、审核数据 | 维护指标库、批量导入 actual、审核异常数据 |
| **部门负责人** | 看本部门 KPI 达成、更新数据、处理预警 | 部门 KPI 看板、手动录入 actual、查看趋势 |
| **高管/决策者** | 一眼看全局、快速定位落后部门 | 公司级 KPI 热力图、部门排名、预警清单 |
| **数据录入员** | 按要求定期录入 KPI 实际值 | 手动录入、Excel 导入、查看录入历史 |

### 2.2 核心场景

#### 场景 1：部门负责人查看本部门 KPI 达成
```
从侧边栏点击「执行 → 组织绩效管理 → KPI管理」
    │
    ▼
进入 cockpit.html#exe/kpi（renderKpiManagement 页面）
    │
    ▼
筛选「本部门」+「当前周期」
    │
    ▼
查看 KPI 列表：目标 / 实际 / 达成率 / 状态 / 趋势
    │
    ▼
点击 KPI 名称 → 详情弹窗（历史趋势 + 关联重点工作）
```

#### 场景 2：数据录入员更新 KPI 实际值
```
进入 KPI 管理 → 找到待录入 KPI
    │
    ▼
点击「编辑」→ 弹窗中修改实际值
    │
    ▼
系统自动计算达成率、状态、更新时间
    │
    ▼
保存后列表即时刷新，偏差 KPI 自动标记预警
```

#### 场景 3：部门负责人批量录入本季度 KPI 实际值
```
进入 cockpit.html#exe/kpi
    │
    ▼
筛选「本部门」+「当前周期」+「状态：未录入」
    │
    ▼
对未录入的 KPI 逐一点击「编辑」→ 录入 actual
    │
    ▼
或使用「📥 导入」批量上传 Excel
    │
    ▼
保存后系统自动计算达成率、标记状态、刷新列表
```

#### 场景 4：发现目标基线严重偏离实际
```
在 KPI 详情中发现目标基线严重偏离实际
    │
    ▼
记录偏差原因和证据（用于经营分析会讨论）
    │
    ▼
在经营分析会中讨论，必要时由 ABP 侧修订目标基线
    │
    ▼
ABP 重新发布后，OMP 自动同步新版目标基线
```

## 三、数据模型设计

### 3.1 核心实体关系

```
┌──────────────┐       ┌──────────────┐       ┌──────────────┐
│   指标定义    │◄─────│   KPI 实例    │◄─────│  KPI 数据记录 │
│  Indicator   │  1:N  │ KPIInstance  │  1:N  │  KPIData     │
└──────────────┘       └──────────────┘       └──────────────┘
```

### 3.2 指标定义 (OMP_Indicator)

企业级指标的标准定义，由绩效管理员维护。

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | string | `ind_时间戳` |
| `code` | string | 指标编码，唯一 |
| `name` | string | 指标名称，唯一 |
| `description` | string | 指标定义 |
| `formula` | string | 计算公式（文本说明） |
| `unit` | string | 单位（万、%、天、人） |
| `frequency` | enum | `monthly` / `quarterly` / `yearly` |
| `category` | string | BSC 维度：财务/客户/流程/学习成长 |
| `subCategory` | string | 二级分类 |
| `dataSource` | string | 数据来源 |
| `isPositive` | boolean | 是否正向指标（越高越好） |
| `status` | enum | `active` / `inactive`，被引用时不可删除 |
| `version` | number | 版本号，公式变更时升级 |
| `createdAt` / `updatedAt` | string | ISO 时间戳 |

**关键规则**:
- 指标名称和编码必须唯一
- 已被 KPI 实例引用的指标不可删除，只能置为 `inactive`
- BSC 四维度为系统内置分类，不可删除

### 3.3 KPI 实例 (OMP_KPIInstance)

ABP 发布后同步到 OMP 的执行期 KPI 目标基线，或在 OMP 内手工创建的执行期实例。

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | string | `kpi_时间戳` |
| `indicatorId` | string | 关联指标定义 ID |
| `annualPlanKpiId` | string\|null | 来源 ABP 的 KPI ID，非空表示来自年度计划 |
| `period` | string | 周期，如 `2026-Q2`、`2026` |
| `name` | string | 实例名称（默认继承指标名，可覆盖） |
| `targetValue` | number | 目标值 |
| `challengeValue` | number | 挑战值 |
| `actualValue` | number\|null | 实际值 |
| `achievementRate` | number\|null | 达成率（%），自动计算 |
| `status` | enum | `achieved` / `warning` / `lagging` / `unset`，自动计算 |
| `weight` | number | 权重（0-1） |
| `owner` | string | 责任人 |
| `dept` | string | 责任部门 |
| `frequency` | enum | 继承指标定义 |
| `parentInstanceId` | string\|null | 父 KPI 实例 ID，用于层级分解展示 |
| `createdAt` / `updatedAt` | string | ISO 时间戳 |

**达成率计算规则**:

| 指标类型 | 达成率公式 | 状态阈值（默认可配置） |
|----------|-----------|----------------------|
| 正向指标 | `actual / target × 100%` | ≥ 100%: achieved；≥ 80%: warning；< 80%: lagging |
| 负向指标 | `target / actual × 100%` | ≥ 100%: achieved；≥ 80%: warning；< 80%: lagging |
| 无 actual | — | `unset` |

> MVP 内仅支持正向/负向两类，暂不支持区间型、计数型等复杂计分规则。

### 3.4 KPI 数据记录 (OMP_KPIDataRecord)

单期 actual 的录入记录，支持修正留痕。

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | string | `kdr_时间戳` |
| `kpiInstanceId` | string | 关联 KPI 实例 |
| `period` | string | 数据周期 |
| `actualValue` | number | 实际值 |
| `targetValue` | number | 当期目标值（快照） |
| `achievementRate` | number | 当期达成率（快照） |
| `collectionType` | enum | `manual` / `import` |
| `inputBy` | string | 录入人 |
| `inputAt` | string | 录入时间 |
| `isCorrected` | boolean | 是否修正 |
| `originalValue` | number\|null | 修正前原值 |
| `correctionReason` | string\|null | 修正原因 |

**关键规则**:
- 同一 KPI 实例同一周期只有一条有效记录
- 修正必须保留原值和修正原因
- MVP 内暂不做审核流程，所有录入即时生效

## 五、页面结构与状态流转

### 5.1 页面入口

- **路由**: `cockpit.html#exe/kpi`
- **渲染函数**: `renderKpiManagement()`（位于 `cockpit.html` IIFE 内）
- **导航入口**: 侧边栏「执行 → 组织绩效管理 → KPI管理」
- **页面形态**: `cockpit.html` 内部独立 PAGES 路由，与 `exe/tasks`、`exe/meetings` 同级，不是任何页面的 Tab
- **当前代码位置**: `src/cockpit.html` 约第 3438 行 `function renderKpiManagement()`

**与 `exe/tasks` 的关系**:
- `exe/kpi`: 只关注 KPI 管理（指标库、KPI 目标基线、actual 录入、达成分析）
- `exe/tasks`: 组织绩效管理总览 + 重点工作管理 + 甘特图
- 两页面通过侧边栏并列切换，通过面包屑或链接互相跳转

│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

**指标卡设计规范（mini 版）**:

| 元素 | 说明 | 样式 |
|------|------|------|
| 指标名称 | 左对齐，最多一行，超出省略 | 15px / 600 weight / `var(--text-primary)` |
| 超链接标志 | 右上角 `[↗]` 或 `🔗` 图标 | 13px / `var(--primary)` / hover 加深 |
| 目标值 | 「目标」+ 数值 + 单位，紧凑排列 | 12px / `var(--text-secondary)` |
| 完成值 | 「实际」+ 数值 + 单位，紧凑排列 | 12px / `var(--text-secondary)` |
| 完成率 | 大字号，与状态徽章同行 | 16px / 600 weight |
| 状态徽章 | 🟢 达成 / 🟡 预警 / 🔴 落后 / ⚪ 未录入 | 12px / 状态色圆点 + 文字 |

**卡片尺寸**:
- 固定宽度：160~180px
- 固定高度：约 110px
- 内边距：12px
- 卡片间距：12px
- 圆角：8px
- 边框：1px solid `var(--border-light)`
- hover：轻微上浮 + 阴影加深

**交互规则**:
- 点击卡片主体 → 打开 KPI 详情弹窗（详情中展示部门、周期、趋势、关联工作等完整信息）
- 点击右上角 `[↗]` → 打开下级数据看板（MVP 内为详情弹窗的「趋势 + 关联工作」视图；P1 可扩展为独立 drill-down 页面）
- 点击顶部状态统计 pill（达成 / 预警 / 落后 / 未录入）→ 快速筛选对应状态的 KPI，再次点击取消筛选
- 拖拽 mini 指标卡 → 调整卡片在页面中的展示顺序（仅影响展示顺序，非业务优先级；排序为共享，非个人个性化）
- 未录入 actual 的卡片：完成率显示「—」，状态为 ⚪ 未录入
- 卡片按当前筛选/排序条件动态重排
- 响应式：大屏 5~6 列 / 中屏 4 列 / 小屏 3 列 / 移动端 2 列

> **说明**: KPI 总览看板（统计卡片、热力图、预警清单）保留在 `exe/tasks` 页面，由 `omp_renderDashboard()` 维护。`exe/kpi` 页面通过 mini 指标卡解决平铺列表看不清全局的问题。

### 5.3 多维视图设计

为解决「平铺列表看不清全局、指标堆叠无逻辑」的问题，`exe/kpi` 页面以 **mini 指标卡** 为基本展示单元，并提供 5 种视图模式组织这些卡片。

#### 5.3.1 视图模式总览

| 视图模式 | 展示方式 | 默认展开 | 适合人群 | 主要价值 |
|---------|---------|---------|---------|---------|
| **指标卡视图**（默认） | 卡片网格，无分组 | — | 管理层、部门负责人 | 一眼看到所有 KPI 核心状态 |
| **BSC 维度分组** | 卡片按 BSC 四维度分组 | 全部展开 | 管理层、绩效管理员 | 按战略维度看执行健康度 |
| **组织层级分组** | 卡片按公司→部门→团队分组 | 展开到部门 | 高管、部门负责人 | 看目标分解与责任归属 |

| **平铺列表** | 传统表格，无卡片 | — | 绩效管理员、数据录入员 | 精确筛选、批量操作、导出 |

```

### 5.9 删除规则

- **数据记录**: 单条可删除，删除后 KPI 实例 `actualValue` 和状态需重新计算

## 六、与周边模块的接口与边界

### 6.1 输入接口（上游依赖）

| 来源模块 | 数据 | 当前实现方式 | MVP 优先级 |
|----------|------|-------------|-----------|
| **ABP 年度经营计划** | KPI 目标基线（targetValue、challengeValue、weight、owner、dept） | 当前无 API，MVP 内支持手动创建 KPI 实例；P1 对接 ABP 发布事件，自动同步新版目标基线 | P1 |
| **指标库（PRD 模块2）** | 指标定义 | OMP 内自建简化指标库，P1 与指标库模块统一 | P1 |
| **人员与组织目录** | 责任人、部门数据 | 当前用字符串，P1 接入人员选择器 | P1 |

### 6.2 输出接口（下游消费）

| 目标模块 | 数据 | 触发条件 |
|----------|------|---------|
| **预警中心** | 偏差 KPI 事件（kpiInstanceId、偏差率、状态、时间） | actual 更新后达成率进入 warning/lagging |
| **经营分析会** | 绩效数据包（KPI 达成列表、预警清单、部门排名） | 会前自动生成或手动导出 |
| **重点工作管理** | 关联 KPI 列表、KPI 达成状态 | 创建/编辑重点工作时选择关联 KPI |
| **AI 助手** | KPI 数据、趋势、偏差信息 | AI 分析请求时 |

### 6.3 边界场景示例

**场景：KPI 落后，应该做什么？**

1. KPI 管理识别到 `lagging`
2. 输出预警事件到 **预警中心**（通知责任人）
3. 在 **经营分析会** 材料中自动标红该 KPI
4. 会议讨论后产生决议，转化为 **重点工作** 进行改进
5. 重点工作推进后，actual 更新，KPI 状态改善

**KPI 管理不做的事**:
- 不直接创建工作（由重点工作管理负责）
- 不直接发送通知（由预警中心/消息中心负责）
- 不制定新的年度目标（由 ABP 负责）

## 八、非功能需求

### 8.1 性能要求

| 指标 | 要求 | 说明 |
|------|------|------|
| KPI 列表加载 | < 500ms | 50 条以内 |
| 详情弹窗打开 | < 300ms | 含历史趋势数据 |
| Excel 导入 100 条 | < 3s | 含校验与保存 |
| 达成率计算 | 即时 | 前端本地计算 |

### 8.2 安全要求

- KPI 数据按部门隔离，用户只能查看本部门及授权部门数据
- ABP 来源的目标基线在 OMP 内只读，不可直接修改
- 数据修正保留原值和原因
- XSS 加固：所有用户输入字段（指标名、责任人等）渲染前 `escapeHtml`

### 8.3 可用性要求

- 未录入 actual 的 KPI 明确显示为「未录入」，避免与「落后」混淆
- 达成率、状态字段只读展示，不可手动修改
- 删除操作必须二次确认
- 导入失败时给出明确错误行号/原因

### 8.4 技术约束（继承项目规范）

- 代码位于 `src/cockpit.html` IIFE 内，所有函数加 `omp_` 前缀
- 页面通过 `PAGES['exe/kpi'] = renderKpiManagement` 注册为内部路由
- 事件处理优先使用 `data-action` 事件委托
- 弹窗内表单提交使用 `window.omp_xxx` 全局函数
- CSS 必须使用 `var(--primary)` 等变量，禁止硬编码颜色
- 数据持久化使用 localStorage，键名格式 `dste_omp_xxx_v1`
- 页面间跳转统一使用 `navigate('exe/xxx')`，由 cockpit 内部路由处理
