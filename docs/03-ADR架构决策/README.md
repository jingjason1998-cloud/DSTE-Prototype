# ADR 流程 — Architecture Decision Records

> 每个技术选型、架构变更、引入新依赖，决策时写 ADR。

---

## 什么是 ADR？

ADR 是**架构决策记录**——一个轻量级的文档，记录我们为什么做出某个技术决策。

它不长，通常 1-2 页，包含：
- 我们要做什么决定？
- 考虑了哪些选项？
- 选了哪个？为什么？
- 有什么后果（好的和坏的）？

---

## 什么时候写 ADR？

| 场景 | 需要 ADR？ |
|------|-----------|
| 引入新的构建工具 | ✅ 必须 |
| 更换前端框架 | ✅ 必须 |
| 引入新的测试工具 | ✅ 必须 |
| 修改部署架构 | ✅ 必须 |
| 修改代码规范 | ⚠️ 建议 |
| 升级依赖版本 | ❌ 不需要 |
| 重构函数/组件 | ❌ 不需要 |

---

## ADR 一旦写定，不要修改

ADR 是**历史记录**，不是活文档。如果决策变了：
1. 写一个新的 ADR 替代旧的
2. 在旧 ADR 顶部标注 `superseded by ADR-XXX`
