# 组织绩效管理模块 — 前端架构方案

> **版本**: v1.0 | **日期**: 2026-06-05 | **目标路径**: `cockpit.html#exe/tasks`
> **状态**: 方案设计，待确认后开发

---

## 一、现状分析

### 1.1 当前 cockpit.html 架构

```
cockpit.html (2320 行)
├── IIFE 全局包裹
│   ├── 配置层: TOP_NAV, SIDEBAR_CONFIG, PAGES, EXTERNAL_PAGES
│   ├── 状态层: currentPhase, currentPage
│   ├── 渲染层: renderDashboard, renderTasks, renderMeetings... (每个页面一个函数)
│   ├── 数据层: localStorage + API 兜底
│   ├── 事件层: bindPageEvents (data-action 事件委托) + window.xxx 全局函数
│   └── 工具层: escapeHtml, renderBreadcrumb, renderPlaceholder
```

### 1.2 现有 `renderTasks()` 的问题

当前 `exe/tasks` 页面只有 **40 行** 的静态占位代码：
- 静态表格，无交互
- 无数据持久化
- 无筛选/搜索/分页
- 无新建/编辑/详情功能
- 无 KPI 关联

### 1.3 核心约束（不可违背）

| # | 约束 | 说明 | 影响 |
|---|------|------|------|
| 1 | **IIFE 内运行** | 所有代码在 IIFE 中，局部函数无法被 onclick 访问 | 必须用 `window.fnName` 暴露或事件委托 |
| 2 | **事件委托优先** | 渲染用 `data-action`，init 中绑定事件监听 | 避免内联 onclick |
| 3 | **CSS 变量** | 必须用 `var(--primary)` 等，禁止硬编码颜色 | 主题切换兼容 |
| 4 | **localStorage 兜底** | 先存 localStorage，有 API 时再同步 | 数据不丢失 |
| 5 | **PAGES 对象模式** | 每个页面一个 `renderXxx()` 函数返回 HTML 字符串 | 遵循现有模式 |

## 二、架构方案选型

### 方案对比

| 维度 | 方案A: 单文件扩展（推荐） | 方案B: 独立 HTML 页面 | 方案C: 模块拆分为 JS 文件 |
|------|------------------------|---------------------|------------------------|
| **实现方式** | 在 cockpit.html 内扩展 `renderTasks()` | 新建 `omp.html`，配置 EXTERNAL_PAGES | 将 renderTasks 拆分到 `src/pages/omp/main.js` |
| **代码位置** | cockpit.html 内（预计 +800~1200 行） | 独立文件 | 外部 JS + cockpit.html 引用 |
| **导航集成** | ✅ 天然集成，无需改配置 | ✅ 需改 EXTERNAL_PAGES | ✅ 需改 PAGES 对象动态加载 |
| **代码隔离** | ⚠️ 共享 IIFE 命名空间 | ✅ 完全隔离 | ✅ 模块级隔离 |
| **构建复杂度** | ✅ 无需改构建 | ✅ 无需改构建 | ⚠️ 需处理 JS 模块加载 |
| **cockpit 膨胀** | ⚠️ 文件变大（2320 → ~3500 行） | ✅ cockpit 不变 | ✅ cockpit 不变 |
| **现有模式一致性** | ✅ 完全符合 | ✅ 有先例（meetings.html） | ⚠️ 新模式 |
| **开发效率** | ✅ 高，无需处理跨文件 | ✅ 高 | ⚠️ 中，需处理模块通信 |

### 推荐方案：方案A（单文件扩展）

**理由：**
1. `exe/tasks` 已在 `PAGES` 对象中注册，天然集成
2. 组织绩效管理模块**需要与 cockpit 共享主题、用户状态、导航状态**，独立页面会增加通信成本
3. 现有 `renderInsightsTopics` 已证明单文件内可以实现复杂 CRUD（600+ 行），模式成熟
4. 无需改 vite.config.js、config.js 等任何配置
5. 开发周期最短，风险最低

**风险缓解：**
- 代码量大 → 通过**严格分层**和**注释分块**管理复杂度
- 命名冲突 → 所有函数加 `omp_` 前缀
- 文件膨胀 → 未来如需可再迁移到独立页面（届时已有完整功能作为迁移基线）

### 3.4 事件处理方案

**采用「事件委托 + window 全局函数」混合模式：**

```javascript
// 1. 列表行点击、按钮点击 → 事件委托（data-action）
// 渲染时
`<button data-action="edit-kpi" data-id="${kpi.id}">编辑</button>`

// 绑定在 renderTasks 返回的容器上
container.addEventListener('click', (e) => {
  const btn = e.target.closest('[data-action]');
  if (!btn) return;
  const action = btn.dataset.action;
  const id = btn.dataset.id;
  
  switch(action) {
    case 'edit-kpi': omp_openKpiEditModal(id); break;
    case 'view-task': omp_openTaskDetailModal(id); break;
    case 'switch-tab': omp_switchTab(btn.dataset.tab); break;
    // ...
  }
});

// 2. 弹窗内的表单提交、动态添加 → window 全局函数
// 因为弹窗是动态插入 body 的，不在容器内
window.omp_saveKpi = function() { ... };
window.omp_addMilestoneRow = function() { ... };
window.omp_confirmDeleteTask = function(id) { ... };
```

## 六、关键技术决策

### 6.1 甘特图实现

**方案**: 纯 CSS Grid 实现，不用 SVG/Canvas

```css
/* 甘特图行 */
.gantt-row {
  display: grid;
  grid-template-columns: 200px repeat(12, 1fr); /* 工作名 + 12个月 */
  gap: 1px;
}
/* 进度条用绝对定位的 div */
.gantt-bar {
  position: relative;
  height: 24px;
  border-radius: 4px;
  background: var(--primary);
}
```

**理由**: SVG 需要额外学习成本和维护，CSS Grid 足够表达简单的甘特图，且与现有代码风格一致。

### 6.2 趋势图实现

**方案**: 纯 CSS 迷你折线图（用 div 的 border-bottom 模拟）

```html
<!-- 用一系列 div 的高度模拟折线 -->
<div style="display:flex;align-items:flex-end;gap:2px;height:40px;">
  <div style="width:4px;height:60%;background:var(--primary);"></div>
  <div style="width:4px;height:70%;background:var(--primary);"></div>
  <div style="width:4px;height:55%;background:var(--primary);"></div>
  <div style="width:4px;height:80%;background:var(--primary);"></div>
</div>
```

**理由**: 不引入 ECharts/Chart.js 等图表库，保持项目轻量。迷你图仅用于视觉提示，无需精确交互。

### 6.3 弹窗实现

**方案**: 与 `renderInsightsTopics` 一致，动态创建 overlay + modal div，插入 body

```javascript
function omp_openModal(title, contentHtml) {
  const overlay = document.createElement('div');
  overlay.className = 'omp-modal-overlay';
  overlay.innerHTML = `<div class="omp-modal">...</div>`;
  document.body.appendChild(overlay);
  // 点击遮罩关闭
  overlay.addEventListener('click', (e) => { if (e.target === overlay) overlay.remove(); });
}
```

### 6.4 数据持久化

```javascript
const OMP_STORAGE_KEYS = {
  indicators: 'dste_omp_indicators_v1',
  kpiInstances: 'dste_omp_kpi_instances_v1',
  tasks: 'dste_omp_tasks_v1',
  milestones: 'dste_omp_milestones_v1',
  progressRecords: 'dste_omp_progress_v1',
};

function omp_loadData(key) {
  try {
    const raw = localStorage.getItem(OMP_STORAGE_KEYS[key]);
    return raw ? JSON.parse(raw) : null;
  } catch(e) { return null; }
}

function omp_saveData(key, data) {
  try {
    localStorage.setItem(OMP_STORAGE_KEYS[key], JSON.stringify(data));
  } catch(e) { console.warn('save failed:', e); }
}
```

## 八、风险与应对

| 风险 | 概率 | 影响 | 应对 |
|------|------|------|------|
| cockpit.html 过大导致维护困难 | 中 | 中 | 严格函数命名前缀 + 代码分块注释；未来可迁移到独立页面 |
| localStorage 5MB 限制 | 低 | 中 | 数据量预计 < 100KB，远未达限制；超限时清理历史记录 |
| 性能问题（大量 DOM 操作） | 低 | 低 | 列表用虚拟滚动（如 >50 条）；目前 mock 数据仅 8+6 条 |
| 命名冲突 | 低 | 高 | 所有函数/变量加 `omp_` 前缀；状态挂 `window._ompState` |
