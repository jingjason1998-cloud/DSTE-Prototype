# ADR-003: 微前端架构（预留）

> 状态：`accepted` | 日期：2026-05-21

---

## 上下文

项目需要支持多 AI 并行开发不同功能模块。每个 AI 可能独立开发页面，需要避免代码冲突和重复。

## 决策

采用**轻量级微前端接口**（`window.DSTEPage`），而非完整的微前端框架（如 qiankun、Module Federation）。

## 考虑的选项

### 选项 A：完整微前端框架（qiankun / Module Federation）

- ✅ 完整的隔离、通信、生命周期管理
- ❌ 引入复杂框架，与当前纯 HTML 架构不匹配
- ❌ 学习成本高
- ❌ 过度设计（当前只有 4 个页面）

### 选项 B：轻量级接口约定（推荐）

- ✅ 零依赖，纯 JS 对象约定
- ✅ AI 容易理解和实现
- ✅ 渐进式——现在用函数，未来可升级
- ⚠️ 没有真正的运行时隔离

### 选项 C：保持现状（内联渲染函数）

- ✅ 最简单
- ❌ 所有代码挤在一个文件，难以并行开发
- ❌ 没有模块边界

## 为什么选轻量级接口？

1. **项目规模不需要完整微前端框架**
2. **AI 协作需要清晰的模块边界**，但不需要运行时隔离
3. **为未来升级留余地**——接口定好后，底层实现可以替换
4. **现在阶段**：渲染函数仍在 cockpit.html 内，但接口已定义

## 当前实现

```javascript
// 接口定义
window.DSTEPage = {
  name: '',
  phase: '',
  breadcrumb: [],
  init(container) {},
  destroy() {}
};

// 当前实际用法（过渡阶段）
const PAGES = {
  'exe/meetings': renderMeetings,  // 返回 HTML 字符串
};
```

## 演进路径

```
现在：PAGES{} 内联渲染函数
  ↓
短期：render 函数提取到独立文件
  ↓
中期：独立文件暴露 window.DSTEPage
  ↓
长期：需要时引入完整微前端框架
```

## 后果

### 积极的

- 清晰的模块边界
- 支持多 AI 并行开发
- 未来可升级

### 消极的

- 当前未完全解耦（仍在过渡）
- 需要手动管理全局命名空间
