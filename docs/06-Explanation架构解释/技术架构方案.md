## 一、技术架构总览

### 1.1 架构原则

| 原则        | 说明                   |
| --------- | -------------------- |
| **微服务化**  | 按业务域拆分服务，独立部署、独立扩展   |
| **前后端分离** | 前端专注交互，后端专注业务逻辑和数据   |
| **数据驱动**  | 与帆软 BI 深度集成，支持多源数据接入 |
| **云原生**   | 容器化部署，支持弹性伸缩         |
| **安全优先**  | 零信任架构，数据分级保护         |

## 四、核心数据模型

### 4.1 数据库设计原则

- **分库分表**: 按租户隔离，核心表按租户ID分表
- **读写分离**: 报表查询走只读副本
- **数据归档**: 历史数据定期归档到 ClickHouse
- **索引策略**: 高频查询字段建联合索引

### 4.2 核心表结构

#### 租户与组织 (多租户基础)

```sql
-- 租户表
create table tenant (
    tenant_id       bigint primary key auto_increment,
    tenant_name     varchar(100) not null,
    tenant_code     varchar(50) not null unique,
    status          tinyint default 1 comment '0-禁用 1-启用',
    expire_time     datetime,
    created_at      datetime default current_timestamp,
    updated_at      datetime default current_timestamp on update current_timestamp,
    index idx_code (tenant_code)
) engine=innodb default charset=utf8mb4;

-- 组织架构表
create table organization (
    org_id          bigint primary key auto_increment,
    tenant_id       bigint not null,
    parent_id       bigint default 0 comment '0-根节点',
    org_name        varchar(100) not null,
    org_code        varchar(50) not null,
    org_type        tinyint comment '1-公司 2-部门 3-团队',
    level           int default 1,
    path            varchar(500) comment '层级路径，如 /1/2/5/',
    leader_id       bigint comment '负责人ID',
    sort_order      int default 0,
    status          tinyint default 1,
    created_at      datetime default current_timestamp,
    updated_at      datetime default current_timestamp on update current_timestamp,
    unique key uk_tenant_code (tenant_id, org_code),
    index idx_parent (tenant_id, parent_id),
    index idx_path (tenant_id, path)
) engine=innodb default charset=utf8mb4;
```

## 七、安全架构

### 7.1 认证鉴权

```
用户登录 → SSO/LDAP → 颁发 JWT Token → 请求携带 Token
                                              ↓
                                        API Gateway 验签
                                              ↓
                                        RBAC 权限校验
                                              ↓
                                        数据权限过滤 (按组织)
                                              ↓
                                        业务处理
```

### 7.2 数据安全

| 层级 | 措施 |
|------|------|
| **传输** | HTTPS/TLS 1.3，敏感字段额外加密 |
| **存储** | 敏感数据 AES-256 加密，密钥托管 KMS |
| **访问** | 行级权限控制，数据脱敏展示 |
| **审计** | 全量操作日志，保留 180 天 |
| **备份** | 每日全量 + 实时增量，跨地域备份 |

### 7.3 多租户隔离

- **数据库层**: 租户 ID 字段隔离 + 索引优化
- **应用层**: 请求上下文注入租户 ID，自动过滤
- **缓存层**: 租户前缀隔离 (tenant:{id}:key)
- **存储层**: 租户独立 Bucket/目录

## 八、性能优化策略

### 8.1 缓存策略

| 层级 | 技术 | 场景 |
|------|------|------|
| **CDN** | 阿里云/腾讯云 CDN | 静态资源、报表文件 |
| **应用缓存** | Caffeine | 字典数据、配置 |
| **分布式缓存** | Redis | 用户会话、热点 KPI、驾驶舱 |
| **数据库缓存** | MySQL Query Cache | 重复查询 |

### 8.2 数据库优化

- **读写分离**: 报表查询走只读副本
- **分库分表**: 按租户分表，kpi_data 按时间分区
- **索引优化**: 高频查询字段联合索引
- **慢查询治理**: 自动捕获，定期优化

### 8.3 前端优化

- **懒加载**: 战略地图节点按需渲染
- **虚拟滚动**: 长列表性能优化
- **数据预取**: 驾驶舱数据后台预计算
- **组件缓存**: React.memo + useMemo

| **基础设施** | Prometheus + Grafana | CPU/内存/磁盘/网络 |
| **应用性能** | APM (SkyWalking) | 响应时间/吞吐量/错误率 |
| **业务指标** | 自定义 Dashboard | KPI 更新率/会议召开率 |
| **日志** | ELK Stack | 错误日志/审计日志 |
| **链路** | Jaeger | 分布式调用链 |

### 9.2 告警策略

| 级别 | 条件 | 通知方式 |
|------|------|---------|
| **P0-紧急** | 服务不可用/数据丢失 | 电话 + 短信 + 钉钉 |
| **P1-高** | 响应时间 > 5s / 错误率 > 5% | 短信 + 钉钉 |
| **P2-中** | 响应时间 > 2s / 错误率 > 1% | 钉钉 |
| **P3-低** | 磁盘 > 80% / 内存 > 85% | 钉钉 |
