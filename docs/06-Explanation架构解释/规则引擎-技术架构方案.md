## 一、技术架构总览

### 1.1 架构原则

1. **计算与存储分离**: 规则定义存储与规则计算执行分离，支持独立扩展
2. **配置即代码**: 规则内容以结构化JSON存储，支持版本控制和Diff对比
3. **热更新支持**: 规则变更无需重启服务，实时生效
4. **多租户隔离**: 支持不同组织使用独立规则集，数据严格隔离
5. **可观测性**: 规则执行全链路可追踪，计算结果可审计

## 三、服务拆分设计

### 3.1 服务划分

| 服务名 | 职责 | 端口 | 数据库 |
|--------|------|------|--------|
| **rule-config-service** | 规则配置管理（CRUD、版本、审批） | 8001 | MySQL |
| **rule-engine-service** | 规则计算执行（公式解析、计算、缓存） | 8002 | Redis |
| **rule-monitor-service** | 规则监控（日志、影响分析、预警） | 8003 | ES + MySQL |

### 3.2 服务依赖关系

```
rule-config-service
    ├── 依赖: MySQL (规则定义/版本/审批)
    ├── 依赖: Redis (缓存热点规则)
    ├── 依赖: RabbitMQ (发布规则变更事件)
    └── 被依赖: rule-engine-service (获取已发布规则)

rule-engine-service
    ├── 依赖: rule-config-service (获取规则定义)
    ├── 依赖: Redis (计算结果缓存)
    ├── 依赖: RabbitMQ (接收计算任务)
    └── 被依赖: DSTE业务模块 (预算/绩效等)

rule-monitor-service
    ├── 依赖: MySQL (执行日志)
    ├── 依赖: Elasticsearch (日志检索)
    ├── 依赖: RabbitMQ (接收执行事件)
    └── 被依赖: 前端 (监控大盘)
```

## 四、核心数据模型

### 4.1 数据库设计

#### 规则定义表

```sql
CREATE TABLE rule_definition (
    rule_id         BIGINT PRIMARY KEY AUTO_INCREMENT COMMENT '规则ID',
    rule_code       VARCHAR(64) NOT NULL UNIQUE COMMENT '规则编码',
    rule_name       VARCHAR(128) NOT NULL COMMENT '规则名称',
    category_id     BIGINT NOT NULL COMMENT '分类ID',
    rule_type       VARCHAR(32) NOT NULL COMMENT '规则类型',
    sub_type        VARCHAR(32) COMMENT '子类型',
    current_version VARCHAR(16) DEFAULT '1.0.0' COMMENT '当前版本',
    status          VARCHAR(16) DEFAULT 'DRAFT' COMMENT '状态',
    owner           VARCHAR(64) COMMENT '负责人',
    effective_date  DATE COMMENT '生效日期',
    expiry_date     DATE COMMENT '失效日期',
    description     TEXT COMMENT '说明',
    org_id          BIGINT COMMENT '组织ID',
    created_by      VARCHAR(64) NOT NULL,
    created_at      DATETIME DEFAULT CURRENT_TIMESTAMP,
    updated_by      VARCHAR(64),
    updated_at      DATETIME ON UPDATE CURRENT_TIMESTAMP,
    
    INDEX idx_category (category_id),
    INDEX idx_type (rule_type, sub_type),
    INDEX idx_status (status),
    INDEX idx_org (org_id)
) COMMENT='规则定义表';
```

#### 规则版本表

```sql
CREATE TABLE rule_version (
    version_id      BIGINT PRIMARY KEY AUTO_INCREMENT,
    rule_id         BIGINT NOT NULL,
    version_no      VARCHAR(16) NOT NULL,
    content_json    JSON NOT NULL COMMENT '规则内容JSON',
    change_log      TEXT COMMENT '变更说明',
    status          VARCHAR(16) DEFAULT 'DRAFT',
    published_at    DATETIME,
    published_by    VARCHAR(64),
    created_by      VARCHAR(64) NOT NULL,
    created_at      DATETIME DEFAULT CURRENT_TIMESTAMP,
    
    UNIQUE KEY uk_rule_version (rule_id, version_no),
    INDEX idx_rule (rule_id)
) COMMENT='规则版本表';
```

#### 规则分类表

```sql
CREATE TABLE rule_category (
    category_id     BIGINT PRIMARY KEY AUTO_INCREMENT,
    parent_id       BIGINT DEFAULT 0 COMMENT '父分类ID，0为根',
    category_code   VARCHAR(64) NOT NULL UNIQUE,
    category_name   VARCHAR(128) NOT NULL,
    category_type   VARCHAR(32) NOT NULL COMMENT 'MANAGEMENT/INCENTIVE',
    sort_order      INT DEFAULT 0,
    status          VARCHAR(16) DEFAULT 'ACTIVE',
    org_id          BIGINT,
    created_by      VARCHAR(64) NOT NULL,
    created_at      DATETIME DEFAULT CURRENT_TIMESTAMP,
    
    INDEX idx_parent (parent_id),
    INDEX idx_type (category_type)
) COMMENT='规则分类表';
```

#### 规则审批表

```sql
CREATE TABLE rule_approval (
    approval_id     BIGINT PRIMARY KEY AUTO_INCREMENT,
    rule_id         BIGINT NOT NULL,
    version_no      VARCHAR(16) NOT NULL,
    submitter       VARCHAR(64) NOT NULL,
    submitted_at    DATETIME NOT NULL,
    approver        VARCHAR(64),
    approved_at     DATETIME,
    status          VARCHAR(16) DEFAULT 'PENDING' COMMENT 'PENDING/APPROVED/REJECTED',
    comment         TEXT,
    
    INDEX idx_rule (rule_id),
    INDEX idx_status (status)
) COMMENT='规则审批表';
```

#### 规则执行日志表

```sql
CREATE TABLE rule_execution_log (
    log_id          BIGINT PRIMARY KEY AUTO_INCREMENT,
    rule_id         BIGINT NOT NULL,
    version_no      VARCHAR(16) NOT NULL,
    trigger_type    VARCHAR(32) NOT NULL,
    trigger_source  VARCHAR(128),
    biz_key         VARCHAR(128),
    input_params    JSON,
    output_result   JSON,
    exec_duration   INT,
    status          VARCHAR(16) DEFAULT 'SUCCESS',
    error_msg       TEXT,
    created_at      DATETIME DEFAULT CURRENT_TIMESTAMP,
    
    INDEX idx_rule (rule_id),
    INDEX idx_biz (biz_key),
    INDEX idx_time (created_at)
) COMMENT='规则执行日志表';
```

#### 规则参数表

```sql
CREATE TABLE rule_parameter (
    param_id        BIGINT PRIMARY KEY AUTO_INCREMENT,
    param_code      VARCHAR(64) NOT NULL,
    param_name      VARCHAR(128) NOT NULL,
    param_type      VARCHAR(32) NOT NULL COMMENT 'STRING/NUMBER/DATE/BOOLEAN/JSON',
    default_value   VARCHAR(256),
    value_range     VARCHAR(256) COMMENT '取值范围，JSON格式',
    description     TEXT,
    source_type     VARCHAR(32) COMMENT 'MANUAL/AUTO/EXTERNAL',
    source_config   JSON COMMENT '数据来源配置',
    org_id          BIGINT,
    created_by      VARCHAR(64) NOT NULL,
    created_at      DATETIME DEFAULT CURRENT_TIMESTAMP,
    
    UNIQUE KEY uk_param_code (param_code, org_id),
    INDEX idx_org (org_id)
) COMMENT='规则参数表';
```


### 6.2 公式解析器设计

**使用 AviatorScript 作为表达式引擎**

```java
@Service
public class RuleCalculationService {
    
    @Autowired
    private RuleConfigClient ruleConfigClient;
    
    @Autowired
    private ParameterResolver parameterResolver;
    
    public CalculationResult calculate(CalculationRequest request) {
        // 1. 获取规则定义
        RuleDefinition rule = ruleConfigClient.getRule(
            request.getRuleCode(), 
            request.getVersionNo()
        );
        
        // 2. 解析公式
        String expression = rule.getFormula().getExpression();
        Expression compiledExpr = AviatorEvaluator.compile(expression);
        
        // 3. 获取参数值
        Map<String, Object> env = new HashMap<>();
        for (Variable var : rule.getFormula().getVariables()) {
            Object value = parameterResolver.resolve(var, request.getContext());
            env.put(var.getName(), value);
        }
        // 合并手动输入参数
        env.putAll(request.getParameters());
        
        // 4. 执行计算
        long startTime = System.currentTimeMillis();
        Object result = compiledExpr.execute(env);
        long duration = System.currentTimeMillis() - startTime;
        
        // 5. 生成计算明细
        CalculationDetail detail = buildCalculationDetail(
            rule, env, result, duration
        );
        
        // 6. 保存执行日志
        saveExecutionLog(rule, request, result, duration);
        
        return CalculationResult.builder()
            .ruleCode(rule.getRuleCode())
            .versionNo(rule.getVersionNo())
            .result(result)
            .detail(detail)
            .duration(duration)
            .build();
    }
}
```

### 6.3 参数解析器设计

```java
@Component
public class ParameterResolver {
    
    @Autowired
    private DataSourceClient dataSourceClient;
    
    @Autowired
    private RuleConfigClient ruleConfigClient;
    
    public Object resolve(Variable variable, Map<String, Object> context) {
        switch (variable.getSource().getType()) {
            case DATABASE:
                return resolveFromDatabase(variable, context);
            case PARAMETER:
                return resolveFromParameter(variable);
            case RULE_REFERENCE:
                return resolveFromRuleReference(variable, context);
            case MANUAL_INPUT:
                return variable.getDefaultValue();
            case EXTERNAL_API:
                return resolveFromExternalApi(variable, context);
            default:
                throw new UnsupportedOperationException("Unknown source type");
        }
    }
    
    private Object resolveFromDatabase(Variable variable, Map<String, Object> context) {
        DataSourceConfig config = variable.getSource();
        // 构建动态SQL查询
        String sql = buildDynamicSql(config, context);
        return dataSourceClient.queryForObject(sql);
    }
    
    private Object resolveFromRuleReference(Variable variable, Map<String, Object> context) {
        String refRuleCode = variable.getSource().getRuleCode();
        // 递归调用规则计算
        CalculationResult refResult = calculate(
            CalculationRequest.builder()
                .ruleCode(refRuleCode)
                .context(context)
                .build()
        );
        return refResult.getResult();
    }
}
```

## 八、安全架构

### 8.1 认证鉴权

- **认证**: JWT Token，统一由DSTE平台认证中心颁发
- **鉴权**: RBAC权限模型，规则编辑权限按角色+组织维度控制
- **规则隔离**: 多租户数据隔离，org_id字段隔离不同组织规则

### 8.2 数据安全

- 规则内容JSON加密存储（AES-256）
- 敏感参数（如提成比例）单独加密字段存储
- 执行日志脱敏处理（隐藏敏感参数值）

### 8.3 审计日志

- 规则变更记录：谁、何时、改了什么（变更前后值）
- 规则执行记录：何时、何地、何规则、输入输出
- 审批记录：提交人、审批人、审批意见、时间戳

## 九、性能优化策略

### 9.1 缓存策略

| 缓存对象 | 缓存位置 | 过期策略 | 说明 |
|---------|---------|---------|------|
| 规则定义 | Redis | 5分钟 | 热点规则缓存 |
| 编译后表达式 | Caffeine(本地) | 10分钟 | 避免重复编译 |
| 计算结果 | Redis | 1小时 | 相同输入直接返回 |
| 参数值 | Redis | 视参数类型 | 数据库查询结果缓存 |

### 9.2 数据库优化

- 规则定义表按org_id分区
- 执行日志表按时间分表（每月一张）
- 常用查询字段建立复合索引

### 9.3 计算优化

- 表达式预编译，避免运行时编译开销
- 批量计算时并行处理
- 复杂计算异步执行，通过消息队列削峰
