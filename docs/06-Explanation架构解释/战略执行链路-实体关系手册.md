# 战略执行链路 — 实体关系手册

> 版本: v1.0（2026-07-21）
> 定位: 全系统"战略执行"数据链路的单一事实来源。修改任何实体的关联字段时，必须同步更新本手册。
> 维护方式: Docs as Code，与代码同 PR 更新。

## 1. 执行逻辑模型

### 1.1 主链（下行：分解）

```
BP 年度经营计划 → 重点工作 → 会议（议程） → 决议 → 行动项 → 闭环
```

- **决议 ≠ 待办**。决议（decision）是会上"决定做什么"；行动项（action）是决议拆出的执行单元，1:N。
- **BP 不是独立实体**。代码中不存在"BP 计划"这张表，年度经营计划 = Cycle（`dste_cycles_v1`）+ `source='annual_plan'` 的 KPI 实例与重点工作构成的**逻辑视图**（`src/cockpit.html:2635`）。

### 1.2 回环（上行：收敛）

```
行动项闭环 → 决议进度聚合 → 重点工作进度 → KPI 达成 → BP 达成度
```

回环目前**只有"决议←行动项"一段已通**（`computeResolutionProgress`，`resolution-helpers.js:179-190`）；行动项 → 重点工作、重点工作 ← 会议执行结果 均未实现，是本手册记录的待补能力。

### 1.3 链路健康度总览

| 链段 | 关联机制 | 状态 |
|---|---|---|
| BP → 重点工作 | `cycleId` + `source`/`annualPlanTaskId` 派生 | ✅ 已通 |
| 重点工作 → 议程 | `agenda.sourceTaskId`/`sourceTaskName`（AI 推荐回写 + 编辑器手选，v6 迁移） | ✅ 已通 |
| 议程 → 决议 | `decision.sourceTopicId` | ✅ 已通 |
| 决议 → 行动项 | 双向：`decision.actions[]` + `action.sourceDecisionId` | ⚠️ 已通，双写不一致风险（§4.2） |
| 行动项 → 议程 | `action.sourceAgendaId` | ✅ 已通 |
| 议程 → 议程（顺延） | `carriedFromAgendaId`/`postponedHistory` | ✅ 已通 |
| 会议 → 会议（会议链） | `upstreamMeeting`/`downstreamMeeting` | ✅ 已通 |
| 战略地图 → KPI/重点工作 | `kpiRef`/`taskRef` 预留，种子与创建路径全为 `null` | ❌ 悬空 |
| 重点工作 → 会议 | `type='meeting_resolution'` 仅分类标签，无 ID 引用 | ❌ 假关联 |

## 2. L1 实体数据字典

| 字段 | 说明 |
|---|---|

- 存储：`dste_omp_tasks_v1`；API `/api/omp/tasks`
- 出处：`src/cockpit.html:1031-1064`；新增表单 `ap_saveKeyTask`（`:3234-3256`）

| 字段 | 说明 |
|---|---|

| `source` | `annual_plan` / `omp` |

## 4. 已知断点与风险

### 4.1 断点：重点工作 → 议程（✅ 已修复）

~~AI 议程推荐从 `dste_omp_tasks_v1` 只读拉取重点工作作为上下文，但 `candidateToAgendaItem` 生成议程时不回写重点工作 ID。~~

**修复方式**（meetings v6）：议程新增 `sourceTaskId`/`sourceTaskName` 单向引用字段（仿 `sourceMeetingId`/`sourceMeetingTitle` 冗余惯例）。AI 推荐由 `candidateToAgendaItem` 对 `sourceType='key_work'` 候选回写任务 id/名称；会议编辑器议程行新增"关联重点工作"下拉支持手选/留空；v6 迁移为存量议程补 `null`。重点工作侧无反向字段，删除议程无需级联清理；顺延深拷贝自然携带。详情页议程行显示关联徽章并跳转 `cockpit.html#exe/tasks`。

### 4.2 风险：决议 ↔ 行动项双写不一致

`decision.actions[]`（id 列表）与 `action.sourceDecisionId` 是两套并存的反向引用，编辑器只写后者（`meeting-editor.js:756,784`），而 `computeResolutionProgress` 依赖前者（`resolution-helpers.js:179-190`）。二者无一致性维护逻辑，可能出现"决议进度算不到新挂行动项"。建议：定 `action.sourceDecisionId` 为唯一事实来源，`decision.actions[]` 改为派生计算。

### 4.3 悬空：战略地图

Objective 预留 `kpiRef`/`taskRef`（`src/lib/strategy-map-data.js:133-173`），种子数据与创建路径（`src/strategy-map.html:1248`）全部置 `null`。战略地图与执行链完全游离。

### 4.4 假关联：重点工作 type 标签

`type='meeting_resolution'` 只是分类标签，无 `meetingId`/`resolutionId` 字段，不能用于追溯。

## 5. L3 链路实例追溯

以下实例均取自系统种子数据（`src/cockpit.html:962-1097`、`src/meetings/data-store.js:99-243`）。标注【结构】的关联表示字段与代码路径已就绪，种子数据未填充，运行时由编辑器/迁移写入。

### 示例 A：计划 → 执行派生（已通）

```
Cycle: cycle_2026_marketing（2026年度 — 营销线，planning）
└─ 重点工作: task_1「大客户经营能力提升」（source=annual_plan，progress=65，owner=陈总监）
   └─ 派生: omp_task_1（source=omp，annualPlanTaskId=task_1）
      ├─ 里程碑: ms_1 渠道调研完成 ✅ / ms_2 合作伙伴签约 ✅ / ms_3 首批渠道培训 ⏳ / ms_4 渠道业绩达标 ⏳
      └─ 进度记录: rec_1（04-30, 40%）→ rec_2（05-31, 70%，自动回写 task.progress）
```

### 示例 B：会议内执行链（结构已通）

```
会议: 20260415 片联一季度经营分析会（L1/union，completed）
├─ 议程: key_task_management「上月行动项复盘」
│   └─【结构】decision.sourceTopicId → 此议程
├─ 决议: D001「Q2 营收目标 3.2 亿」（approved → closed）
│   ├─ sourceMeetingId = 20260415
│   └─【结构】actions[] / progress ← 由行动项聚合
└─ 行动项: A001「Q2 营收目标上调至 3.2 亿」（owner=李经理，deadline=04-30）
    ├─【结构】sourceDecisionId = D001，sourceAgendaId = 来源议程
    ├─ progressLogs[] 跟进流水 → completedAt 闭环
    └─ 上行回环：闭环后决议 progress 聚合更新 → 决议 closed
```

### 示例 C：重点工作 → 议程（已通，v6 起）

```

   ↓ AI 推荐（agenda-recommender.js 只读拉取上下文）或编辑器"关联重点工作"手选
```
