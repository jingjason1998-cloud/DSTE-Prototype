# RFC-006: 平台每日体检与持续改进

> 状态：`implemented` | 作者：AI-2 | 日期：2026-05-28

---

## 摘要

建立 DSTE 平台的自动化每日体检机制，覆盖代码质量、功能完整度、架构健康度三个维度。体检报告每日推送至企业微信，并在平台内提供可视化历史趋势页面。目标是通过"每天修复一点点"的持续改进策略，让产品质量稳步提升。

## 背景

当前平台存在以下运维盲区：

1. **代码规模失控**：cockpit.html 4179 行（阈值 500），reviewer.html 4229 行
2. **重复代码严重**：主题切换逻辑 reviewer.html 重复 90 次
3. **功能覆盖不完整**：17 个规划功能中 8 个仍是 Placeholder
4. **缺乏持续监控**：没有机制跟踪代码质量退化趋势
5. **问题发现滞后**：用户使用时才发现问题，而非日常巡检发现

## 目标

1. 每日自动运行体检，生成标准化报告
2. 报告推送到企业微信群，确保团队每日可见
3. 平台内提供历史趋势页面，支持问题追溯
4. 建立"自动修复流水线"，低风险问题每日自动修复

## 方案设计

### 体检体系

```
体检体系 = 基础体检 + 业务体检 + 架构体检

基础体检（已实现）：
├── 代码规模检查（各文件行数 vs 阈值）
├── 功能覆盖度检查（PRD 对比）
├── 架构耦合度检查（模块引用频次、渲染函数数）
├── 重复代码检查（主题切换、导航渲染、按钮样式）
├── 代码异味扫描（innerHTML、eval、alert、console.log）
├── AI 协作规范检查（SKILL.md、CI、SonarQube、ESLint）
└── 测试覆盖检查（E2E 文件数、单元测试缺失提示）

业务体检（规划中）：
├── API 健康度（各端点响应时间/成功率）
├── 数据完整性（KV 数据一致性校验）
├── 用户行为异常（错误率、跳出率趋势）
└── 性能指标（首屏加载、交互延迟）

架构体检（规划中）：
├── 依赖安全（npm audit、CVE 扫描）
├── 构建产物分析（bundle size 趋势）
└── 死代码检测（未引用的 CSS/JS）
```

### 技术实现

#### 1. 体检脚本

| 脚本 | 路径 | 说明 |
|------|------|------|
| `health-check.cjs` | `scripts/health-check.cjs` | 主体检脚本，输出带颜色的终端报告 |
| `daily-health-report.cjs` | `scripts/daily-health-report.cjs` | 运行体检 + 格式化 Markdown + 推送企业微信 |

#### 2. 企业微信推送

- **推送方式**：群机器人 Webhook
- **推送时间**：每天 8:07（避免整点拥挤）
- **消息格式**：Markdown
- **内容结构**：健康评分 + Top 5 问题 + 趋势提示

推送效果示例：

```
DSTE 每日体检报告 — 2026/05/28
健康评分：25% (2/8) 等级 D

需关注的问题（Top 5）：
1. 🔴 cockpit.html 4179 行 (阈值: 500)
2. 🔴 innerHTML/outerHTML: 81 处
3. 🔴 主题切换 reviewer.html 90 次
4. 🔴 cockpit.html 渲染函数: 20 个
5. ⚠️ 无单元测试（建议补充 vitest）

💡 建议：每天修复一点点，产品越来越好
```

#### 3. 报告归档

- **保存路径**：`reports/health-report-YYYYMMDD-HHMM.txt`
- **保留策略**：建议保留最近 90 天报告，支持趋势对比
- **文件格式**：纯文本，包含完整的终端输出（含颜色控制字符）

### 系统健康页面（规划中）

```
驾驶舱 / 系统管理 / 平台健康

页面布局：
┌─────────────────────────────────────────┐
│ 健康评分卡片    │  7天趋势图              │
│ (今日分数+等级)  │  (折线图)               │
├─────────────────────────────────────────┤
│ 问题清单（可筛选：全部/严重/警告/已修复）│
├─────────────────────────────────────────┤
│ 修复记录（已修复问题归档）               │
├─────────────────────────────────────────┤
│ 体检报告下载（历史报告 txt 列表）        │
└─────────────────────────────────────────┘
```

**数据来源**：读取 `reports/` 目录下的历史报告文件，解析评分和关键指标。

### 自动修复流水线（规划中）

| 级别 | 修复类型 | 示例 | 自动化程度 |
|------|---------|------|-----------|
| P0 | 无风险修复 | 格式化、排序、删注释 | 全自动，无需人工确认 |
| P1 | 低风险修复 | 修复 typo、统一命名、删死代码 | 全自动 + 测试通过后合并 |
| P2 | 中风险修复 | 重构重复代码、提取函数 | AI 生成 PR，人工 Review |
| P3 | 高风险修复 | 架构调整、API 变更 | 人工决策，AI 辅助分析 |

**P0/P1 自动修复示例**：

```yaml
# .github/workflows/auto-fix.yml
每日自动修复:
  触发: cron("0 6 * * *")  # 每天早 6 点
  步骤:
    1. 运行 eslint --fix
    2. 运行 prettier --write
    3. 删除未引用的 CSS 变量
    4. 统一 import 排序
    5. 运行测试套件
    6. 测试通过 → 自动提交 PR
    7. 测试失败 → 告警，人工介入
```

## 替代方案

| 方案 | 说明 | 未选原因 |
|------|------|---------|
| SonarQube  alone | 仅依赖 SonarQube 做代码质量 | SonarQube 关注代码质量，不关注业务完整度和产品健康度，与本体检互补而非替代 |
| 手动体检 | 每周人工检查 | 无法做到每日、无法追溯趋势、容易遗漏 |
| 只推送到邮件 | 邮件发送报告 | 邮件打开率低，企业微信更即时 |
