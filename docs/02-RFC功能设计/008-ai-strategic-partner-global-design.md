# RFC-008：DSTE 战略合作伙伴 — AI 全局能力设计方案

> 状态：草案（待评审）  
> 适用范围：DSTE 战略管理平台全流程（SP / BP / Execute / Review / 公共支撑）  
> 更新日期：2026-06-24

---

## 一、背景与目标

当前 DSTE 已经落地了若干单点 AI 能力：

| 能力 | 实现状态 | 后端 |
|------|----------|------|
| 会议材料智能审核 | 完整 | Kimi 代理 |
| 经营分析会议程推荐 | 完整 | Claude (Cloudflare Worker) → 计划迁移到 Kimi |
| 业务专题 AI 智能匹配 | 纯前端算法 | 无 |
| 业务专题 AI 浮窗 | UI 占位 | 无 |
| 驾驶舱 AI 战略助手 | UI 占位 | 无 |

这些能力彼此独立、后端不统一、数据未打通，缺少一份贯穿 DSTE 全流程的 AI 顶层 design。本 RFC 的目标是：

1. **把 AI 从「零散工具」升级为「战略合作伙伴」**，覆盖战略制定、解码、执行、评估四大阶段。
2. **统一 AI 技术底座**：网关、模型路由、上下文、知识库、观测。
3. **明确交互范式**：全局助手、场景化嵌入、主动洞察、批量报告四位一体。
4. **接入帆软 KMS 等企业知识源**，让 AI 的回答基于真实企业上下文。
5. **给出可落地的实施路线**，优先落地「经营分析会智能闭环」和「BP/年度经营计划 AI 辅助」。

---

## 二、设计原则

| 原则 | 说明 |
|------|------|
| **统一入口，分层能力** | 用户只感知一个 AI 伙伴，但能力按场景分层：全局问答、页面嵌入、主动洞察、报告生成。 |
| **数据可解释，答案可溯源** | AI 结论必须给出数据来源（哪个会议、哪条 KPI、哪个文档），避免黑盒幻觉。 |
| **渐进落地，mock 兜底** | 真实模型可配置；未配置时自动降级为规则/模板/mock，保证功能可用。 |
| **人在回路** | 涉及决策、目标调整、预算分配等关键操作，必须人工确认后方可写入系统。 |
| **一次设计，分阶段实现** | 先补齐 Execute 和 BP 的高价值闭环，再逐步向 SP/Review 扩展。 |

### 3.1 SP 战略制定

| 能力 | 用户价值 | 输出形态 |
|------|----------|----------|
| 市场/行业洞察生成 | 基于 KMS 文档和外部数据，生成战略洞察草稿 | 可编辑卡片 + 引用来源 |
| 战略地图辅助 | 根据洞察自动生成 BSC 四维度候选目标 | 候选节点列表，一键加入地图 |
| 战略专题识别 | 扫描经营数据，识别需要高层关注的专题 | 主动洞察卡片 |

| 能力 | 用户价值 | 输出形态 |

| 能力 | 用户价值 | 输出形态 |

| 能力 | 用户价值 | 输出形态 |

| 预算偏差分析 | 识别预算执行偏差并给出调整建议 | 偏差分析卡片 |
| 复盘辅助 | 基于数据和会议纪要生成复盘提纲 | 复盘模板 |

### 4.1 AI SDK / Client（前端）

目标：所有前端 AI 调用统一走 `src/lib/ai-client.js`，替代现有 `agenda-recommender.js`、`reviewer.js` 中分散的 fetch。

核心职责：

- **统一构造上下文**：从各模块读取数据并序列化为标准 `AIContext`。
- **工具调用（Function Calling）**：让 AI 可以调用系统函数查询数据、跳转页面、创建草稿。
- **会话管理**：维护会话 ID、历史消息、用户反馈（👍/👎）。
- **降级策略**：模型不可用或返回异常时，自动切换规则引擎/mock。
- **安全**：敏感字段脱敏、请求审计日志。

标准请求结构：

```typescript
interface AIRequest {
  sessionId: string;
  userId?: string;
  intent: 'chat' | 'recommend' | 'review' | 'generate' | 'analyze' | 'simulate';
  phase?: 'sp' | 'bp' | 'execute' | 'review';
  pageContext?: {
    pageId: string;
    entityId?: string;
    entityType?: 'meeting' | 'topic' | 'kpi' | 'task' | 'plan';
  };
  message: string;
  attachments?: Attachment[];      // 文档链接、KMS 链接
  tools?: string[];                // 允许 AI 调用的工具名
  stream?: boolean;
}
```

标准响应结构：

```typescript
interface AIResponse {
  sessionId: string;
  message?: string;
  cards?: AICard[];                // 结构化卡片
  actions?: AIAction[];            // 可执行操作
  citations?: Citation[];          // 引用来源
  toolCalls?: ToolCall[];          // AI 请求调用的工具
  followups?: string[];            // 推荐追问
  error?: string;
}
```

### 4.2 AI 网关 / Worker

建议基于现有 `api-worker/worker.js` 扩展为统一 AI 网关，新增职责：

1. **认证与限流**：API Key、用户级配额、IP 限流。
2. **模型路由**：根据 `intent` 和 `modelHint` 选择模型。
3. **提示词工程中心**：按场景管理 system prompt、 few-shot 示例。
4. **RAG 检索**：调用向量库/KMS 检索相关文档片段，注入上下文。
5. **后处理与校验**：JSON Schema 校验、敏感词过滤、引用溯源。
6. **审计日志**：记录请求、模型、耗时、token、用户反馈。
7. **缓存**：相同查询缓存结果，降低成本。

### 4.3 模型路由策略（已确认）

基于当前无 Claude 付费账号、且 Kimi 多模态模型能力已足够覆盖 DSTE 业务场景，统一采用 **Kimi 多模态模型**作为唯一大模型 provider。

| 场景 | 推荐模型 | 理由 |
|------|----------|------|
| 通用对话、复杂推理、目标拆解、根因分析、报告生成 | `kimi-k2.6` | 256K 上下文、多模态理解、ToolCalls/JSON Mode 稳定 |
| 长文档审核、会议纪要、大量上下文 | `kimi-k2.6` | 长文本能力已验证，可直接替代原 Claude 场景 |
| 知识库向量检索 | 独立 embedding 模型 | 检索与大模型解耦，可用 text-embedding-3 / BGE-m3 等 |
| 未来私有化/信创场景 | 私有化部署模型 | 数据不出域 |

**推荐路线**：统一 AI 网关 + Kimi 单模型路由。所有业务场景默认使用 `kimi-k2.6`，统一复用会议审核现有的 Kimi API 接入方式（`https://api.moonshot.cn/v1` 或现有代理）。不启用 `kimi-k2.7-code` 专用代码通道，后续如有代码生成需求再评估。

> **迁移注意**：现有 `agenda-recommender.js` 调用的是 Claude（Cloudflare Worker）。接入统一网关后，前端不再关心底层模型，由网关统一转发到 Kimi `k2.6`。

### 4.4 上下文与记忆

- **短期记忆**：会话级消息历史，保存在前端 `sessionStorage` 或后端缓存。
- **长期记忆**：用户对 AI 建议的采纳/反馈、常用查询、偏好模板，写入 `dste_ai_memory`。
- **业务上下文**：当前页面实体（会议、专题、KPI 等）自动注入，避免用户重复描述。
- **企业上下文**：KMS 文档片段、制度文件、历史会议纪要通过 RAG 注入。

### 4.5 评估与观测

| 指标 | 说明 |
|------|------|
| **功能可用率** | AI 调用成功率、降级触发率 |
| **用户采纳率** | AI 推荐/生成内容的采纳比例 |
| **任务完成率** | 用户通过 AI 完成目标任务的比例 |
| **幻觉率** | 人工抽检中不可溯源或错误的结论比例 |
| **响应延迟** | P50/P95 响应时间 |
| **成本** | 按场景统计 token 消耗与费用 |

## 五、交互范式

### 5.1 全局 AI 助手（cockpit#ai）

把当前占位的 AI 页升级为真正的战略助手入口：

- **左侧对话区**：自然语言问答、任务执行。
- **右侧洞察面板**：根据用户角色和当前周期展示主动洞察卡片。
- **快捷操作**：「生成本月经营分析报告」「帮我看看逾期行动项」「拆解年度目标」等。
- **工具调用可视化**：AI 查询数据、跳转页面时展示过程。

示例对话：

> 用户：帮我看看 Q2 有哪些 KPI 落后了？  
> AI：正在扫描 OMP 数据……发现 3 项 KPI 落后：  
> 1. 华南区营收完成率 78%（目标 85%）—— 来源：OMP KPI #KPI-1024  
> 2. 新客户转化率 12%（目标 18%）—— 来源：OMP KPI #KPI-1025  
> [查看 OMP] [生成根因分析] [加入下次经营分析会议题]

### 5.2 场景化嵌入式 AI

在关键页面右下角或工具栏提供嵌入式 AI：

| 页面 | 嵌入能力 |
|------|----------|
| `meetings.html` | 议程推荐、材料审核、会中助手、纪要生成、决议/行动项提取 |
| `cockpit.html#bp/annual-plan` | 目标拆解、KPI 推荐、预算模拟、计划书生成 |
| `cockpit.html#exe/business-topics` | 专题智能关联、KMS 资料推荐、AI 全局报告 |
| `cockpit.html#exe/tasks` (OMP) | KPI 根因分析、趋势预测、重点工作建议 |
| `cockpit.html#sp/strategy-map` | 战略地图节点推荐、目标关联建议 |

### 5.3 主动洞察卡片

在驾驶舱、列表页、详情页展示 AI 主动发现的洞察：

```
┌─────────────────────────────────────┐
│ ⚠️ 主动洞察                          │
│ 华南区营收连续 2 个月落后，建议      │
│ 纳入下周经营分析会议题。             │
│ [查看详情] [加入会议] [忽略]         │
└─────────────────────────────────────┘
```

触发来源：

- 数据异常：KPI 连续 N 期落后、预算超支、里程碑逾期。
- 流程异常：会议材料未审核、决议长期未闭环、行动项逾期未更新。
- 机会识别：某战区超额完成、新项目达成里程碑、历史最佳实践。

### 5.4 报告/批量生成中心

在「报表中心」或独立「AI 报告中心」提供：

- 经营月报/季报/年报生成
- 会议纪要批量整理
- 决议/行动项汇总
- 预算执行分析报告
- 专题研究报告

用户可选择模板、时间范围、输出格式（HTML/PDF/Word），人工确认后发布。

## 六、数据接入与知识库

### 6.1 本地业务数据统一上下文

建立 `src/lib/ai-context.js`，统一从各模块读取数据并脱敏：

```javascript
// 示例：构造会议场景上下文
const meetingContext = {
  meeting: { id, title, date, scenario, status, agenda_items, actions, resolutions },
  relatedMeetings: [...],      // 同系列历史会议
  openActions: [...],          // 未闭环行动项
  openResolutions: [...],      // 未闭环决议
  ompTasks: [...],             // OMP 重点工作
  kpiSnapshots: [...],         // 相关 KPI 快照
  businessTopics: [...],       // 相关业务专题
};
```

数据来源：

| 模块 | localStorage Key / API | 说明 |
|------|------------------------|------|
| 经营分析会 | `dste_meetings` | 会议、议程、决议、行动项 |
| OMP | `dste_omp_*_v1` | KPI、重点工作、里程碑、进度记录 |
| 业务专题 | `dste_topics` | 专题、议题、里程碑 |
| 战略指标库 | `dste_kpi_*` | 指标定义、目标值、实际值 |
| 年度经营计划 | `dste_annual_plan_*` | 计划目标、预算、分解 |
| 人员目录 | `dste_employee_directory` | 负责人、组织关系 |
| 材料审核 | `dste_review_scores` | 历史审核记录 |

### 6.2 帆软 KMS 接入

利用已有 KMS Confluence API（见项目记忆），将 KMS 页面作为企业知识库：

- **文档索引**：定期抓取 KMS 全部可访问空间页面，提取文本并分块。
- **向量存储**：使用 embedding 模型生成向量，存储于向量数据库或 Cloudflare Vectorize。
- **检索增强**：用户提问时，先检索 KMS 相关片段，再注入大模型上下文。
- **引用溯源**：AI 回答中标注来源页面标题和链接。

> 已确认：KMS 有 API 且可接入全部空间，不先做范围限制，按用户可见权限拉取。

### 6.3 未来扩展

- 接入会议纪要、制度 PDF、Excel 模板。
- 与帆软报表/BI 系统打通，获取实时经营数据。
- 建立「企业知识图谱」：人-组织-KPI-会议-专题-文档关系网络。

## 七、后端选型建议

### 7.1 推荐方案：统一 AI 网关 + Kimi 多模型路由

**核心思路**：

- 以现有 `api-worker/worker.js` 为基础，升级为 `AI Gateway`。
- 保留并标准化 `/api/ai/agenda`、新增 `/api/ai/chat`、`/api/ai/review`、`/api/ai/generate`、`/api/ai/analyze`、`/api/ai/simulate`。
- 新增 `ProviderRegistry`：KimiProvider、EmbeddingProvider、RuleProvider、MockProvider。
- 新增 `RAGService`：对接 KMS 与向量库。
- 将现有议程推荐后端从 Claude 迁移到 Kimi，会议材料审核继续走 Kimi，实现「一个 Kimi API Key 管所有」。

**为什么是 Kimi 统一路由？**

- 当前无 Claude 付费账号，继续使用 Claude 会产生不可控成本。
- Kimi K2.6 在 256K 上下文、ToolCalls、JSON Mode、多模态理解上已满足 DSTE 全部业务场景。
- 会议材料审核场景已经在 Kimi 上验证通过，迁移风险低。
- 统一 provider 后，配置、审计、缓存、降级都更简单。
- 多模型路由保留在 Kimi 品牌内部（`k2.5` / `k2.6` / `k2.7-code`），未来切换或私有化只需改网关配置。

### 7.2 部署形态

| 环境 | 形态 | 说明 |
|------|------|------|
| 本地开发 | `localhost:8766` 本地网关 | 复用现有 `ai-agenda-local-server.cjs` 与会议审核代理，扩展为多能力网关 |
| 生产 | 调用现有会议审核代理（nginx 反向代理） | 保证可用性，Cloudflare Worker 链路暂不启用 |
| 测试/预发 | 与生产一致，独立配置 | 可指向测试代理或官方 API |
| 未来 | 私有化部署网关 | 数据不出域时使用本地模型 |

## 八、安全、合规与降级

### 8.1 数据安全

- **全量数据可进入 AI 上下文与记忆**：已确认当前阶段允许财务、薪酬、干部评估等业务数据进入 AI 上下文，用于生成分析、报告与洞察。
- **最小必要原则**：按场景只传必要字段，不传全量 localStorage，降低 token 成本与泄露面。
- **审计日志**：记录谁、何时、调用了什么 AI 能力、输入了什么数据、返回了什么结论。
- **权限校验**：AI 操作涉及写入时，校验用户是否有对应模块的编辑权限；KMS 检索范围受用户原始权限控制。

### 8.2 合规与可信

- **引用溯源**：所有 AI 结论必须附带数据来源，便于人工复核。
- **置信度标注**：对推荐/分析结果标注置信度，低置信度内容需要人工确认。
- **人在回路**：涉及创建、修改、删除、审批等操作，必须先展示草稿，由用户确认后执行。
- **反馈闭环**：用户可对 AI 回答点赞/点踩/纠错，持续优化提示词和模型选择。

### 8.3 降级策略

| 场景 | 降级方式 |
|------|----------|
| 无 API Key / 网络异常 | 返回 mock 数据或规则引擎结果 |
| 模型响应超时 | 返回「建议稍后重试」+ 预置模板 |
| 返回内容不符合 schema | 用规则后处理修正或提示用户 |
| 敏感查询 | 拒绝回答并提示联系管理员 |

## 九、实施路线图

### Phase 1：统一 AI 底座 + 全局助手（C，约 2-3 周）

目标：先让 AI 有一个统一的入口和可工作的全局助手，后续场景再往上挂。

| 模块 | 任务 |
|------|------|
| 统一底座 | 创建 `src/lib/ai-client.js`；扩展本地代理为统一 AI 网关；定义标准请求/响应 schema；将议程推荐后端从 Claude 迁移至 Kimi |
| 全局助手 | 重写 `cockpit.html#ai`：自然语言问答、快捷操作、工具调用可视化、会话管理 |
| 上下文接入 | 打通会议/OMP/业务专题/KPI/年度计划等数据，构造标准 `AIContext` |
| KMS 基础 | 建立 KMS 文档抓取脚本，先支持少量空间索引，全局助手可查询 |
| 验证 | 新增/更新单元测试、E2E 测试；跑通 `npm run build` 与 `check:scope` |

### Phase 2：经营分析会智能闭环（A，约 2-3 周）

目标：把会议会前/会中/会后的 AI 辅助跑通，全部接入统一 client。

- 议程推荐接入统一 client，后端已迁移到 Kimi。
- 材料审核与会议流程打通（审核报告关联到会议详情）。
- 新增会中助手侧边栏：实时问答、补充数据、KMS 资料查询。
- 新增会议纪要/决议/行动项生成。
- 督办：逾期催办文案生成、升级建议。

### Phase 3：BP 辅助 + 主动洞察 + 报告中心（约 2-3 周）

目标：把 AI 能力扩展到 BP/年度计划，并实现主动洞察与报告生成。

- 年度经营计划页面新增「AI 拆解」按钮：目标拆解、KPI 推荐、预算 What-if 模拟、计划书生成。
- 主动洞察卡片：逾期行动项、落后 KPI、预算异常、会议材料未审核等。
- AI 报告中心：经营月报、会议纪要汇总、决议跟踪报告。
- KMS 知识库全面接入。

### Phase 4：SP/Review 高级能力（约 2-3 周）

- 战略地图 AI 辅助生成节点与关联。
- 战略洞察与专题自动生成。
- KPI 根因分析与趋势预测。
- 复盘辅助与绩效评估建议。

### Phase 5：企业级扩展（长期）

- 私有化模型接入。
- 与帆软 BI/报表实时数据打通。
- 企业知识图谱。
- 多语言、语音输入、智能体编排。
