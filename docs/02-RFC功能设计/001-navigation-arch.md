# DSTE 导航架构 V2 设计方案

> 目标：顶部 = DSTE 流程一级导航，左侧 = 功能二级导航，驾驶舱作为融合概览

---

## 一、核心原则

1. **顶部 = 战略阶段轴**（DSTE 闭环流程）
2. **左侧 = 功能清单**（当前阶段下的可操作模块）
3. **驾驶舱 = 融合概览**（跨阶段数据汇总，不看左侧分组）
4. **经营分析会 = 中央枢纽**（横跨执行+评估，在执行阶段主入口）

## 四、维度融合设计

### 4.1 驾驶舱 = 融合视图

驾驶舱不归属于任何单一阶段，而是**跨阶段融合概览**：
- 顶部：SP/BP/Execute/Review 的关键指标卡片
- 中部：战略地图四维度快照
- 底部：重点工作 + 经营分析会日历

### 4.2 经营分析会 = 跨阶段枢纽

经营分析会**主要挂在执行阶段**（因为它是推动执行的核心机制），但在以下场景可快速跳转：
- 评估阶段的"战略复盘"可链接到相关经营分析会纪要
- 驾驶舱显示下次会议倒计时

### 4.3 快捷跳转（面包屑+关联入口）

每个页面底部或侧边栏底部，显示：
```
关联功能 → [战略地图] [KPI看板] [经营分析会]
```

## 八、开发方式对比

### 方案 A：独立 HTML 页面（多页）

每个功能一个独立 `.html` 文件，浏览器整页刷新跳转。

| 维度 | 详情 |
|------|------|
| **实现难度** | ⭐ 低 — 纯静态页面，无复杂JS |
| **代码复用** | 顶部导航 + 左侧栏在每个页面重复，用模板/复制维护 |
| **切换体验** | 整页刷新，有短暂白屏/闪烁 |
| **状态保持** | 困难 — 主题、展开/收起状态需手动同步（localStorage） |
| **前进后退** | ✅ 浏览器原生支持 |
| **SEO/分享** | ✅ 每个页面独立URL，可直接分享 |
| **适合场景** | 原型演示、静态站点、功能相对独立 |

**工作量估算：** 每个页面 ~300-500 行 HTML，共 10+ 个页面，总计 ~4000-6000 行。重复代码多。

---

### 方案 B：单页应用 SPA（推荐）

只有一个 `cockpit.html`，顶部+左侧作为固定框架，右侧内容区通过 JS 动态替换。

| 维度 | 详情 |
|------|------|
| **实现难度** | ⭐⭐ 中 — 需要简单的 JS 路由+内容切换逻辑 |
| **代码复用** | 顶部+左侧只写一次，所有页面共享 |
| **切换体验** | 无刷新，内容区域淡入淡出，像 App |
| **状态保持** | 自然 — 主题、侧边栏状态全局一致 |
| **前进后退** | 需用 `history.pushState` 模拟（约 20 行代码） |
| **SEO/分享** | 可用锚点 `#sp/strategy-map` 模拟独立URL |
| **适合场景** | 管理后台、仪表盘、需要频繁切换的协同工具 |

**工作量估算：** 框架 ~200 行 + 每个页面内容 ~150-300 行，总计 ~2000-3500 行。代码更集中。

---

### 对比总结

| 对比项 | 多页 A | SPA B |
|--------|--------|-------|
| 开发速度 | 快（直接写HTML） | 稍慢（需先搭框架） |
| 维护成本 | 高（改导航要改10个文件） | 低（改一处全局生效） |
| 切换流畅度 | 有闪烁 | 丝滑 |
| 原型→产品 | 需重构 | 可直接扩展 |
| **DSTE匹配度** | 一般 | **高** — 闭环管理需要频繁跨阶段切换 |

### 我的推荐：方案 B（SPA）

理由：
1. **DSTE 是闭环流程**，用户会在 SP→BP→Execute→Review 之间频繁切换，SPA 的无刷新体验更符合"作战指挥舱"的实时感
2. **驾驶舱需要状态一致性**，多页切换会导致主题、侧边栏展开状态丢失
3. **代码量更少**，维护更简单
4. **原型可直接扩展为产品**，不需要推倒重来

实现也很简单：
```
cockpit.html（框架：顶部+左侧+内容容器）
  ├── JS 路由监听顶部点击
  ├── 根据路由加载对应 HTML 片段到内容区
  └── history.pushState 支持前进后退
```
